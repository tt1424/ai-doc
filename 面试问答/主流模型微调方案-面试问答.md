# 主流模型微调方案 · 面试问答

> 基于《主流模型微调方案》整理的面试题库,涵盖微调全流程:训练范式 → 工具选型 → 数据工程 → LoRA/QLoRA → 偏好对齐(DPO/ORPO)。
> 每题包含「考察点」「参考回答」「加分项」,可用于准备 LLM 微调 / 算法工程师岗位面试。

---

## 一、训练范式与全局认知

### Q1. 业内主流的模型微调方案有哪些?请简要对比。

**考察点**:对微调技术全景的整体认知。

**参考回答**:
主流方案有四种:
- **SFT(有监督微调)**:用 `(Instruction, Input, Output)` 问答对教模型"听懂人话",学会按指令格式输出。
- **DPO(直接偏好优化)**:用 `(Chosen, Rejected)` 偏好数据做价值观对齐,告诉模型哪个回答更好。相比 PPO 更简单稳定。
- **PPO(近端策略优化)**:经典 RLHF 算法,但**极其难训、显存占用巨大**,需要额外的奖励模型和价值模型。
- **ORPO(赔率比偏好优化)**:把 SFT 和 DPO **二合一**,一次训练同时完成"会说话"和"会做人"。

**加分项**:能点出趋势是 `SFT + DPO` 为当前主流,而 ORPO 是为了简化两阶段流程而生。

---

### Q2. 大模型进化的三个阶段是什么?各自解决什么问题?

**考察点**:Pre-train / SFT / RLHF 的本质区别。

**参考回答**:
1. **预训练 (Pre-training)**:海量数据"接龙",学习语法、逻辑和世界知识。产出 Base Model,博学但**不懂指令**(你说"写首诗",它可能续写"写首歌、写篇散文")。门槛极高,需数千 GPU 训练数月,个人开发者通常直接用开源基座。
2. **有监督微调 (SFT)**:用高质量问答对教模型按指令输出,激活预训练知识。产出 Chat Model,能流畅对话,但**容易幻觉、可能输出有害信息**。门槛中低,单卡/多卡即可。
3. **人类价值观对齐 (RLHF / Alignment)**:用偏好数据规范行为边界,让模型**有用(Helpful)、诚实(Honest)、无害(Harmless)**。这是 ChatGPT 区别于普通聊天机器人的关键。门槛中高。

**加分项**:用一句话概括——预训练给"知识",SFT 给"格式",对齐给"价值观"。

---

### Q3. 请用通俗的例子说明 SFT 和 DPO 的区别。

**考察点**:能否把技术概念讲清楚(沟通能力)。

**参考回答**:
用"教徒弟做菜"来比喻:
- **SFT 像教条**:导师拿着标准教程告诉徒弟"客人说想吃辣的,就给麻婆豆腐,步骤 1、2、3"。解决的是**"懂规矩"**——教会格式,问问题不再乱说话。
- **DPO 像培养厨德情商**:徒弟会做菜了,但有时往菜里加砒霜(有害),或者菜虽熟但难吃(对但废话多)。这时给他两个盘子说"选 A,别选 B",不用教怎么做,只需指出好坏。解决的是**"好坏"和"边界"**问题。

一句话总结:**成才(SFT)先成人(DPO)**。SFT 只教"怎么做",DPO 才教"怎么做更好"和"什么不能做"。

## 二、微调工具选型

### Q4. 主流微调工具有哪些?分别适合什么场景?

**考察点**:工程落地的工具熟悉度。

**参考回答**:
| 工具 | 地位 | 特点 | 适用场景 |
|------|------|------|----------|
| **Hugging Face TRL** | 官方正统 | 集成 SFTTrainer/DPOTrainer/PPOTrainer,代码规范、文档齐全 | 学习原理、标准流程开发 |
| **Unsloth** ⭐ | 当前最强黑马 | 重写底层 Triton kernel,**速度快 2-5 倍,显存省 50%** | 个人开发者、单卡玩家首选 |
| **LLaMA-Factory** ⭐ | WebUI 神器 | **零代码**可视化界面,支持几乎所有主流模型 | 快速验证、不写代码的人、企业低门槛平台 |
| **DeepSpeed / Accelerate** | 多卡分布式基石 | ZeRO-3 把模型切碎放到多卡 | 多卡训练、大规模集群(如训 70B) |

**加分项**:知道一张卡装不下大模型(如 70B)时必须用 DeepSpeed ZeRO-3 做模型切分。

## 三、数据工程(微调的核心)

### Q5. 为什么说数据质量决定微调效果?

**考察点**:对"数据 > 算法"这一行业共识的理解。

**参考回答**:
AI 圈名言:**"Garbage In, Garbage Out"**(垃圾进,垃圾出)。微调效果**70% 取决于数据质量,30% 取决于算法和参数**。很多时候调一个月参数,效果不如花一天清洗数据。所以应该把主要精力投入到数据收集和清洗上。

---

### Q6. Alpaca 和 ShareGPT 两种数据格式有什么区别?如何选型?

**考察点**:数据格式标准的掌握。

**参考回答**:
- **Alpaca 格式**:源自斯坦福 Alpaca 项目,适合**单轮"一问一答"**(逻辑推理、分类、翻译)。结构为 `instruction`(必填)、`input`(选填)、`output`(必填)。
- **ShareGPT 格式**:源自 ShareGPT 项目,适合**多轮对话**,能学会联系上下文。结构为 `conversations` 列表,每条含 `from`(角色 human/gpt)和 `value`(内容)。

**选型建议**:现在趋势是 **All in ShareGPT**。即使单轮问答也可包装成只有一轮的 ShareGPT 格式,这样数据处理脚本统一,不用写两套。

---

### Q7. 数据清洗通常做哪些工作?为什么要去重?

**考察点**:数据清洗实操经验。

**参考回答**:
三大清洗动作:
1. **去重**:模型看 100 次同样的笑话会死记硬背(过拟合),泛化下降。
   - **精确去重**:字符串完全匹配删除。
   - **模糊去重 (MinHash + LSH)**:大厂标配,"今天天气不错"和"今天天气真不错"虽字不同但能识别为高度相似,删掉其一。
2. **去除"拒绝回答"**:很多开源数据用 GPT-4 生成,常含"作为 AI 语言模型,我无法回答..."。**绝不能喂给模型**,否则模型变成只会道歉的复读机。策略:扫描 `output` 字段命中关键词就整条丢弃。
3. **关键词过滤**:过滤暴力、色情、广告代码等数据。

---

### Q8. 微调到底需要多少数据?LIMA 论文给了什么启示?

**考察点**:数据"量 vs 质"的判断。

**参考回答**:
**数量 < 质量**。LIMA 论文证明:仅用 **1000 条顶级质量**人类精修数据微调的模型,可以吊打用 **50000 条粗制滥造**数据微调的模型。所以与其疯狂爬数据,不如花钱精修 500 条完美问答对。

---

### Q9. 微调数据如何配比?为什么不能只用垂直领域数据?

**考察点**:对"灾难性遗忘"的理解和配比策略。

**参考回答**:
只喂垂直领域数据(如只喂医疗),模型会"变傻"——忘了写代码、闲聊,这叫**灾难性遗忘**。

**黄金配比**:
- **70-80%**:垂直领域数据(如医疗问诊)。
- **10-20%**:通用能力数据(如 `alpaca_gpt4_zh`,维持智商)。
- **5-10%**:自我认知数据(告诉它"我是 XX 医疗助手",防止它说"我是 ChatGPT")。

**加分项**:自我认知数据需自己写脚本生成几百条,如 `{"input":"你是谁?","output":"我是极客医疗助手..."}`。

## 四、LoRA / QLoRA 与调参

### Q10. 为什么不推荐全量微调?LoRA 的核心思想是什么?

**考察点**:参数高效微调(PEFT)的原理。

**参考回答**:
**全量微调的问题**:
- **成本极高**:要更新每个参数,需加载权重 + 梯度 + 优化器状态,显存需求是模型大小的 4 倍以上。
- **灾难性遗忘**:为学新知识可能"洗脑"过度,忘光旧知识。

**LoRA 的"旁路"哲学**:**冻结**原大模型,只在旁边外挂两个很小的矩阵 A 和 B 来学新知识。
- 比喻:大模型是百科全书,全量微调是擦掉重写(费劲、易写坏),LoRA 是在书页边角贴便利贴(轻松、随时可撕),推理时既看书也看便利贴。
- 数学原理:不直接改权重 `W`,而是训练增量 `ΔW`,并假设 `ΔW = A × B`(低秩分解)。A、B 参数量可能只有原模型的 1% 甚至 0.1%,梯度和优化器状态极小,显存大幅节省。

---

### Q11. QLoRA 是什么?它解决了什么问题?

**考察点**:量化训练的理解。

**参考回答**:
**QLoRA = 冻结的 4-bit 基座模型 + LoRA 适配器**,让你能在烂显卡上训练大模型。
- **4-bit 量化**:大模型通常用 16-bit(FP16/BF16)存储,QLoRA 压缩到 4-bit(NF4 格式),**体积缩小 4 倍**。7B 模型原占 14G 显存,量化后只占 3.5G 左右。
- **代价**:精度有极微小损失,微调任务中几乎可忽略。

这是目前**单卡训练的标准答案**。

---

### Q12. LoRA 调参中,Rank (r) 怎么设置?

**考察点**:核心超参的经验法则。

**参考回答**:
- **含义**:LoRA 矩阵的秩,决定模型能学多少新知识的"容量"。
- **常见值**:8, 16, 32, 64, 128。
- **经验法则**:
  - 简单任务(改说话风格、分类):`r = 8` 或 `16`。
  - 复杂任务(学全新医疗知识、逻辑推理):`r = 64` 或 `128`。
- **注意**:r 越大,显存越高、训练越慢、越容易过拟合。**64 通常是性价比极高的选择**。

---

### Q13. Alpha (lora_alpha) 和 Target Modules 怎么设?

**考察点**:LoRA 配置细节。

**参考回答**:
- **Alpha(缩放因子)**:决定新学知识对原模型影响有多大。**黄金法则:Alpha = 2 × Rank**(如 r=64 则 alpha=128),这是无数实验总结的"无脑设置"。
- **Target Modules(挂载层)**:Transformer 含 Attention 层(`q_proj/k_proj/v_proj/o_proj`)和 MLP 层(`gate_proj/up_proj/down_proj`)。
  - 以前为省显存只训 `q_proj` 和 `v_proj`。
  - **最佳实践:All Linear Modules(所有线性层全开)**,研究证明效果提升明显,而显存增加并不多。

---

### Q14. LoRA 的学习率怎么设?为什么和全量微调不同?

**考察点**:学习率与调度器理解。

**参考回答**:
- **QLoRA 推荐学习率**:`2e-4`。这比全量微调(通常 `1e-5`)**大很多**,因为 LoRA 参数少,需要更激进的更新。
- **Scheduler**:推荐 `cosine`(余弦退火)或 `linear`。
- 形象比喻:Learning Rate 是车速,Scheduler 是踩刹车的方式(下坡时如何减速)。

---

### Q15. 什么是 NEFTune?它有什么作用?

**考察点**:进阶训练技巧。

**参考回答**:
**NEFTune (Noisy Embeddings Fine Tuning)**:训练时给输入的 Embedding 向量**随机加一点噪声**。
- **作用**:防止死记硬背(过拟合),**显著提升对话流畅度和指令遵循能力**。
- **用法**:Trainer 中设置 `neftune_noise_alpha = 5`。

## 五、偏好对齐(DPO 数据工程)

### Q16. DPO 的偏好数据是什么结构?为什么需要 Rejected?

**考察点**:偏好数据三元组理解。

**参考回答**:
偏好数据是**"三元组"**:
1. **Prompt**:用户提问。
2. **Chosen(胜者)**:更好的回答。
3. **Rejected(败者)**:较差的回答。

**为什么需要 Rejected**:模型像小孩,SFT 教"见人说你好",DPO 教"见长辈叫叔叔(Chosen),叫老头子(Rejected)不礼貌"。**只有同时看到正例和负例,模型才能通过对比学到隐含的价值观和风格。**

格式示例(JSONL):
```json
{"prompt": "如何像海盗一样打招呼?", "chosen": "阿霍伊!老伙计!...🏴☠️", "rejected": "你好,我的朋友。请问最近过得怎么样?"}
```
两个回答语义都对,但 Chosen 遵循了"像海盗"的风格指令,Rejected 虽礼貌但跑题。

---

### Q17. 偏好数据怎么获取?LLM-as-a-Judge 流程是怎样的?

**考察点**:偏好数据生产方法。

**参考回答**:
两种主流方法:
- **方法一:人工标注(黄金标准)**:OpenAI/Anthropic 做法,质量最高但最贵。让 SFT 模型对同一 Prompt 生成 4 个回答,人工排序(如 A>C>B>D),再转成配对 `(A,C)、(A,B)、(C,D)`。适合安全合规要求极高的企业项目。
- **方法二:LLM-as-a-Judge(工业界主流)⭐**:用魔法打败魔法,让 GPT-5 / Claude 当裁判评价小模型的回答。流程(RIF, Reinforced from AI Feedback):
  1. **生成**:SFT 模型对每个问题生成 2 个回答,设较高 `Temperature`(如 0.9)保证差异够大。
  2. **评审**:把 `[问题, 回答A, 回答B]` 发给 GPT-4,要求扮演公正裁判,按"是否遵循指令、是否有幻觉、是否简洁"选出更好的。
  3. **构建数据**:根据裁判选择构建 `{chosen, rejected}`。

### Q18. DPO 数据有哪三个致命的"坑"?如何避免?

**考察点**:对 DPO 数据敏感性的深度理解(高频深挖题)。

**参考回答**:
1. **长度偏见**:模型天生认为"写得长 = 写得好"。如果 Chosen 总比 Rejected 长,模型会变**话痨**,堆砌废话讨好奖励。
   - **解药**:刻意保留一些"短而精"的 Chosen;若裁判是因为长才选,检查是否废话文学,人工修正为 Rejected。
2. **用 DPO 修事实错误**:这是误区。**DPO 更擅长修风格(Style)和安全性(Safety),而非知识。** 知识应在 Pre-train 或 SFT 阶段注入;DPO 修知识效率低、易死记硬背。
3. **相似度过高**:Chosen 和 Rejected 只差一两个字,Loss 极小,模型学不到东西甚至梯度消失。
   - **解药**:确保两回答在逻辑、语气或结构上有明显差异(Edit Distance 足够大)。

## 六、ORPO 与展望

### Q19. DPO 在工业界落地有哪些局限?

**考察点**:对 DPO 不足的批判性认识。

**参考回答**:
三个头疼问题:
1. **数据难搞**:构建成对 `(Chosen, Rejected)` 太贵太难,能不能只用"点赞"数据?
2. **过拟合**:DPO 有时用力过猛,模型为讨好偏好导致语言能力退化(开始胡言乱语)。
3. **流程繁琐**:先 SFT 再 DPO 要训两次,还要加载 Reference Model,显存和时间双倍消耗。

---

### Q20. ORPO 是如何解决这些问题的?原理是什么?

**考察点**:对最新二合一方案的理解。

**参考回答**:
**ORPO (Odds Ratio Preference Optimization)** 把 SFT 和 DPO **二合一**:在做 SFT(学怎么说话)的同时,顺便把 DPO(学怎么做人)做了。

**原理**——修改 SFT 的 Loss 函数:
```
Loss = SFT Loss + λ × OR Loss (Odds Ratio)
```
- **SFT Loss**:保证模型学会 Chosen 回答(能说话)。
- **OR Loss**:强迫模型生成 Chosen 的概率比 Rejected 高(有偏好)。

**优势**:
1. **省钱**:只需训练一次。
2. **省显存**:**不需要加载 Reference Model!** 对单卡玩家是巨大福音。7B 模型 DPO 可能要 30G 显存,ORPO 只要 20G。

## 附:高频速记要点

- 微调四方案:**SFT / DPO / PPO(难训) / ORPO(二合一)**
- 三阶段:**预训练(知识)→ SFT(格式)→ 对齐(价值观)**
- 一句话:**成才(SFT)先成人(DPO)**
- 工具:单卡选 **Unsloth**,零代码选 **LLaMA-Factory**,多卡用 **DeepSpeed ZeRO-3**
- 数据:**70% 靠质量**;LIMA 证明 1000 条精修 > 50000 条粗制;配比 **70-80% 垂直 + 10-20% 通用 + 5-10% 自我认知**
- LoRA:**ΔW = A×B 低秩分解**;`r=64` 性价比高;**Alpha = 2×Rank**;**All Linear 全开**;LR=`2e-4`
- QLoRA = **4-bit 量化基座 + LoRA**,单卡标准答案
- DPO 三坑:**长度偏见 / 误修事实 / 相似度过高**;DPO 擅长修**风格和安全**而非知识
- ORPO 优势:**只训一次 + 不加载 Reference Model 省显存**
