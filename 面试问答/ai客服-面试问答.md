# AI 实时语音对话智能客服 · 面试问答

> 基于《ai客服》项目文档整理的面试题库。项目是一个**基于火山引擎 RTC + RAG 的实时语音智能客服**,技术栈:React + WebRTC + Redux-Toolkit(前端)、Python FastAPI(AI 后台)、豆包 1.6/Ark 模型、VikingDB 知识库。
> 每题含「考察点」「参考回答」「加分项」,覆盖架构、RTC、RAG、LLM 性能优化、ASR/TTS 等高频考点。

---

## 一、项目与架构

### Q1. 简单介绍一下这个项目,它的整体调用链路是怎样的?

**考察点**:项目整体把握、端到端链路的清晰表达与亮点提炼。

**参考回答**:
这是一个**实时语音对话的智能客服系统**,典型场景是**电商商品咨询**(用户一边浏览商品、一边语音问客服)。

**核心交互**:通过 **RTC 开一个房间**,把**用户**和我们的 **AI Agent** 都拉进同一个房间里建立通信。用户有**两种交互方式——既可以说话,也可以打字**;其中**支持语音实时对话**是这个产品的核心特点。

**整体调用链路(端到端)**如下:

1. **语音输入 → ASR**:用户说话(比如"我要咨询一下这个产品"),先通过 **ASR 语音转文字**。
2. **RTC → Python 服务**:RTC 云端把识别出的文本**推送到我们的 Python(FastAPI)服务**,服务里有一个**callback 回调接口**专门接收,这样就拿到了用户的提问。
3. **进入 AI Agent 处理流程**,拿到提问后分几步:
   - **① 意图识别**:结合**会话上下文**,用大模型识别用户到底在咨询什么。这一步是有层次的——先做**产品分类**(比如是耳机 → 蓝牙耳机 / 有线耳机 → 具体型号),再结合用户当前行为(如"我正在浏览这个耳机")**定位到具体的商品 ID**。
   - **② 知识库 / LLM 检索**:定位到商品后,再检索用户**具体想问什么**(产品特征、参数、售后等),从知识库拿到相关内容,其中也会带出**商品 ID**。
   - **③ 查 Redis 拿实时数据**:拿着商品 ID 去 **Redis** 查**商品的实时信息、库存**等动态数据。
   - **④ 组装 Prompt → 大模型生成**:把检索到的知识 + Redis 的实时商品数据组织起来,通过**提示词**喂给大模型,生成回复。
4. **回复 → TTS → RTC 播放**:大模型返回的是**文本**,再通过 **TTS 转成语音**,最后通过 **RTC 通道**推送给用户播放。

一句话总结链路:**用户语音 → ASR → RTC 回调到 Python 服务 → 意图识别(产品分类+定位商品ID) → 知识库检索 + Redis 查实时库存 → 组装 Prompt 调大模型 → TTS → RTC 推送给用户。**

**加分项**:
- 能点出**意图识别是分层的**(先分类、再定位 ID),这是检索精准的关键前置。
- 能区分**静态知识(知识库/RAG)** 和 **动态数据(Redis 实时库存)** 两类数据源,并说明为什么库存这类要走 Redis 而不是知识库。
- 能补充技术亮点:**实时语音 + RAG + 全链路性能优化**,把复杂 RAG 场景下的首字延迟(TTFT)从 15s 压到 2s 以内。

### Q2. 什么是 AI 应用?它的运行流程是怎样的?

## 二、RTC 通信(核心考点)

### Q3. 为什么语音方案要用 RTC,而不是 WebSocket / HTTP?

**考察点**:RTC 选型的深度理解(最高频)。

**参考回答**:
四个核心原因:
1. **极低延迟(最核心)**:WebRTC 底层基于 **UDP**,语音通话中"实时性 > 完整性"。丢几个包 UDP 直接丢弃(短暂杂音),而 TCP(WebSocket/HTTP)必须按序到达,一旦丢包后续全部等待重传(队头阻塞 Head-of-Line Blocking)。在 AI 对话中,如果用户说"停",TCP 方案可能因重传延迟 1-2 秒,AI 还在喋喋不休,**完全破坏打断体验**。
2. **内置 3A 音频算法**:RTC SDK 内置强大音频引擎:
   - **AEC(回声消除)**:消除扬声器声音被麦克风录回去产生的无限回音;
   - **ANS(降噪)**:去除环境噪音;
   - **AGC(自动增益)**:自动调节麦克风音量。
   - WebSocket 只传二进制流,这些得自己手写(极难)。
3. **架构解耦**:RTC 方案下信令走 HTTP(把 AI 拉进房间),音频流**直接**从浏览器到 RTC 服务器再转给 AI,后端不处理音频转发,负载极低。Socket 方案音频要经过自己的后端中转,后端变成流量瓶颈。
4. **DataChannel 指令通道**:WebRTC 除了音视频还有 DataChannel,走 UDP 比 WebSocket 更快,专门发"打断""静音"等高优先级指令。

### Q4. 火山引擎 RTC 用的是什么架构?为什么能支撑多人?

**考察点**:SFU 架构理解。

**参考回答**:
用的是 **SFU(Selective Forwarding Unit,选择性转发单元)** 架构,本质就是**服务器转发,只是底层协议是 UDP**。
- **逻辑上像点对点**:业务层可以指定发给某个 `userId`。
- **物理上是星型结构**:你只发 **1 路**自己的流到火山边缘服务器,服务器根据订阅请求转发给房间内其他人。
- **优点**:本地上行始终是 1 路流,极大节省带宽,能支撑百人甚至千人会议。

### Q5. RTC 语音会话的核心步骤是什么?麦克风采集、字幕、AI 语音分别从哪来?

**考察点**:对语音全链路数据流的掌握。

**参考回答**:
房间建立:**创建 RTC 房间 → 我加入 → AI 加入**。然后:
| 环节 | 来源 / 位置 |
|------|------------|
| **麦克风采集** | RTC SDK 引擎层,`RtcClient.ts` 的 `startAudioCapture`,向 OS 请求权限并转数字音频流 |
| **发给 AI 的声音** | 以**实时流**推送,`publishStream(MediaType.AUDIO)`,AI Agent 作为"隐形监听者"在频道里拉流 |
| **我的字幕** | 不是前端本地识别,而是**云端 ASR 识别后回传**的二进制消息,`handleRoomBinaryMessageReceived` 接收,`handler.ts` 解析 |
| **AI 回复文字** | 云端 **LLM** 生成,同样走 RTC 信道消息,靠 `userId`(botName)区分显示在左侧 |
| **AI 回复语音** | 云端 **TTS** 合成,AI Agent 像真实用户一样发布音频流,因 `isAutoSubscribeAudio: true` 浏览器自动拉取播放 |

一句话总结:**声音流(推/拉)由 RtcClient 负责,文字流(指令解析)由 handler 负责,界面展示由 Conversation 订阅 Redux 输出。**

### Q6. AI 的回复语音在前端是怎么播放的?为什么代码里找不到 play()?

**考察点**:RTC 远端流订阅模式。

**参考回答**:
找不到手动 `play(audioBuffer)`,因为走的是 RTC 的**远端音频流订阅模式**,由 SDK 自动处理:
1. **自动订阅**:`joinRoom` 时设了 `isAutoSubscribeAudio: true`,只要房间有音频就自动播放。
2. **云端产生**:AI Agent 把文字 TTS 成语音,像真实远端用户一样发布音频轨道。
3. **前端感知**:`handleUserPublishStream` 监听到 AI 发布音频流,同步 UI(音波图)。
4. **自动播放限制**:浏览器禁止无交互自动播放,`handleAutoPlayFail` 监听 `onAutoplayFailed`,失败时提示用户点击屏幕恢复声音。

### Q7. 解析云端回传的消息时,`subv` 和 `conv` 两种消息有什么区别?

**考察点**:消息协议细节。

**参考回答**:
- **`subv`(Subtitle,字幕消息)**:对话的**文本内容**,含 ASR 识别的用户话或 LLM 生成的回复。关键字段:`text`、`definite`(是否最终结果)、`userId`(区分你/AI)、`paragraph`(是否换行)。驱动**文字显示**,`definite=false` 时呈现打字机流式效果。
- **`conv`(Conversation State,业务状态简报)**:对话的**逻辑状态**,不含聊天文字。关键字段 `Stage.Code`(如 `Code:3`=THINKING 思考中,`Code:5`=FINISHED 发言结束)。驱动 **UI 动画**,如收到 Code:5 停止音波动画、切到"请开始说话"。

**加分项**:能说出流控字段——`roundId`(对话轮次)、`sequence`(序列号,防网络抖动文字乱序)、`timestamp`。

## 三、SSE 与流式输出

### Q8. 什么是 SSE?为什么 AI 聊天都用 SSE?

**考察点**:流式输出技术。

**参考回答**:
**SSE(Server-Sent Events,服务器发送事件)** 是基于 HTTP 的**单向**通信技术,服务器有新数据时主动持续推送给客户端。ChatGPT、豆包文字一个个"跳"出来基本都靠它。

工作原理:
1. 客户端请求时 Header 带 `Accept: text/event-stream`;
2. 服务器响应头设 `Content-Type: text/event-stream`,不关闭连接;
3. 数据以 `data: ` 开头、`\n\n` 结尾持续推送。

**AI 聊天选 SSE 的原因**:
1. **轻量化**:基于现有 HTTP,无需额外配置;
2. **实时性**:每生成一个 Token 立即推送,体验好;
3. **防火墙友好**:本质是 HTTP 长连接,不易被拦截。

---

### Q9. SSE 和 WebSocket 有什么区别?

**考察点**:对比理解。

**参考回答**:
- **SSE**:基于 HTTP,**单向**(服务器→客户端),纯文本,自带断线重连(`retry` 字段),轻量。适合 AI 流式输出、消息推送。
- **WebSocket**:独立协议(需握手升级),**全双工双向**,支持二进制,功能强但更重。适合双向实时交互(如游戏、协同编辑)。

AI 聊天是"服务器单向推 Token"的场景,所以 SSE 更合适。

**加分项**:SSE 由 Ian Hickson(HTML5 规范主编)定义,2004 年由 Opera 提出原型,后纳入 HTML5;解决了轮询/长轮询浪费带宽、重复握手的问题。

## 四、RAG 架构改造

### Q10. 为什么要做 RAG 架构改造?改造后的流程是怎样的?

**考察点**:RAG 落地的工程改造能力。

**参考回答**:
火山官方 demo 是 **RTC 云端直接调 Ark 模型**,无法接入我们自己的知识库和定制 Prompt,完全不满足需求。改造后流程:
1. **ASR**:RTC 把语音转文本;
2. **中转**:RTC 云端通过"第三方大模型"配置把文本发给**我的 Python Server**;
3. **RAG 检索**:Python Server 调火山知识库(Ark Knowledge Base)检索相关文档片段;
4. **Prompt 构建**:从数据库读定制 Prompt,结合检索知识组装系统提示词;
5. **生成**:调 Ark(Doubao)生成回复;
6. **返回**:回复返回 RTC 云端;
7. **TTS**:RTC 云端转语音播放。

关键是把 LLM 调用从云端"抢"回自己的 Server,这样才能插入 RAG 和管控。

**加分项**:本地 Server 用 **ngrok** 做公网映射,供 RTC 云端 callback。

### Q11. 知识库、embedding、向量数据库三者是什么关系?

**考察点**:RAG 基础概念。

**参考回答**:
- **embedding 模型**:只负责把**文本变成向量**,是**无状态**的,不存数据。
- **向量数据库**(如 VikingDB):负责**向量存储**和检索。
- **知识库 = embedding 模型 + 向量数据库**。

选型:
- **非常复杂的项目(10 万级文档)**:自己组合 embedding + 向量数据库,如阿里云/腾讯云的云服务知识库检索。
- **一般项目(如智能客服)**:直接用现成知识库即可。

## 五、知识库 / 检索优化(项目亮点)

### Q12. 火山引擎默认怎么做文档切片?有什么问题?你怎么优化的?

**考察点**:Chunking 策略(高频亮点题)。

**参考回答**:
- **默认规则**:设一个切片长度(如 200)加 `chunk_overlap` 重叠长度,内部以**完整句子**为维度划分,避免半句话被切断。
- **问题**:可能把"一个问题对应的回答"截断成两半。
- **朴素办法**:调大 `Limit` 让大模型多拿几个结果——但不一定解决,还会让上下文变长、性能变慢、token 消耗变多。
- **我的优化**:用 `chunking_strategy` 自定义分隔符。智能客服场景按 **H1 标签**划分,搜索既精准又省 token、性能好。
- **代价**:需要人工维护知识库,**用人工换 AI 的智能程度**。
- **什么时候用长度拆分**:文档很散、很乱时(杂志、新闻)。

### Q13. 知识库的索引算法 `index_type` 有哪几种?怎么选?

**考察点**:向量索引原理。

**参考回答**:
- **flat**:像没加索引的 MySQL,**全表扫描**逐个计算向量是否接近。精度最高(接近精准搜索),但慢。
- **hnsw**:建立"导航地图",内存消耗大,**查询快**,适合语义查询。
- **hnsw_hybrid**:**混合检索**,即稀疏向量 + 稠密向量结合。

补充概念:
- **稀疏向量** = 带权重的传统全文检索(关键词精准);
- **稠密向量** = 纯语义查询,可能过度联想(问 Python 回 Java)。

**加分项(假设性原则)**:可以讲"早期用混合检索是两个数据库——查 MySQL + 查向量库;后来升级到 hnsw_hybrid,一个数据库同时解决精准 + 语义"。

  1. flat（暴力检索 / 全表扫描）

  - 怎么做：拿查询向量，和库里每一条向量都算一遍距离，挑最近的。
  - 特点：精度最高（一定能找到真正最近的，是"精确检索"），但最慢，数据量大就扛不住。
  - 像什么：没加索引的 MySQL，一行行 LIKE 扫。
  - 什么时候用：数据量小（几千条以内）、或对召回精度要求极高、能接受慢。

  2. hnsw（分层导航图，最常用）

  - 怎么做：提前把所有向量构建成一张多层"导航地图"——上层稀疏、跳得远，下层密集、定位准。查询时从上层快速跳到大致区域，再
    逐层下沉精确定位。
  - 特点：查询非常快（复杂度从 O(N) 降到接近 O(log N)），精度也很高（近似最近邻，召回
    95%+）。代价是内存占用大（要存图结构），且建索引慢。
  - 像什么：开车用导航，不用把全城每条路都开一遍。
  - 什么时候用：绝大多数语义检索场景的默认选择，数据量中大、要快、内存够。

  3. hnsw_hybrid（混合检索）—— 你问的重点

  - 一句话：在 hnsw 的基础上，同时结合「语义检索」和「关键词检索」两种能力。下面单独展开。

  选型口诀：

  ▎ 数据少求精确 → flat；数据多求快求语义 → hnsw；既要语义又要关键词精准命中（尤其有专有名词/型号/编号）→ hnsw_hybrid。

  三、什么是 hnsw_hybrid 混合检索（核心）

  要讲清楚它，先得理解两种「向量」代表的两种检索思路：

  ┌──────────┬───────────────────────────────────┬──────────────────────────────┐
  │          │         稠密向量（Dense）         │      稀疏向量（Sparse）      │
  ├──────────┼───────────────────────────────────┼──────────────────────────────┤
  │ 本质     │ 语义检索                          │ 关键词检索（升级版全文检索） │
  ├──────────┼───────────────────────────────────┼──────────────────────────────┤
  │ 怎么匹配 │ 按意思找，不在乎用词              │ 按词是否命中找，带权重       │
  ├──────────┼───────────────────────────────────┼──────────────────────────────┤
  │ 优点     │ 能理解同义、换种说法也能找到      │ 专有名词/型号/编号精准命中   │
  ├──────────┼───────────────────────────────────┼──────────────────────────────┤
  │ 缺点     │ 可能过度联想（问 Python 回 Java） │ 换个说法就找不到，不懂语义   │
  └──────────┴───────────────────────────────────┴──────────────────────────────┘

  hnsw_hybrid = 把这两种检索结合起来，同时跑、再融合打分。

  为什么需要它？ 看一个电商客服的例子（贴你的项目）：

  ▎ 用户问："RX-580 这款显卡还有货吗？"

  - 纯稠密（hnsw）：理解了"显卡""有货"的语义，但 "RX-580"
    这种型号是专有名词，语义模型不一定抓得准，可能给你返回一堆其他显卡。
  - 纯稀疏（关键词）："RX-580" 精准命中了，但如果用户换成"这个 580 的卡"，关键词又可能匹配不上。
  - hnsw_hybrid：稠密负责理解"找显卡库存"的意图，稀疏负责死死咬住"RX-580"这个型号 → 两者融合，既懂意思又精准命中型号 →
    结果最准。

  所以混合检索特别适合有大量专有名词、型号、编号、SKU 的场景——电商商品、法规条款、技术文档都是典型。

  四、混合检索里关键的调节参数：dense_weight

  混合检索要把两路结果"加权融合"，这个权重就是 dense_weight（稠密权重）：

  - dense_weight 调高 → 更偏语义，适合用户说话模糊、口语化（"那个能打游戏的卡"）。
  - dense_weight 调低 → 更偏关键词，适合用户提问里有精确型号/术语（"RX-580"）。
  - 一般 0.5 起手，再根据业务里用户的提问风格微调。（这正是你文档 Q14 里那条。）

### Q14. 知识库还有哪些可调参数?量化和维度怎么权衡?

**考察点**:检索精度/性能/成本权衡。

**参考回答**:
- **quant 量化**:`"int8"` 把 float32 压成 int8,**性能和内存提升 4 倍,精度损失不到 1%**。
- **embedding_dimension 维度**:维度高能理解隐晦表达;但智能客服里学生 90% 提问很直接,所以**降到 1000 维**就够,并配合 AI 对话引导用户提问。
- **limit(检索条数)**:1-3 条,合理控制以保证 LLM 响应速度。
- **field 标签过滤**:按文档标签查询,核心是**数据分类要做好**(如给课程打"0基础/有基础"标签,用户"java 出身要学多久"能精准命中)。
- **dense_weight 稠密权重**:0.5 起手,用户说话模糊就调高,语义直接就调低。

### Q15. Rerank 重排是做什么的?什么时候开,什么时候关?

**考察点**:重排机制理解(高频亮点)。

**参考回答**:
**Rerank** 是对初步检索结果**二次重排**,更好识别用户提问语义。
- **例子**:用户问"公司除了法定节假日,还有什么带薪假?"
  - **关闭**:Top1 可能是《法定节假日标准》(关键词重合度高),真正的《年假管理办法》排第 3 → AI 答非所问。
  - **开启**:重排模型识别出"除了...还有"的意图,把《年假管理办法》从第 3 拎到第 1 → 回答精准。
- **rerank 指数**:默认 25。标准 FAQ 推荐 20,复杂长文档/法规 30-50,极致追求速度 10-15。
- **rerank 模型**:智能客服推荐 `base-multilingual-rerank`(最快最轻、支持 70+ 语言)。
- **开启场景**:知识库复杂专业、用户提问逻辑复杂、轻性能重准确性。
- **关闭场景**:知识库结构简单、提问简单、重性能(关闭则需提高 Limit 数量保证准确)。

### Q16. Prompt Pilot 是怎么用的?对项目有什么价值?

**考察点**:提示词工程化管理。

**参考回答**:
写提示词有三种方式:自己写、让 AI(Gemini)生成、用 **Prompt Pilot**(更专业)。Prompt Pilot 的价值:
1. **建立"黄金标准"(Golden Dataset)**:每条评测集含 Prompt + Ideal Response,积累成业务"标准题库",换模型/改 Prompt 后批量跑分对比。
2. **自动化评分**:自动算输出与"理想回答"的相似度、准确性,不用人工逐条看。
3. **智能优化 Prompt**:攒够 10+ 条评测数据后点"智能优化",系统用强化学习/反思机制自动改写 Prompt。
4. **为 Fine-tuning 做准备**:高质量评测数据可直接转成 SFT 训练数据。

## 六、LLM 性能优化(核心亮点)

### Q17. 你是怎么把 LLM 推理时间从 15s 优化到 2s 的?

**考察点**:全链路性能优化(简历核心亮点,必问)。

**参考回答**:
分几步,层层递进:
1. **Prompt Cache(15s→10s)**:同样的提示词多次调用会缩短,利用 LLM API 的缓存机制。
2. **切换模型(15s→6-7s 再到 2-3s)**:`doubao-seed-1.6` → `Doubao-Seed-1.6-flash`。
3. **开启深度思考(反而更快)**:开/关思考背后资源池优先级不同;开启能更好利用 prompt cache;关闭没有思维链会一次性憋出结果,可能更慢。
4. **优化系统提示词(7s→2s)**:让**知识库风格和提示词人设风格统一**,极大减少推理过程。
5. **System Prompt 合并(2s→1.5s)**:把 RAG 检索结果合并进 system 提示词为 1 条,更好利用**前缀缓存**和 **KV Cache**。

**加分项**:前缀缓存优化技巧——RAG 多条结果**按首字母排序**(不按语义相似度,因为不稳定会导致顺序微变);会话 Message 带上历史消息也能利用前缀缓存。

- **图文混合知识库**:`Doubao-Seed-1.6-lite`,支持图片理解;

### Q19. 什么是批量推理?在项目里怎么用?

**考察点**:离线评测能力。

**参考回答**:
**批量推理(Batch Inference)** 适用于无需实时响应的场景,一次性离线处理大量数据,额度限制更高,适合模型评测、批量回归。在项目里:
- **产品**:收集用户高频提问、给理想答案、定评分标准;
- **开发**:每次调 LLM 参数或换模型后,拿提问表批量推理,和理想答案对比,让 LLM 自评分。

这就像传统开发的**单元测试**——改了核心代码就跑一次回归。

### Q20. 语音对话场景 max_tokens 怎么设?为什么?

**考察点**:语音场景特有约束。

**参考回答**:
语音对话建议 `max_tokens` 设 **150-300**:
- **听感极限**:人说话约每秒 4-5 字,150 tokens≈100-120 字朗读 20-30 秒,300 tokens 约 50-60 秒;AI 连说超 1 分钟用户就疲劳想打断。
- **首字延迟(TTFT)**:小的限制引导模型更快输出短句。

但**只限 max_tokens 不够**(可能话没说完被掐断,语音突然停止),要配合:
1. **Prompt 引导**:"你在语音通话,回复必须简短口语化,每句话 3 行以内,避免列表长句";
2. **代码参数**:给 Reasoning 留空间防截断报错;
3. **开启流式(Streaming)**:让 TTS 边接收文本边读,而不是等全部 token 跑完。

**加分项**:能解释上下文窗口(256k=输入+输出+推理)、最大输入(224k)、最大输出(64k)、TPM(每分钟 token 额度)、RPM(每分钟请求数)的区别。

### Q21. temperature = 0 和 0.3 有什么区别?客服场景怎么选?

**考察点**:采样参数理解。

**参考回答**:
- **Temperature = 0**:完全"复读"知识库,逻辑极稳,绝不出错,但**生硬**,像严格执行说明书的机器人。
- **Temperature = 0.3**:在保证事实准确前提下语言更自然,会加感叹词语气助词,核心数据(价格、日期)仍锁定在知识库范围内,像**热情的咨询老师**。

客服场景选 **0.3** 左右,兼顾准确性和对话感。

## 七、ASR / TTS / 弱网优化

### Q22. ASR 语音转文字你做了哪些优化?

**考察点**:语音识别工程优化。

**参考回答**:
1. **双向流模式**:语气词不发包;降低 RTF(识别时长/音频时长);首字时延 300ms、尾字时延 500ms、单包 100-200ms。
2. **情绪检测**:根据用户情绪回复不同话术,TTS 也带对应情绪。
3. **敏感词过滤**:过滤行业黑话等。
4. **热词设计**:用户说"皮果"实际想说"苹果",把专有名词(苹果、Agent、应用开发)设为热词;收集日志观察错别字,甚至用 LLM 分析高频错别字。

### Q23. TTS 文字转语音的优化点有哪些?1.0 和 2.0 架构有什么区别?

**考察点**:语音合成与端到端架构。

**参考回答**:
TTS 优化:双向流式版本(流式文本直接输入,比多次调用更自然情绪更饱满)、语速调节、混音、缓存、**音色复刻**、声音情绪。

架构演进:
- **1.0(级联)**:声音→ASR→文字→LLM→文字→TTS→声音。环节多,情绪会丢失。
- **2.0(端到端)**:声音→声音,直接端到端,情绪化更好、更优秀。

### Q24. RTC 弱网优化里 FEC、ARQ、HARQ 分别是什么?

**考察点**:弱网对抗策略(网络深度)。

**参考回答**:
- **FEC(前向纠错)**:"买份备份带身上"。发送时额外加冗余包,丢包时接收方用剩余包+冗余包**直接算出**丢失内容,无需重传,实时性最高,但占额外带宽。
- **ARQ(自动重传请求)**:"发现丢了立刻喊重发"。更省带宽(丢了才补),但增加延迟(往返一次),适合延迟不敏感或丢包率低的情况。
- **HARQ(混合 ARQ)**:FEC+ARQ 结合。收到错包不扔,存起来等重传包到了**合并解码**,提高重传成功率,是 5G 和高性能 RTC 标配。

火山 veRTC 综合用这些策略实现 **50% 丢包无感知恢复,最高 80% 抗丢包**。

---

### Q25. 自适应 Jitter Buffer 和自适应码率是干什么的?

**考察点**:缓冲与流控。

**参考回答**:
- **自适应 Jitter Buffer(抖动缓冲区)**:"快递中转站,攒齐按顺序发"。网络丢包导致包到达忽快忽慢(抖动),Jitter Buffer 先缓存、按序排队、匀速送给解码器。"自适应"指网络好缓冲区小(延迟低)、网络差缓冲区大(牺牲延迟换不卡顿)。
- **自适应码率下发**:"根据路况调车速"。检测到网络拥塞就主动降质量(降采样率/分辨率),宁可声音糊一点也要连续不中断。

## 八、工程与性能优化

### Q26. `__pycache__` 文件是什么?

**考察点**:Python 基础。

**参考回答**:
`__pycache__` 存放 **Python 编译后的字节码(Bytecode)文件**(`.pyc`)。运行 Python 时不是直接读 `.py` 源码,而是先翻译成机器更易理解的中间格式缓存起来,下次运行若源码没变就直接用缓存,加快启动。

### Q27. 你怎么把 Python 服务热更新从 10s 优化到 1-2s 的?

**考察点**:开发效率优化。

**参考回答**:
通过精细化 reload 排除策略:
1. **排除 `.pyc` 编译缓存文件**;
2. **排除 venv 依赖包目录**。

reload 时不再监听这些无关文件,重启耗时从 10s 降到 1-2s,大幅提升团队迭代效率。
