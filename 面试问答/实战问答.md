# 实战问答

## 1. 讲讲你对向量数据库的理解，了解哪些向量数据库，和传统数据库有什么区别，为什么比传统数据库快？

### 一、对向量数据库的理解

向量数据库（Vector Database）是一种专门用来**存储、索引和检索高维向量（Embedding）**的数据库。

在大模型 / RAG 场景里，文本、图片、音频等非结构化数据会先经过 Embedding 模型（如 BGE、text-embedding-3、OpenAI ada 等）转换成一个几百到几千维的浮点数向量，这个向量代表了数据在语义空间中的位置。语义相近的内容，向量之间的距离也更近。

向量数据库的核心能力就是做 **「相似度检索」（ANN，Approximate Nearest Neighbor，近似最近邻搜索）**：给定一个查询向量，快速从海量向量中找出 Top-K 个最相似的向量，相似度一般用**余弦相似度、内积、欧氏距离**来衡量。

它在工程中最典型的用途就是 RAG：把知识库切块 → 向量化 → 存入向量库；用户提问时把问题向量化 → 去库里检索最相关的若干片段 → 拼进 Prompt 交给大模型生成答案。

### 二、了解哪些向量数据库

常见的可以分几类：

**专用向量数据库：**
- **Milvus**：开源、分布式、能撑十亿级向量，云原生架构，工业界用得最多之一。
- **Pinecone**：全托管的商业云服务，免运维，开箱即用。
- **Qdrant**：Rust 写的，性能好、支持丰富的元数据过滤，部署轻量。
- **Weaviate**：自带模块化能力，支持混合检索（向量 + 关键词）。
- **Chroma**：轻量级，常用于本地开发和小型 RAG demo。

**传统数据库 / 搜索引擎的向量扩展：**
- **PgVector**：PostgreSQL 的向量插件，适合数据量不大、又想复用现有 PG 体系的场景。
- **Elasticsearch / OpenSearch**：支持 kNN 向量检索，适合需要向量 + 全文混合检索的场景。
- **Redis**：通过 RediSearch 模块支持向量检索，适合做低延迟缓存型检索。

选型上：要海量+分布式选 Milvus；要省事选 Pinecone；轻量本地用 Chroma；已有 PG 体系用 PgVector；要混合检索用 ES/Weaviate。

### 三、和传统数据库的区别

| 维度 | 传统数据库（MySQL 等） | 向量数据库 |
|------|------------------------|------------|
| 存储数据 | 结构化数据（数字、字符串、日期） | 高维向量（Embedding） |
| 查询方式 | 精确匹配 / 范围查询（=、>、LIKE） | 相似度检索（找「最像」的，而非「相等」的） |
| 查询逻辑 | 基于值是否相等 | 基于向量在空间中的距离远近 |
| 索引结构 | B+ 树、哈希索引 | HNSW、IVF、PQ 等 ANN 索引 |
| 返回结果 | 满足条件的精确结果 | Top-K 近似最相似结果（带相似度分数） |
| 典型场景 | 事务、报表、CRUD | 语义检索、RAG、推荐、以图搜图 |

一句话概括：**传统数据库回答「哪些数据等于 X」，向量数据库回答「哪些数据最像 X」。**

### 四、为什么比传统数据库快（在相似度检索这件事上）

要强调一个前提：**不是所有场景都快，而是在「高维相似度检索」这个特定任务上快。** 原因有三层：

**1. 核心是「近似」而非「精确」**
如果要找最相似的向量，最朴素的做法是暴力计算查询向量和库里每一个向量的距离（KNN 暴力检索），复杂度是 O(N)，一亿条数据就要算一亿次，根本扛不住。

向量数据库用的是 **ANN（近似最近邻）**：用一点点精度的损失（可能不是 100% 最优，但 95%+ 召回），换来几个数量级的速度提升。这是它快的本质——**用近似换速度**。

**2. 专门的索引结构（关键）**
- **HNSW（分层可导航小世界图）**：构建一个多层图，检索时从上层稀疏图快速定位大致区域，再逐层下沉精确定位，复杂度从 O(N) 降到接近 **O(log N)**。这是目前最主流的索引。
- **IVF（倒排文件索引）**：先对向量做聚类分成很多桶（簇），查询时只在最接近的几个簇里搜，避免全量扫描。
- **PQ（乘积量化）**：把高维向量压缩编码，大幅降低内存占用和距离计算开销。

而传统数据库的 B+ 树是为一维可排序的标量设计的，**根本无法对高维向量做有效的距离检索**——高维空间没有天然的「大小顺序」，B+ 树在这里失效。

**3. 工程层面的优化**
向量库针对向量计算做了大量优化：SIMD 指令加速距离计算、向量量化压缩、GPU 加速、分布式分片并行检索等。

**总结一句话：** 向量数据库快，不是因为它「魔法」，而是因为它放弃了「精确」，改用 HNSW / IVF 这类专为高维向量设计的 ANN 索引，把 O(N) 的暴力扫描降到接近 O(log N)，再叠加 SIMD、量化、分布式等工程优化。传统数据库的 B+ 树索引天生处理不了高维向量的相似度检索，所以在这个特定场景下被向量库甩开。

## 2. 你的项目是多模态的，如果用户同时发图片和文字请求，你是怎么处理的？怎么让它识别更准？

### 一、先讲清楚整体思路

用户同时发「图片 + 文字」，本质上是一个**多模态理解任务**：文字往往是用户的「意图 / 问题」，图片是「上下文 / 证据」。我的处理思路是——**让两个模态各司其职，再在合适的层面融合，最后用工程手段兜住准确率。**

整体链路大致是：

```
用户输入（图片 + 文字）
   ↓ 预处理（图片压缩/格式归一、文字清洗）
   ↓ 多模态大模型理解（VLM，如 GPT-4o / Qwen-VL / Claude）
   ↓ 必要时结合 RAG 检索补充知识
   ↓ 结构化输出 / 生成回答
```

### 二、具体怎么处理（分步骤）

**1. 输入预处理**
- 图片：做格式归一（统一转 JPEG/PNG）、分辨率压缩（控制 token 成本，但保留关键细节，比如涉及文字的图不能压太狠）、必要时做旋转矫正。
- 文字：清洗无关字符、补全上下文（比如把多轮对话的历史带上）。
- 关键一步：**判断用户真实意图**——是「让我描述这张图」，还是「针对图里某个东西回答问题」，还是「图 + 文一起构成一个完整问题」。意图不同，后面的处理策略不同。

**2. 模态融合的两种做法**

这里是回答的重点，要讲清楚两条技术路线：

- **方案 A：用多模态大模型（VLM）直接端到端理解（我项目主要用的）**
  直接把图片和文字一起喂给多模态大模型（如 GPT-4o、Qwen-VL、Claude 这类原生支持图文输入的模型）。模型内部已经把图像编码（ViT 等视觉编码器）和文本在同一语义空间对齐了，能直接「看图答文」。
  优点：理解能力强、开发简单、对复杂图文关系处理得好。

- **方案 B：图先转文，再走纯文本链路（兜底 / 特定场景）**
  先用专门模型把图片信息抽取成文本——比如用 OCR 提取图里的文字，用图像描述（Caption）模型生成图片描述，用目标检测识别图里的物体——然后把「图片转出来的文本 + 用户文字」一起送进文本大模型或 RAG。
  优点：可控、可解释、便于结合知识库检索；适合图里关键信息是文字（票据、表格、文档截图）的场景。

实际项目里我常常是**两者结合**：VLM 做主理解，OCR / 检测做关键信息的精确抽取来「校准」VLM 的输出。

**3. 结合 RAG 补充领域知识**
如果是垂直领域（比如商品、医疗、特定文档），光靠 VLM 的通用知识不够。我会把图文理解出来的关键信息（或图片向量、文本向量）拿去**向量库检索**，把检索到的领域知识拼进 Prompt，再让模型生成答案。这样能避免模型「凭空编」。

### 三、怎么让它识别更准（这是面试官最想听的）

我会从**输入、模型、融合、后处理、评估**五个层面来提准确率：

**1. 输入质量层面**
- 图片预处理保真：涉及文字 / 细节的图，控制压缩比，避免关键信息丢失。
- Prompt 明确化：告诉模型「请重点看图片的哪个区域」「先识别再回答」，引导它聚焦。

**2. 模型 / Prompt 层面**
- 选对模型：识别文字密集型图（票据、文档）优先上 OCR + VLM；理解场景语义型图用强 VLM。
- 用 **CoT（思维链）/ 先描述再回答**：让模型先把「我从图里看到了什么」描述出来，再基于描述回答，能显著降低幻觉。
- Few-shot 示例：给几个图文问答的范例，规范输出。

**3. 多模型 / 多模态交叉校验**
- **OCR 校准 VLM**：VLM 读图里的数字 / 文字容易错位，用 OCR 的精确结果去校正。
- **目标检测 + VLM**：先检测定位，再让 VLM 针对框出来的区域回答，减少「看错对象」。
- 关键场景做**多次采样投票**或多模型结果对比，取一致结果。

**4. 后处理 / 结构化约束**
- 让模型按 **JSON Schema 等结构化格式输出**，字段级约束，便于校验和纠错。
- 加规则校验：比如金额、日期、身份证号等用正则 / 业务规则二次校验，不合规就回退或重试。
- 置信度低时**触发兜底**：让模型说「无法确定」而不是硬编，或转人工 / 追问用户。

**5. 评估与迭代（体现工程闭环）**
- 建立标注好的图文测试集，用准确率 / 召回率持续评估。
- 收集线上 badcase，针对性优化 Prompt、补充检索知识、微调模型。

### 四、总结一句话

我的处理核心是：**意图先行、模态分工、融合理解、检索补知识、多手段校验**。让 VLM 负责图文语义理解，OCR / 检测负责精确信息抽取，RAG 负责领域知识，再用结构化输出 + 规则校验 + 低置信兜底把准确率稳稳兜住，最后靠 badcase 闭环持续迭代。

## 3. 什么是 LangChain 和 LangGraph？

### 一、一句话先讲清楚

- **LangChain**：一个用来**构建大模型应用的开发框架**，把调用大模型这件事里反复要用的能力（调模型、写 Prompt、接向量库做 RAG、调工具、记忆对话等）封装成标准化组件，让你像「搭积木」一样快速搭出 LLM 应用。
- **LangGraph**：LangChain 团队推出的、用来**编排复杂 Agent 工作流的框架**。它把应用抽象成一张**图（Graph）**——节点是步骤、边是流转，天生支持**循环、分支、状态管理、人工介入**，专门解决 LangChain 链式结构「拐不了弯、转不了圈」的问题。

一句话区别：**LangChain 擅长把流程「串成一条链」，LangGraph 擅长把流程「画成一张能循环能分支的图」。**

### 二、LangChain 详解

**1. 它解决什么问题**
直接调大模型 API 时，RAG、记忆、工具调用、多步流程这些都得自己手写一遍，重复又繁琐。LangChain 把这些抽象成统一组件，并能把它们**串联（Chain）**起来。

**2. 核心组件（面试常考）**
- **Model（模型）**：统一封装各家大模型接口（OpenAI、Claude、通义等），换模型只改配置。
- **Prompt（提示词模板）**：把 Prompt 模板化，支持变量填充、Few-shot。
- **Chain（链）**：把多个步骤串成一条流水线，前一步输出是后一步输入。
- **Retriever / VectorStore（检索）**：对接向量库，是搭 RAG 的核心。
- **Memory（记忆）**：保存多轮对话历史，让对话有上下文。
- **Tools / Agent（工具与智能体）**：让模型能调用外部工具（搜索、计算器、数据库、API），由模型自己决定调哪个。
- **LCEL（LangChain 表达式语言）**：用 `|` 管道符把组件优雅地拼起来，支持流式、并行、异步。

**3. 典型用法**
搭 RAG：`文档加载 → 切分 → 向量化入库 → 检索 → 拼 Prompt → 调模型 → 输出`，这一整套用 LangChain 的组件几行就能串起来。

**4. 局限**
LangChain 的 Chain 本质是**有向无环、单向流动**的——适合「固定步骤、走一遍就完」的流程。但真正的 Agent 经常需要**循环**（反复思考-行动-观察直到完成）、**条件分支**（根据结果决定下一步）、**回退重试**，这些用纯 Chain 写会非常别扭。于是有了 LangGraph。

### 三、LangGraph 详解

**1. 核心思想：把应用建模成「状态图」**
- **State（状态）**：一个贯穿全流程的共享状态对象，每个节点读它、改它、传下去。这是 LangGraph 的灵魂。
- **Node（节点）**：一个具体的处理步骤（调一次模型、调一个工具、做一次判断），输入输出都是 State。
- **Edge（边）**：节点间的流转关系。普通边是固定走向；**条件边**根据 State 内容动态决定下一个去哪个节点——这就实现了分支和循环。

**2. 它强在哪（相比 LangChain）**
- **支持循环**：可以让 Agent「思考 → 调工具 → 看结果 → 再思考」反复迭代，直到满足条件才退出（典型的 ReAct、反思 Reflection 模式）。
- **支持条件分支**：根据中间结果走不同路径。
- **状态可控**：全局 State 让多步、多 Agent 之间的数据流转清晰可追踪。
- **Human-in-the-loop（人工介入）**：可以在图的某个节点暂停，等人工审核 / 修改后再继续，适合高风险场景。
- **持久化与可恢复**：支持断点续跑、记忆持久化。
- **多 Agent 协作**：天然适合编排多个 Agent 分工协作（如主管 Agent 调度多个子 Agent）。

**3. 典型场景**
复杂 Agent、需要反复调工具的任务、多 Agent 系统、需要人工审核的工作流、带反思 / 自我纠错的流程。

### 四、两者关系与选型

- **关系**：LangGraph 不是要取代 LangChain，而是**互补**。LangGraph 负责「上层的流程编排（图）」，节点内部具体调模型、做检索、用工具这些活，照样可以用 LangChain 的组件来干。两者经常一起用。
- **选型**：
  - 流程**简单、线性、走一遍就完**（如标准 RAG 问答、文本总结）→ 用 **LangChain** 就够，开发快。
  - 流程**复杂、要循环 / 分支 / 多 Agent / 人工介入 / 可控状态**（如自主 Agent、工作流编排）→ 用 **LangGraph**。

### 五、总结一句话

**LangChain 是「LLM 应用的组件工具箱 + 链式编排」，让你快速搭出 RAG、Agent 等应用；LangGraph 是「基于状态图的高级编排框架」，用节点和条件边支持循环、分支、状态管理和人工介入，专门用来构建复杂、可控的 Agent 工作流。简单线性流程用 LangChain，复杂有状态、要循环的 Agent 用 LangGraph，二者互补、常配合使用。**
