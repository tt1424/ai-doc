# 基于 MCP 协议的多 Agent 共享服务集群 · 面试问答

> 基于《agent集群》项目文档整理的面试题库。项目背景:一家公司有多个 AI 业务系统(语音客服、生图、写作等)和大量 MCP 服务,需要把它们打通成统一的「Agent 集群」。核心是用 **MCP 协议**把 LLM 网关、RAG、记忆、Prompt 抽离成共享服务,业务项目经网关统一接入,并部署到阿里云 Serverless。
> 每题含「考察点」「参考回答」「加分项」,覆盖集群架构、MCP 协议、路由/记忆/可观测/安全四大支柱、云部署与网络。

---

## 一、背景与整体架构

### Q1. 为什么要做 Agent 集群?它解决什么问题?

**考察点**:对架构动机的理解。

**参考回答**:
背景是:未来所有软件公司都要接入 AI、所有数据库都要结合向量库做语义化。一家公司往往有很多系统(几十个 ERP + 多个 AI 系统),这些 AI 系统**目前是孤立的**——各自配一堆 key、各自实现 RAG/记忆/LLM 调用,重复造轮子且互不相通。

Agent 集群就是把它们打通,架构分两部分:
1. **MCP 集群(公共服务)**:LLM 网关、RAG 检索、记忆、Prompt 管理等通用能力抽成独立 MCP Server;
2. **AI 项目业务集群**:各业务 Agent 只写自己的编排逻辑,通过网关复用公共服务。

价值:**新业务接入只需写 Agent 编排层**,基础设施全部复用,像调组件库一样。

---

### Q2. 当 MCP 服务有几十个时会遇到什么问题?怎么解决?(路由调度层)

**考察点**:四大支柱之一——Router Layer(高频)。

**参考回答**:
只有 3 个 MCP 服务时 Agent 还能手动连;但有 30/80 个时,**没法把所有工具说明塞进 Prompt(Context Window 会爆)**。

解决:加一个 **Agent Router(路由调度层)**,根据用户需求**动态选择加载哪几个 MCP Server**。比如用户提到"画图",Router 才把 `Image-MCP` 的接口描述喂给当前业务 Agent。

价值:**节省 Token 成本、提升响应速度**(不把无关工具喂给模型)。

---

### Q3. 不同业务 Agent 之间记忆是孤立的,怎么打通?(全局记忆)

**考察点**:四大支柱之二——Global Memory。

**参考回答**:
问题:用户在"语音客服"提到的学习进度,打开"生图 Agent"做教案时后者并不知道。

解决:建一个独立的 MCP 资源服务专门管"用户记忆",即 **User-Centric Profile(用户画像中心)**:
- **短期记忆**:跨项目的 Session 共享;
- **长期记忆**:用户教育背景、偏好风格、历史纠错记录。

价值:让所有 AI 产品看起来像**统一的 AI 助手集群**,而不是一堆互不认识的工具。

---

### Q4. MCP 架构下请求链路变长,出问题怎么定位?(可观测性)

**考察点**:四大支柱之三——Observability。

**参考回答**:
MCP 架构链路是 `用户 → 业务 Agent → MCP Proxy → MCP Server → 大模型/工作流`,报错或变慢时定位很痛苦。

解决:**链路追踪(Tracing)**:
1. **日志对齐**:所有业务 Agent 和 MCP 服务带上同一个 `trace_id`;
2. **评估系统**:引入 LangSmith / Arize Phoenix,监控每个 MCP 节点的耗时和 Token 消耗。

价值:能回答"为什么这通电话延迟了 3 秒""哪个 MCP 服务最费钱"。

---

### Q5. 多项目共用 MCP,怎么做权限和成本控制?(安全隔离)

**考察点**:四大支柱之四——Security & Multitenancy。

**参考回答**:
作为公司层面要考虑不同项目的权限:
- **访问控制(AuthZ)**:不是所有 Agent 都能调"生图 MCP"(很贵),需要**网关层校验** `Agent_A` 是否有权调 `Server_B`;
- **资源配额**:给每个项目设 Token / GPU 调用上限,防止某个实验性项目把公司 API 额度瞬间刷爆。

**加分项**:这就是多租户(Multitenancy)的核心——按 project_id 维度做认证 + 配额计量,确保安全隔离与成本管控。

## 二、MCP 协议与传输

### Q6. SSE 和 Streamable HTTP 有什么区别?MCP 为什么选 Streamable HTTP?

**考察点**:MCP 传输协议(高频)。

**参考回答**:
- **SSE(单向大喇叭)**:服务器往客户端**单向推数据**,像听广播/看直播——客户端只能听,没法顺着喊回去。ChatGPT 吐字就是典型 SSE。
- **Streamable HTTP(双向对讲机)**:基于 HTTP 的**双向实时聊天**,在**同一个连接**里客户端可以不断给服务器发指令。

**MCP 选 Streamable HTTP 的原因**:MCP 需要 Client 和 Server 频繁"切磋"(Client 问有什么工具→Server 答→Client 让算 1+1→Server 答 2)。如果用 SSE,一问一答就断开,下次调用又要重新握手、验证身份,太慢。Streamable HTTP 像修了条"双向高速专用车道":
1. **省事**:握手一次,会话期间终身受益;
2. **快**:AI 反复确认、多次调用的场景,长连接效率最高。

---

### Q7. 为什么要把能力抽象成通用 MCP 服务?好处是什么?

**考察点**:服务化设计思想。

**参考回答**:
最大好处:**不用配一大堆 key,直接访问地址就能用**,像组件库一样复用。
具体做法是把 LLM 网关、RAG、记忆、Prompt 抽成 4 个独立 MCP Server,通过 Streamable HTTP 暴露标准化的 **Tool / Prompt 原语**,上层业务经 HTTP 网关统一接入。这样**业务和基础设施彻底分离**,新项目只写 Agent 编排层。

---

### Q8. LLM Gateway(网关)的作用是什么?

**考察点**:统一网关设计(简历核心亮点)。

**参考回答**:
**项目里所有大模型调用都走这个 MCP 服务**。作用:
- **统一封装** 豆包 ARK API(OpenAI 兼容);
- **模型路由**:逻辑模型名(auto/pro/lite)映射到具体 Endpoint,通过 YAML 配置**热切换模型,业务层无感**;
- **Embedding 兼容**:支持火山 text 与 vision 两种向量化路径;
- **配额计量**:Token 消耗统一回传网关配额模块扣减。

价值:换模型、调路由策略、成本核算全部集中在一个地方,业务代码不用动。

## 三、核心服务设计(简历亮点深挖)

### Q9. REST → MCP 的桥接网关怎么设计的?

**考察点**:网关与中间件链。

**参考回答**:
基于 FastAPI 构建**四层中间件链**:**认证 → 日志 → 配额 → 路由**。
- 通过 **Trace ID** 实现全链路追踪;
- 内置 **MCP Client Manager** 管理 Streamable HTTP 短连接;
- 按 **project 维度**做 API Key 认证 + Token 配额计量,保证多租户安全隔离与成本管控。

上层业务发普通 REST 请求,网关把它桥接成 MCP 协议调用,业务侧无需感知 MCP 细节。

---

### Q10. 多租户 RAG 是怎么做隔离的?

**考察点**:多租户知识库设计。

**参考回答**:
基于火山引擎 **Viking DB**,**按 `project_id` 划分独立 Collection 实现知识库物理隔离**——A 项目检索不到 B 项目的知识。
- 文档按 **Markdown 标题层级 + 段落边界**语义切分,避免跨主题截断;
- 检索侧支持 top-k 语义召回;
- 新项目调用 `ingest_document` Tool 即可**自助接入**,不用改服务端。

---

### Q11. 双层记忆架构是怎么设计的?怎么保证写入幂等?

**考察点**:记忆系统与幂等(高频)。

**参考回答**:
**双层记忆**:
- **对话记忆**:按 `project_id + session_id` 隔离,支持最近 N 轮召回与会话清空;
- **用户画像**:按 `user_id` **跨项目共享**,基于 PostgreSQL **`UPSERT`(ON CONFLICT DO UPDATE)实现幂等写入**(同一用户重复写不会产生重复记录)。

联动:客服 Agent 用 LLM 自动提取用户事实(过敏/预算/偏好)写入画像,写作 Agent 即时读取,**实现跨项目用户认知同步**。

---

### Q12. Prompt 模板中心是怎么做的?

**考察点**:Prompt 工程化解耦。

**参考回答**:
基于 **MCP 原生 Prompt 原语**构建模板中心:
- 用 **YAML 定义参数 Schema**,服务端渲染后以 `PromptMessage` 结构返回;
- Agent 按需组合**画像 + 检索片段 + 对话历史**装配 Prompt;
- **模板修改即时生效无需重启**,实现提示词工程与业务逻辑解耦。

---

### Q13. Agent 的编排流水线是怎样的?为什么 Agent 不持有状态?

**考察点**:无状态设计思想(核心)。

**参考回答**:
**七步编排流水线**:召回历史 → 读取画像 → 检索知识 → 渲染 Prompt → LLM 推理 → 保存记忆 → 提取事实。全部通过 GatewayClient 统一调用。

**Agent 不持有状态**:所有状态(记忆、画像、知识)都**下沉到共享 MCP 服务**。好处:
- 业务与基础设施彻底分离;
- Agent 是无状态的,**契合容器化、可任意横向扩展**(任何一个实例都能处理任何请求)。

## 四、部署与网络

### Q14. AI 服务的后端部署/CICD 流程是怎样的?

**考察点**:CICD 与容器化。

**参考回答**:
后端走 CICD:
1. GitLab 仓库 push 代码;
2. 触发钩子(项目里的 shell 脚本):打包镜像 → 上传阿里云镜像仓库;
3. 部署到云函数 / SAE。

镜像可本地打(Dockerfile.aliyun)再 push 给阿里云。各服务独立构建 Docker 镜像,**按依赖拓扑分层发布(MCP 服务 → 网关 → 业务项目)**,单服务独立更新实现灰度发布与秒级回滚。

**加分项**:SIGTERM 信号触发**优雅终止**,确保在途请求不丢失。

---

### Q15. 为什么用 Serverless(云函数/SAE)?和传统 ECS 比有什么不同?

**考察点**:Serverless 理解。

**参考回答**:
**Serverless 核心是含量计费**——按流量、CPU 计算量付费,**自动弹性扩容**(写规则、充钱即可,不用像 K8s 那样运维手动扩)。

对比:
- **传统 ECS**:买一台固定配置的服务器(如 500/年),流量小时浪费、流量大时扛不住要手动升级;
- **Serverless(SAE/函数计算)**:流量大自动分配资源,流量小按量付费,不操心底层硬件。

适合流量波动大的 AI 业务。

---

### Q16. 正向代理和反向代理有什么区别?

**考察点**:网络基础(高频必问)。

**参考回答**:
- **正向代理(Forward Proxy)**:**代表客户端**去访问,**服务器不知道真正的客户端是谁**。比喻:未成年小明让大哥哥帮他买烟,老板只知道卖给了大哥哥。用途:科学上网、隐藏真实 IP、公司内网访问外网控制。
- **反向代理(Reverse Proxy)**:**代表服务器**接收请求,**客户端不知道背后是哪台服务器**。比喻:拨 10086,你不知道接电话的是北京张三还是上海李四。用途:负载均衡、保护服务器真实 IP、SSL 卸载。

**加分项**:Nginx 作为反向代理还能做负载均衡(谁闲给谁)、动静分离(静态文件自己秒回)、虚拟主机(一台机器跑多个域名)。

---

### Q17. CLB(云负载均衡)和自建 Nginx 有什么区别?

**考察点**:云产品 vs 自建权衡。

**参考回答**:
四个核心差异:
1. **高可用**:自建 Nginx 那台 ECS 挂了整站断网(要自己配 Keepalived 主备);CLB 底层是集群,跨可用区容灾、流量自动漂移、秒级切换机房。
2. **自动扩容**:自建 Nginx 流量翻倍可能 CPU 爆满宕机要手动升配;CLB 弹性,流量大自动分配资源。
3. **云生态集成**:弹性伸缩新增服务器 CLB 自动加白名单(Nginx 要手改配置重启)、一键挂 SSL 证书、成熟的健康检查(故障实例自动踢出/恢复拉回)。
4. **安全**:CLB 天然集成 DDoS 基础防护、可一键开 WAF 过滤 SQL 注入。

---

### Q18. 什么是 NAT 网关?SNAT 和 DNAT 分别干什么?

**考察点**:云网络。

**参考回答**:
**NAT 网关**是云上私网服务器(ECS)的"上外网/被外网访问"企业级公共出口,像没窗户办公楼的"传达室"。
- **SNAT(让服务器上外网)**:多台没有公网 IP 的内网服务器通过同一个 NAT 公网 IP 访问互联网(下载补丁、调外部 API)。好处:隐藏后端真实位置、省钱(不用每台买公网 IP)。
- **DNAT(让外网找进来)**:设规则,访问 NAT IP 的 8080 端口自动转发到内网 A 服务器的 80 端口。比直接暴露服务器更受控安全。

### Q19. 前端是怎么部署的?

**考察点**:前端部署链路。

**参考回答**:
1. `npm run build` 打包出 HTML/CSS/JS(也可走 CICD);
2. 上传到 **OSS(源站)**;
3. 配 **CDN(内容分发)** 加速。

配合后端:CLB 负载均衡(SAE 提供内网地址 → CLB 暴露公网 IP)、域名 DNS 解析到公网 IP、CLB 里配免费 HTTPS 证书(**火山引擎 RTC 语音服务必须 HTTPS**)。

## 五、AI 编程经验

### Q20. 你在项目中用 AI 编程,碰到过什么解决不了的问题?

**考察点**:对 AI 辅助编程边界的认知(文档原面试题)。

**参考回答**:
碰到过。典型例子:用 **embedding 模型**时,因为模型是"训练即封印"的——它的知识停在训练那一刻,你必须**主动把最新的文档信息告诉 AI**,否则它凭旧知识写,bug 永远解决不掉。
所以我的体会是:**Claude Code 能解决 99% 的问题,但 1% 必须人工介入**——尤其是涉及模型本身不知道的新 API、新文档时,得人把准确信息喂给它。

## 附:高频速记要点

- **集群两部分**:MCP 集群(公共服务)+ AI 业务集群
- **四大支柱**:路由调度(动态加载省 token)/ 全局记忆(用户画像中心)/ 可观测(trace_id + LangSmith)/ 安全隔离(AuthZ + 配额)
- **MCP 传输**:SSE 单向(大喇叭)vs Streamable HTTP 双向(对讲机);MCP 选后者——握手一次复用、多次调用快
- **抽象 MCP 的好处**:不用配一堆 key,访问地址即用,像组件库
- **LLM 网关**:所有模型调用走它,YAML 热切换、配额计量
- **核心服务**:RAG 按 project_id 物理隔离;记忆双层(对话 project+session / 画像 user_id 跨项目)+ UPSERT 幂等;Prompt 原语模板中心
- **Agent 无状态**:状态全下沉到 MCP,契合横向扩展
- **正向代理**(服务器不知客户端)vs **反向代理**(客户端不知服务器);Nginx 还做负载均衡/动静分离
- **CLB vs 自建 Nginx**:高可用/自动扩容/云集成/安全防护全面胜出
- **NAT**:SNAT(内网上外网)/ DNAT(外网转发进内网)
- **Serverless 含量计费 + 自动扩容**;前端 build→OSS→CDN;RTC 必须 HTTPS
- **AI 编程边界**:模型训练即封印,新文档要主动喂;Claude Code 解决 99%,1% 人工介入
