# 第一部分:项目经验逐条详解

## 项目一、AI 实时语音对话-智能客服(2025.07~08)

> **项目一句话**:基于火山引擎 RTC 的实时语音电商客服,用户语音/打字提问 → ASR → RAG 检索 → LLM 生成 → TTS 语音回复。亮点是 RAG 检索优化 + LLM 推理提速(TTFT 15s→2s)+ 语音链路弱网优化。

### 条目 1:知识库消费数据中台 RAG 数据,自定义标识符切片解决语义截断,HNSW_HYBRID 混合检索 + Int8 量化,Metadata 标签过滤,存储降低 75%

**这条在说什么**:你的知识库数据不是自己临时整理的,而是直接用「项目三 AI 数据中台」清洗导出的标准 RAG 数据(体现项目间打通)。检索上做了三件优化:① 切片时用**自定义分隔符**(而非按固定长度),避免把"一个问题对应的答案"从中间切断(语义截断);② 用 **HNSW_HYBRID 混合检索**(稠密语义 + 稀疏关键词)兼顾语义和专有名词;③ 用 **Int8 量化**把向量从 float32 压成 int8,存储和内存降到约 1/4,所以"存储空间降低 75%"。Metadata 标签过滤 = 给文档打标签(如商品分类),检索时先按标签缩小范围。

**可能的提问与答:**

- **Q:存储降低 75% 怎么来的?**
  A:主要来自 Int8 量化——向量原本是 float32(每维 4 字节),量化成 int8(每维 1 字节),存储和内存占用降到 1/4,即降低约 75%,而精度损失不到 1%。
- **Q:什么是语义截断?自定义标识符怎么解决?**
  A:默认按固定长度切片,可能把一个完整的"问题+答案"切成两段,检索时只命中半段,回答不完整。我按文档结构里的标识符(如标题 H1 标签)切,保证每个切片是一个语义完整的单元。代价是需要人工维护知识库结构,用人工换检索精度。

### 条目 2:接入多模态素材中心,命中图片/视频后以「文字+图片+视频」混排分条返回

**这条在说什么**:客服不只回文字。当检索命中了「项目五多模态素材中心」里的图片/视频素材时,回复会以"文字+图+视频"混合排版、分条返回,让语音/图文客服具备多模态应答能力。这条体现你项目五和项目一的联动。

**可能的提问与答:**
- **Q:图片/视频是怎么被检索命中的?**
  A:关键在素材中心的设计——不向量化媒体本身,而是向量化它的「检索文本」(描述这张图/视频能回答什么问题),和用户的文本提问在同一语义空间对齐,所以能被命中。(详见项目五)
- **Q:为什么要分条返回、限制数量?**
  A:统一应答策略是"文字必发、图≤3、视频≤1",避免一次塞太多素材刷屏、影响体验,也控制成本。

### 条目 3:语音场景降维 + 精简 Limit,引入 base-rerank 轻量模型,动态平衡 dense_weight,检索速度提升 4 倍

**这条在说什么**:语音对话对延迟极敏感,所以为语音场景专门做检索提速:① **降维**(如 embedding 维度从高降到 1000 维),距离计算和内存减半;② **精简 Limit**(检索条数控制在 1-3 条),少给 LLM 喂内容;③ 用 **base-rerank 轻量重排模型**(最快最轻)做精排;④ 动态调 **dense_weight**(稠密权重)——用户说话模糊调高偏语义,提问含专有名词调低偏关键词。综合让检索速度提升 4 倍,同时保住专有名词命中率。

**可能的提问与答:**
- **Q:降维为什么能提速?会不会掉精度?**
  A:向量距离计算量和维度成正比,维度减半,计算和存储约减半。会损失一点语义表达力,但客服场景用户提问大多直接、不隐晦,1000 维够用,这个 trade-off 划算。
- **Q:dense_weight 怎么动态调?**
  A:它控制稠密(语义)和稀疏(关键词)在混合检索里的融合比例,0.5 起手。用户口语化模糊就调高靠语义,提问里有精确型号/术语就调低靠关键词命中。

### 条目 4:极致优化 LLM 推理——固定 RAG 排序、合并 System Prompt,触发 Prefix Caching + KV Cache,TTFT 15s→2s;模型选型与资源池优先级调优

**这条在说什么**:这是简历核心亮点。把复杂 RAG 场景的首字延迟(TTFT)从 15s 压到 2s 内,靠几招:① **固定 RAG 检索片段排序**(按首字母等固定规则,而非每次抖动的相似度排序),保证 prompt 前缀逐 token 一致,才能命中前缀缓存;② **合并 System Prompt**(把 RAG 结果合并进 system 成一整段固定前缀);③ 由此触发 **Prefix Caching(跨请求复用公共前缀的 KV)+ KV Cache**,跳过重复 prefill;④ 配合模型选型(换 flash 模型)和资源池优先级调优。

1.准确性2.回复速度   ：1.模型的迭代，flash最快，如果知识库的性格和提示词的性格不一致则影响速度2.提示词排序，最常用的放前面，利用缓存 3.大模型的深度思考，资源池优先



**可能的提问与答:**

- **Q:为什么 RAG 排序要固定?**
  A:前缀缓存命中要求前缀逐 token 完全一致。如果 RAG 结果按相似度排序,分数微小波动就会让顺序变化、前缀对不上、缓存失效。按首字母等固定规则排,顺序稳定,缓存才能命中。
- **Q:KV Cache 和 Prefix Caching 区别?**
  A:KV Cache 是单次生成内部缓存已生成 token 的 K/V,避免逐字重算前文;Prefix Caching 是它的跨请求版本,把多次请求共享的固定前缀(如 system prompt)的 KV 复用,专门降 TTFT。(详见技能模块四)

### 条目 5:优化 RTC 语音链路(ASR/TTS)——ASR 捕获情感波动 + TTS 声音复刻实现情绪化回复;FEC + 自适应 Jitter Buffer,50% 丢包无感知恢复

**这条在说什么**:语音体验优化两块。① **情绪化**:ASR 实时捕获用户情感波动,配合 TTS 声音复刻(克隆指定音色)给出带情绪的回复,更像真人;② **弱网对抗**:用 FEC(前向纠错,发冗余包让接收端丢包时直接算出来,不重传)+ 自适应 Jitter Buffer(抖动缓冲,网络差就加大缓冲),在 50% 高丢包环境下也能让语音对话无感知恢复。

**可能的提问与答:**
- **Q:为什么弱网用 FEC 而不用重传(ARQ)?**
  A:重传要一个往返,延迟太高,实时语音受不了。FEC 提前发冗余数据,丢包时接收端直接用冗余恢复,不用重传,实时性最高,代价是占额外带宽。
- **Q:声音复刻是什么?**
  A:用少量音频样本克隆出特定音色,让 TTS 用这个音色说话,可以做成统一的品牌音色甚至"我的音色"。

### 条目 6:Prompt Pilot 管理提示词,评测集自动化评分与版本对比,负样本智能优化重构,高质量样本回流数据中台为 Fine-tuning 储备语料

**这条在说什么**:提示词工程化管理。用 Prompt Pilot 平台:① 建评测集(Prompt + 理想回答)做**自动化评分**,换模型/改 prompt 后批量跑分对比版本,不靠人工逐条看;② 对答错的**负样本**用智能优化算法(强化学习/反思)自动重构 prompt;③ 把高质量样本**回流到数据中台**沉淀,为后续模型微调储备语料(又一次体现和数据中台的闭环)。

**可能的提问与答:**

- **Q:为什么要把 prompt 当代码一样版本管理?**
  A:Prompt 是 LLM 应用的"源代码",改一个字可能影响所有输出。版本化 + 评测集回归,能防止"修好一个 case 却悄悄改坏十个",用数据决定是否上线。
- **Q:负样本优化具体怎么做?**
  A:收集模型答错/越界的真实案例,针对性在 prompt 里加约束、加反例,或纳入评测集做回归,用失败案例驱动迭代。

### 条目 7:接入微调后的模型作为 LLM 层,缩短 system prompt 降 TTFT;Max Token 分级控制,按 95% 覆盖率边界动态配置推理/输出比例

**这条在说什么**:① 把「项目四微调好的豆包模型」接进来当 LLM 层——因为模型已经通过微调学会了话术风格,**system prompt 可以写更短**,prompt 短了 prefill 少、TTFT 更低;② **Max Token 分级控制**:统计业务里回复长度分布,按覆盖 95% 情况的边界来动态配置 max_tokens 和推理/输出比例,既不浪费 token 又不会把话截断。

**可能的提问与答:**
- **Q:微调为什么能缩短 system prompt?**
  A:没微调时,话术风格、人设要靠长长的 system prompt 来约束;微调把风格"刻"进了模型参数,prompt 里就不用再反复强调,自然能写短,降低 TTFT。这就是"微调管风格、RAG 管知识"。
- **Q:语音场景 max_tokens 一般怎么设?**
  A:语音建议 150-300——人每秒说 4-5 字,150 token 约朗读 20-30 秒,连说超 1 分钟用户就想打断。还要配合 prompt 引导简短 + 流式输出防截断。

## 项目二、自媒体运营 AI 智能体(2025.10~11)

> **项目一句话**:电商内容生成 Agent,基于 LangGraph 编排"选题→写作→配图"多阶段工作流,自动产出商品图/介绍/文案,产出图片回流素材中心反哺多模态知识库。技术亮点是 LangGraph 工程化 + 异步并发 + 多模型路由 + 可观测/服务治理。

### 条目 1:LangGraph 多阶段内容生成工作流,SubGraph 拆选题/写作/配图为独立编排单元;PostgreSQL Checkpointer 持久化,支持历史节点回滚与增量重试,故障恢复粒度从整图细化到单节点

**这条在说什么**:用 LangGraph 把内容生成拆成多阶段工作流,**每个阶段(选题、写作、配图)是一个 SubGraph 子图**,独立编排、可单独开发测试复用。用 **PostgreSQL Checkpointer** 把每步状态持久化到数据库,所以某一步失败时**只需从那个节点重试,不用整个流程从头跑**——这就是"故障恢复粒度从整图细化到单节点"。

**可能的提问与答:**
- **Q:为什么用 LangGraph 而不是 LangChain 的 Chain?**
  A:内容生成是多阶段、可能要分支/重试/回滚的有状态流程,Chain 是单向走一遍,拐不了弯。LangGraph 用"节点+边+共享状态"建模,支持循环、分支、断点续跑,适合这种复杂编排。
- **Q:Checkpointer 怎么实现单节点回滚?**
  A:它在每步保存状态快照并和 thread_id 绑定。失败后可以定位到出错前的检查点,从那一步恢复继续,而不是丢弃整个流程重跑,既省时间又省 token。

### 条目 2:MetricsContext 上下文管理器做 LLM 全链路追踪,采集各节点 Token 消耗/推理延迟/错误率,结合火山定价完成成本核算,为模型选型提供数据支撑

**这条在说什么**:自己写了一个 `MetricsContext` 上下文管理器(Python 的 `with` 用法),包住每个 LLM 调用节点,自动采集 token 消耗、延迟、错误率,再结合火山引擎的定价算出每次/每节点的**实际成本**,用这些数据反过来指导"哪个环节该换更便宜的模型"。

**可能的提问与答:**
- **Q:为什么用上下文管理器(with)来做追踪?**
  A:用 `with` 能优雅地在代码块进入时开始计时/计数、退出时(哪怕报错也能)自动收尾上报,不用在每个调用点手写埋点,代码干净且不漏采。
- **Q:成本核算有什么用?**
  A:LLM 调用按 token 计费,不核算就是黑盒花钱。有了每节点的 token 和成本数据,才能做"简单任务换小模型"这类有依据的成本优化(对应条目 5)。

### 条目 3:SSE 流式输出与 LangGraph astream_events 深度集成,监听 on_chat_model_stream 捕获 Token 级实时流,配合前端 ReadableStream 将感知 TTFT 降低 60%,保证流式/非流式下 Checkpointer 数据结构一致

**这条在说什么**:用 LangGraph 的 `astream_events` 事件流,监听 `on_chat_model_stream` 这个事件拿到**逐 token 的实时输出**,通过 SSE 推给前端,前端用 ReadableStream 边收边渲染,用户更早看到内容,**感知首字延迟降 60%**。同时保证不管流式还是非流式,存到 Checkpointer 里的数据结构一致(不会因为开了流式就存歪)。

**可能的提问与答:**
- **Q:"感知 TTFT 降 60%"是真的算得更快了吗?**
  A:不是计算变快,是**体感**变快。流式让用户在第一个 token 出来就看到内容,而不是干等整段生成完,所以感知延迟大幅下降,实际总耗时没变。
- **Q:为什么强调流式/非流式 Checkpointer 数据结构一致?**
  A:因为流式是一段段拼的、非流式是一次性的,如果不统一,存档结构会不一样,导致回滚/恢复时出 bug。统一后两种模式可以共用同一套持久化和恢复逻辑。

### 条目 4:配图批量生成异步并行——asyncio.gather + run_in_executor 把同步 API 封装为协程并发,指数退避重试(1s→2s→4s),整体耗时缩减 70%

**这条在说什么**:一次要生成多张配图,串行太慢。用 `asyncio.gather` 把多个生成任务**并发**跑;但画图 SDK 是同步阻塞的,直接在协程里调会卡住事件循环,所以用 `run_in_executor` 把同步调用丢到线程池("同异步桥接")。再加**指数退避重试**(失败后等 1s、2s、4s 再试)应对服务抖动。综合耗时降 70%。

**可能的提问与答:**

- **Q:为什么同步 API 要用 run_in_executor 包?**
  A:在 async 事件循环里直接调阻塞的同步函数会卡住整个循环,所有并发任务一起变慢。run_in_executor 把它丢到线程池执行,事件循环就能继续调度别的任务。
- **Q:为什么用指数退避而不是固定间隔重试?**
  A:固定间隔在服务过载时会形成"重试风暴"加剧雪崩。指数退避让间隔成倍增长,给下游恢复时间,通常还加随机抖动避免大家同时重试。

### 条目 5:多模型动态路由——简单任务走 Flash 模型、复杂任务走旗舰模型,with_structured_output 做 Pydantic Schema 强约束,JSON Mode + 手动解析解决结构化输出与 Token 统计不兼容,降 40% 推理成本

**这条在说什么**:① **动态路由**:按任务难度选模型,简单的(如分类、起标题)走便宜快的 Flash,复杂的(如长文写作)走旗舰模型,综合降 40% 成本;② **结构化输出**:用 `with_structured_output` + Pydantic Schema 强约束模型返回的 JSON 结构;③ 踩过一个坑——某些结构化输出方式拿不到 token 统计,所以改用 **JSON Mode + 手动解析**,既拿到结构化结果又能统计 token(对应条目 2 的成本核算)。

内容生成 Agent 要批量产出结构化的商品文案（标题、卖点、价格、分类等字段），下游还要入库、关联素材中心。如果靠模型
  ▎ 自由发挥的文本，解析又脏又不稳。所以我用 Pydantic 定义输出 Schema + with_structured_output 强约束，让模型必须按字段
  ▎ 返回，并自动校验解析——保证拿到的就是干净可用的结构化对象，下游能直接消费，可靠性和可维护性都大幅提升

**可能的提问与答:**

- **Q:怎么判断任务该走哪个模型?**
  A:按任务类型/难度预先分级路由,简单确定性任务走小模型,开放复杂任务走大模型;也可做降级——大模型超时/限流时切备用模型。核心是在成本、延迟、质量间权衡。
- **Q:为什么不用 with_structured_output 而改 JSON Mode + 手动解析?**
  A:因为发现用某些结构化输出封装时拿不到 token 用量,而我要做成本核算必须有 token 数。所以改成 JSON Mode 保证输出是合法 JSON、自己手动解析,同时保留 token 统计,两者兼得。

### 条目 6:环境变量驱动的 Mock 开关(llm/image/checkpointer)解耦外部服务,Swagger UI + Ngrok,本地调试效率提升 3 倍;LangSmith + structlog + request_id 全链路可观测,问题定位小时级→分钟级

**这条在说什么**:工程效率两块。① **Mock 开关**:用环境变量控制 LLM、画图、checkpointer 是走真实服务还是 Mock(假数据),调试时不用每次真调外部服务(慢且花钱),效率提升 3 倍;配合 Swagger UI 发测试请求、Ngrok 把本地映射到公网。② **可观测**:LangSmith 追踪每步链路 + structlog 结构化日志 + request_id 串联整条请求,出问题按 request_id 一查到底,定位从小时级压到分钟级。

**可能的提问与答:**
- **Q:Mock 开关为什么能提效 3 倍?**
  A:外部服务慢、贵、还可能不稳定。开发某一环时把其他环 Mock 成固定假数据,就能快速反复调本环逻辑,不被外部拖累,也不烧钱。
- **Q:request_id 是干什么的?**
  A:每个请求入口生成唯一 id,贯穿它经过的所有函数和日志。高并发异步下日志是交错的,有了 request_id 就能把一次请求的完整链路串起来,快速定位哪一步出错。

### 条目 7:SlowAPI 限流 + 熔断降级 + /health 健康检查 + 优雅关闭保障容器化高可用;生成的商品图写入素材中心,经离线增强后成为客服多模态素材,形成"生成即沉淀"闭环

**这条在说什么**:① **服务治理**:SlowAPI 限流(防突发流量/控成本)+ 熔断降级(下游挂了快速失败/切备用)+ /health 健康检查(供容器探活)+ 优雅关闭(收到停止信号先处理完在途请求再退),保障容器化部署的高可用;② **闭环**:这个 Agent 生成的商品图直接写进素材中心,经过离线增强(OCR/VLM 打标)后,变成 AI 客服能检索命中的多模态素材——"生成即沉淀"。

**可能的提问与答:**
- **Q:熔断和降级区别?**
  A:熔断是下游连续失败就"跳闸",一段时间内直接快速失败不再请求,保护自己不被拖垮;降级是主功能不可用时给个次优可用结果(切备用模型、返回缓存/友好提示)。一个保自己、一个给用户交代。
- **Q:为什么 LLM 服务特别需要优雅关闭?**
  A:LLM 请求可能要几十秒,如果发版时直接杀进程,在途请求会被硬切断、用户报错、数据可能不一致。优雅关闭先停接新请求、等在途的处理完再退出,做到发版用户无感。

## 项目三、AI 数据中台(2024.12~至今)

> **项目一句话**:整个 AI 体系的数据底座,把微信原始聊天记录加工成可训练、可检索、可考核的高质量数据资产,所有 AI 业务(客服/微调/考核/素材)都从这里取数。核心是 ETL 数据流水线 + 脱敏合规 + 多格式导出。

### 条目 1:设计 ETL→原始库→暂存区→人工审核→多下游全链路流水线;从微信 SQLite(MSG0-5.db + MicroMsg.db)抽取,做 wxid→显示名映射、群消息发送者解析、时间过滤、(session,timestamp,content)去重后落 raw_chats 原始库(不可变)

**这条在说什么**:设计了一条完整的数据流水线。数据源是微信本地的 SQLite 数据库(消息在 MSG0-5.db、通讯录在 MicroMsg.db)。抽取时做几件清洗:把 wxid(微信内部 id)映射成可读的显示名、解析群消息里到底是谁发的、按时间过滤、用(会话+时间戳+内容)三元组去重,最后落到 **raw_chats 原始库,且不可变**(原始数据只追加不修改,保证可追溯)。

**可能的提问与答:**
- **Q:为什么原始库要设计成"不可变"?**
  A:原始数据是一切下游的源头,如果能改就可能被污染且无法追溯。不可变(只追加)保证任何时候都能回到最原始的真实数据,后续清洗/加工都在下游做,出问题能对照源头排查。
- **Q:为什么要 wxid→显示名映射?**
  A:微信消息里存的是 wxid 这种内部标识,不可读;映射成显示名后数据才有意义,尤其群聊要能区分每条消息的真实发送者。

### 条目 2:DataFilter 内容过滤 + 脱敏引擎——9 级优先级分类丢噪声,手机号/身份证/银行卡/邮箱/API Key 正则脱敏;多层冗余脱敏(清洗时+写入时+导出时三次),价格脱敏设为硬红线

**这条在说什么**:做了一个过滤+脱敏引擎。① 按 9 个优先级把消息分类(系统消息/媒体/过短/敏感/闲聊/垃圾/高价值…),丢掉噪声只留有价值的;② 用正则识别并脱敏敏感信息(手机号、身份证、银行卡、邮箱、API Key);③ **多层冗余脱敏**——清洗时、写入时、导出时各脱一次,三道防线防漏;④ 尤其**价格脱敏是硬红线**,绝不能让微调出来的模型泄露真实定价。

**可能的提问与答:**
- **Q:为什么要做三次冗余脱敏,一次不够吗?**
  A:脱敏是合规红线,漏一条都可能出事故。单点容易因为规则盲区或代码 bug 漏掉,三道独立防线(清洗/写入/导出)互为兜底,大幅降低泄露概率。属于"宁可重复也不能漏"。
- **Q:价格为什么是硬红线?**
  A:聊天记录里有真实成交价、底价,如果微调进模型,模型可能在对话里说出来,造成定价泄露、商业风险。所以价格必须强制脱敏,优先级最高。

### 条目 3:「一键清洗」对话块构建——5 分钟时间窗切分会话、合并同角色连续消息、去重、自动预标注(分类/质量分/Q&A 提取/敏感标记);高质量进 staging_conversations 暂存区,人工 approve/reject/编辑后作为唯一可信数据源

**这条在说什么**:把零散消息组装成"对话块"。① 用 **5 分钟时间窗**判断哪些消息属于同一轮对话;② 合并同一个人连续发的多条消息;③ 去重;④ 自动**预标注**(打分类、质量分、提取问答对、标敏感)。处理好的高质量数据进 **staging_conversations 暂存区**,再经**人工审核**(通过/拒绝/编辑),通过的才成为唯一可信的数据源(给下游微调/知识库用)。

**可能的提问与答:**
- **Q:为什么要 5 分钟时间窗切分会话?**
  A:聊天是连续消息流,没有天然的"一轮对话"边界。用时间间隔(5 分钟内算同一会话上下文)是一种简单有效的启发式切分,把流式消息组织成有上下文的对话块,才能用于训练。
- **Q:为什么要"人工审核后才作为可信源"?**
  A:自动清洗+预标注能处理大部分,但仍有误判。微调/知识库的数据质量直接决定模型效果(垃圾进垃圾出),所以加一道人工 approve/reject 把关,保证进入下游的是真正可信的高质量数据。

### 条目 4:训练数据导出清洗(价格脱敏/风格修复/过短过滤/过长截断/占位符清理),一份数据多格式导出——ShareGPT/Alpaca/OpenAI JSONL(供微调)+ RAG CSV(供知识库)

**这条在说什么**:导出环节再做一轮清洗(再次价格脱敏、修复语言风格、过滤太短的、截断太长的、清理占位符),然后**同一份数据导出成多种格式**:ShareGPT/Alpaca/OpenAI JSONL 这些是微调训练用的标准格式,RAG CSV 是知识库用的。一次加工、多处复用,打通微调和检索两条链路。

**可能的提问与答:**
- **Q:ShareGPT、Alpaca、OpenAI JSONL 这些格式有什么区别?**
  A:都是微调指令数据的格式。Alpaca 是 instruction/input/output 三段式,适合单轮指令;ShareGPT 是多轮对话格式(human/gpt 交替),适合学多轮对话风格;OpenAI JSONL 是 messages 数组格式,适合走 OpenAI 兼容的微调接口。按训练框架/平台要求选。
- **Q:为什么微调和知识库要导出不同格式?**
  A:用途不同。微调要的是"对话/指令样本"去改模型参数(学风格),知识库要的是"可检索的知识条目"(CSV 入库做 RAG)。这对应"微调管风格、RAG 管知识"的分工。

### 条目 5:向量知识库下游——对 approved 数据做 embedding(DashScope text-embedding-v3)写入 knowledge_chunks,经 LLM 提炼为结构化 knowledge_articles(场景/客户话术/推荐回复/置信度),供项目一 RAG 和项目六 AI 考核出题

**这条在说什么**:审核通过的数据进一步加工成知识库。① 用 **DashScope(阿里) text-embedding-v3** 把文本向量化,写入 knowledge_chunks 表;② 再用 LLM 把原始对话**提炼成结构化的 knowledge_articles**,每条带场景、客户话术、推荐回复、置信度字段。这些知识同时喂给项目一的客服 RAG 和项目五的 AI 销售考核出题。

**可能的提问与答:**
- **Q:为什么除了 chunks 还要提炼 articles?**
  A:chunks 是原始切片,适合做向量检索;articles 是 LLM 提炼后的结构化知识(场景/话术/推荐回复),更干净、更适合直接给客服参考或给考核系统出题。两层满足不同下游需求。
- **Q:置信度字段有什么用?**
  A:标记这条提炼知识的可靠程度,下游用的时候可以按置信度过滤——比如出题或回复时优先用高置信度的知识,低置信度的谨慎使用或人工复核。

### 条目 6:中台作为数据总线对接全业务——ASR 转写结果回流入库,素材中心图片/视频反向补充多模态语料,实现"AI 的一切都是数据"的闭环

**这条在说什么**:总结中台的"数据总线"定位。它不只是单向供数,还**接收回流**:客服产生的 ASR 转写结果回流入库留存、素材中心的图片/视频反向补充多模态语料。所有 AI 业务的数据进进出出都过中台,形成闭环。

**可能的提问与答:**
- **Q:"数据总线"这个定位你怎么理解?**
  A:中台是所有 AI 业务的数据中枢——对上游(微信记录、ASR、素材)做汇聚加工,对下游(客服/微调/考核)做供给,且下游产生的新数据又回流沉淀。数据在这里统一管理、循环增值,而不是各业务各搞一套。
- **Q:这个项目和其他项目是什么关系?**
  A:它是底座。项目一客服的知识库、项目四微调的语料、项目五考核的题目素材,都来自这里;反过来它们产生的数据(ASR 转写、生成素材)又回流到中台。是整个 AI 体系的数据根基。

## 项目四、基于聊天数据的大模型微调(2025.07~08)

> **项目一句话**:为了让 AI 客服回复更像真人,用数据中台导出的 SFT/DPO 数据微调豆包模型,再回灌到客服系统。核心是"本地小模型验证 + 云端大模型生产训练"两阶段,以及 SFT→DPO 的训练路径。

### 条目 1:微调语料来自数据中台,对话分流为两套——SFT 数据(保留多轮上下文学销售话术风格)与 RAG 知识库数据(提炼结构化知识),践行"微调管风格、RAG 管知识"

**这条在说什么**:微调用的数据直接消费数据中台 approved 的 ShareGPT/OpenAI 格式数据。关键是**把数据分两条路**:① 保留多轮上下文的对话 → 做 **SFT**,让模型学销售的**话术风格**;② 课程、价格、优惠这类事实 → 提炼成**结构化知识进 RAG 知识库**。这就是核心理念:**微调负责"怎么说"(风格),RAG 负责"说什么"(知识)**,知识更新只改知识库不用重训模型。

**可能的提问与答:**

- **Q:为什么不把知识也微调进模型?**
  A:知识会频繁变(价格/优惠天天改),微调进模型的话一改就要重训,成本高且滞后,还容易产生幻觉。把知识放 RAG,更新只改知识库即可实时生效;模型只学不变的"风格"。这就是职责分离。
- **Q:怎么判断哪些数据走 SFT、哪些走 RAG?**
  A:体现"风格/语气/话术套路"的多轮对话走 SFT;承载"事实信息"(课程内容、价格、优惠规则)的走 RAG 知识库。前者教模型怎么说话,后者给模型查事实。

### 条目 2:「本地小模型验证 + 云端大模型训练」两阶段——本地 LLaMA-Factory + Qwen2.5-3B-Instruct 做 QLoRA(rank=64,alpha=128,lr=2e-4),单卡 RTX 4090 约 40 分钟一轮快速验证数据质量;通过后上传火山方舟对豆包 1.5 做生产级 LoRA(rank=32,lr=1e-5),按模型规模等比缩放超参,风格一致性盲评从 2.1 提升到 4.3(5 分制)

**这条在说什么**:训练分两阶段,省钱又稳。① **本地快速验证**:先用开源小模型 Qwen2.5-3B,在自己的 RTX 4090 上用 QLoRA(4-bit 量化 + LoRA,省显存)跑,40 分钟一轮,主要目的是**验证数据质量**(数据有没有问题,先小成本试);② **云端生产训练**:数据验证 OK 后,上传到火山方舟,对生产用的豆包 1.5 做正式 LoRA 微调。超参按模型规模等比缩放(小模型 rank=64/lr=2e-4,大模型 rank=32/lr=1e-5)。结果:风格一致性人工盲评从 2.1 升到 4.3(5 分制)。

**可能的提问与答:**

- **Q:为什么先本地小模型验证,不直接上云训练?**
  A:云端训生产大模型贵且慢。先用本地小模型低成本快速验证"数据本身行不行"——如果数据有问题,小模型上也学不好,就不用浪费云端算力。数据过关了再上云正式训,降低试错成本。
- **Q:rank=64、alpha=128、lr=2e-4 这几个参数分别是什么意思?**
  A:这是 LoRA 微调的三个核心超参——
  - **rank(秩,r=64)**:LoRA 不直接改原模型权重,而是在旁边加两个小矩阵 A、B(ΔW = B·A)来学习变化量。rank 就是这两个小矩阵的"秩/宽度",**决定了可训练参数量和模型的学习容量**。rank 越大能学的东西越多、表达越强,但参数多、易过拟合、占显存;rank 越小越省、越稳但学习能力有限。常用 8/16/32/64。
  - **alpha(缩放系数,α=128)**:控制 LoRA 学到的增量**叠加回原模型时放大多少**,实际缩放比例是 **α/r**(这里 128/64 = 2)。**经验法则 alpha 常设为 rank 的 2 倍**,相当于把 LoRA 的影响力调到一个合适强度——太小学不动,太大盖过原模型。
  - **lr(学习率,2e-4 即 0.0002)**:每一步参数更新的**步长**。大了收敛快但可能震荡/学崩,小了稳但慢。LoRA 因为只训练小矩阵,通常能用比全量微调**大一些**的学习率(常见 1e-4~3e-4)。
- **Q:为什么大模型 lr 用更小(1e-5 vs 2e-4)?**
  A:模型越大、越接近生产,越要小步微调防止破坏已有能力(灾难性遗忘),所以学习率更小、更保守;小模型快速验证阶段可以用大一点的 lr 跑得快。简历里"按模型规模等比缩放超参"就是指:换更大的生产模型时,把 rank、lr 这些相应调小调稳。
- **Q:那微调到底"微调"的是什么参数?是改模型本身的权重吗?(高频追问)**
  A:要分两种情况说清楚——
  - **全量微调(Full Fine-tuning)**:直接更新**模型本身的全部权重**,效果上限高,但显存和算力成本极大,还容易灾难性遗忘,几十亿参数的模型个人根本训不动。
  - **LoRA/QLoRA(参数高效微调,我用的)**:**冻结原模型的全部权重不动**,只在旁边**新增并训练一小部分低秩矩阵(A、B)**,可训练参数往往不到原模型的 1%。所以严格说,**我微调时调的不是原模型权重,而是这部分新增的 LoRA 适配器参数**;推理时把它和原模型叠加即可,产出的 adapter 很小、可热插拔。
  - **而 rank/alpha/lr 这些叫"超参数(hyperparameter)",不是被训练出来的**,是我**训练前手动设定**的、控制"怎么训"的旋钮(学多少、学多强、步子多大)。
  - **一句话总结**:微调真正被"训练更新"的是 **LoRA 新增的低秩矩阵参数**(原模型权重冻结);而 rank、alpha、lr 是我训练前设好的**超参数**,负责调控训练过程。**"数据决定上限,超参决定能不能稳稳逼近这个上限"**。

### 条目 3:SFT→DPO 两阶段训练路径——SFT 解决"从不会到会说"的风格模仿;DPO 对同一问题采样多条回复,由销售主管标注 chosen/rejected 偏好对训练,解决"会说但说不好";选 DPO 而非 RLHF 因其跳过奖励模型,在偏好数据有限(几百条)时更稳定、成本更低

**这条在说什么**:两阶段训练。① **SFT(监督微调)**:用"问题→标准回答"对训练,让模型先**学会**按销售风格说话(从不会到会);② **DPO(直接偏好优化)**:对同一个问题让模型生成多条回复,**销售主管选出哪条好(chosen)哪条差(rejected)**组成偏好对,训练模型偏向"好"的那种,解决"会说但说得不够好"。选 DPO 不选 RLHF,是因为 DPO 跳过了奖励模型和强化学习,在偏好数据只有几百条时更稳定、更省成本。

**可能的提问与答:**
- **Q:SFT 和 DPO 各解决什么?为什么要两步?**
  A:SFT 教模型"会说"——模仿标准答案的格式和风格;但 SFT 只学"正确答案",学不到"什么更好"。DPO 用"好/差"偏好对,教模型在会说的基础上"说得更好"。先 SFT 打基础,再 DPO 对齐偏好。
- **Q:DPO 和 RLHF 的区别?为什么选 DPO?**
  A:RLHF 要先训一个奖励模型,再用 PPO 强化学习优化,流程复杂、不稳定、算力贵。DPO 用一个推导出的损失函数直接在偏好对上优化,不需要奖励模型和在线采样。我们偏好数据只有几百条,DPO 更稳、更省,所以选它。

### 条目 4:Embedding→向量检索→Rerank 三阶段 RAG Pipeline——火山 Embedding 向量化 + 向量检索 top-20 粗召回,引入 Cross-Encoder 精排取 top-5,解决双编码器在语义相近文档上区分度不足,知识命中率从 72% 提升到 89%

**这条在说什么**:检索做成"粗召回 + 精排"两阶段。① 用火山 Embedding 把文本向量化,向量检索先粗召回 **top-20**(快但不够准);② 再用 **Cross-Encoder 重排模型**对这 20 条精排,取最相关的 **top-5** 给 LLM。这样解决了纯向量(双编码器/Bi-Encoder)在"长得很像但其实不同"的文档上分不清的问题,知识命中率从 72% 提到 89%。

**可能的提问与答:**
- **Q:Bi-Encoder 和 Cross-Encoder 区别?为什么两阶段?**
  A:Bi-Encoder(embedding)把 query 和 doc 分别独立编码成向量再算相似度,可以预先建索引、检索快但精度有限;Cross-Encoder 把 query 和 doc 拼一起送进模型做交叉注意力,直接打相关性分,精度高但慢、不能预计算。所以用 Bi-Encoder 粗召回大量、Cross-Encoder 精排少量,兼顾速度和精度。
- **Q:命中率 72%→89% 是怎么测的?**
  A:用一批标注好"正确该命中哪条知识"的测试问题,跑检索看 top-K 里有没有命中正确知识,算命中率。加了 Cross-Encoder 精排后,真正相关的被排进 top-5 的比例显著提高。
- **Q:如果召回失败(没检索到相关知识,或检索到的都不对)怎么办?(高频追问)**
  A:这是 RAG 必问题,我分"先判断、再兜底、后优化"三层来答——

  **第一步:先判断是不是真的召回失败。** 用检索结果的**相似度分数 + 设阈值**判断:如果 top 结果分数都低于阈值,就认定"没召回到可信知识",不能硬把不相关的内容塞给 LLM(否则必然幻觉)。

  **第二步:召回失败时的兜底策略(保证不胡说)。**
  
  1. **诚实兜底,不编造**:让 LLM 走"未命中"分支,回答"这个问题我暂时没有相关资料",而不是拿低质内容硬答——**宁可不答,不可乱答**,客服场景尤其如此。
  2. **转人工/追问**:触发转人工,或反问用户澄清(把模糊问题问清,往往就能重新召回)。
  3. **降级检索**:适当放宽——调高 Limit 多取几条、降低 dense_weight 偏关键词、或退回纯关键词检索,再试一次。
  4. **记录 badcase**:把这条未命中的提问记下来,进入优化闭环。
  
  **第三步:从根因上减少召回失败(优化方向)。** 召回失败通常是三类原因,对应三类优化:
  1. **知识库压根没有这条知识** → 补充知识库内容(最常见,也最该先查)。
  2. **知识有,但没检索出来(检索召回问题)** → 优化切片(自定义分隔符防语义截断)、用混合检索(hnsw_hybrid 补关键词命中)、加 Rerank 精排、调 dense_weight、对用户问题做**查询改写/扩展**(把口语化提问改写成更易命中的表述)。
  3. **检索出来了但 LLM 没用好** → 优化 prompt,要求模型严格基于检索内容作答。
  
  **一句话总结**:先用相似度阈值识别召回失败,失败就诚实兜底/转人工/降级重试、绝不拿低质内容硬答;再通过日志定位是"知识缺失、检索没召回、还是模型没用好",分别用补知识库、优化检索(切片/混合检索/Rerank/查询改写)、优化 prompt 来根治。

### 条目 5:微调模型作为 LLM 层接入项目一 RTC 语音链路(客户语音→ASR→RAG→微调 LLM→TTS→语音回复),通过微调缩短 system prompt 降 TTFT,闭合"数据中台→微调→客服"回路

**这条在说什么**:把微调好的模型部署上线,接进项目一的实时语音链路当 LLM 层。因为风格已经微调进模型,system prompt 能写更短 → TTFT 更低。至此形成完整闭环:数据中台出语料 → 微调出模型 → 接入客服 → 客服又产生新数据回流中台。

**可能的提问与答:**

- **Q:微调模型上线后怎么验证效果?**
  A:线上看回复的风格一致性(人工盲评)、用户体验,线下用评测集对比微调前后的回答质量,以及 TTFT 等性能指标。如果不达预期就回到数据/训练阶段迭代。
- **Q:这条体现了什么?**
  A:体现端到端打通能力——不是孤立训个模型,而是把"数据→微调→部署→反馈"串成可持续迭代的闭环,这是生产级 AI 应用的关键。

## 项目五、多模态素材中心 & AI 销售考核培训系统(2025.10~至今)

> **项目一句话**:一条数据线两件事——素材中心把图片/视频做成"可被语义命中的多模态知识库"给客服用;AI 销售考核系统用中台知识自动出题+评判,替代人工培训。核心亮点是"向量化检索文本而非媒体本身"的多模态设计 + LLM 自动出题评判。

### A. 多模态素材中心

#### 条目 A1:统一向量底座 rag_assets——text/image/video 三模态共用一张表(pgvector 1024 维 + HNSW vector_cosine_ops),一条 SQL 跨模态召回;核心理念是不向量化媒体本身,而向量化「检索文本」retrieval_text("这张图/视频能回答什么问题"),与用户文本提问天然对齐

**这条在说什么**:这是整个项目最巧妙的设计。文字/图片/视频三种模态**共用一张表 rag_assets**(用 pgvector 存 1024 维向量、HNSW 余弦索引),一条 SQL 就能跨模态检索。关键思路:**不直接向量化图片/视频本身**(那样要多模态 embedding 且和文本提问对不齐),而是给每个素材写一段**「检索文本」retrieval_text**——描述"这张图/这个视频能回答什么问题",再把这段文本向量化。这样用户的文本提问和素材的检索文本在**同一个文本语义空间**里,天然对齐、好命中。

**可能的提问与答:**
- **Q:为什么不直接向量化图片本身,而是向量化"检索文本"?**
  A:① 用户提问是文本,如果图片走图像 embedding,跨模态对齐难、效果不稳;② 把"这图能回答什么问题"写成文本再 embedding,就和用户文本提问在同一语义空间,匹配更准;③ 还能复用现成的文本 embedding 和一张表统一检索,工程简单。
- **Q:三模态共用一张表有什么好处?**
  A:一条 SQL 跨模态召回,不用查三个库再合并;统一的向量底座(pgvector+HNSW)便于维护和做统一的相似度阈值控制。

#### 条目 A2:人工即时命中(manual)路线——上传时人工维护标题/描述/标签/分类拼接为 retrieval_text 经 text-embedding-v3 入库即可命中;隐私红线为命中返回打码版 TOS URL

**这条在说什么**:素材入库有两条路线,这是第一条"人工即时"。图片/视频上传时,**人工填标题/描述/标签/分类**,拼成 retrieval_text,用 text-embedding-v3 向量化入库,马上就能被检索命中。隐私上有红线:命中后返回的是**打码版的 TOS(对象存储) URL**,不直接暴露原图。

**可能的提问与答:**
- **Q:为什么要人工维护描述,不全自动?**
  A:人工描述准、即时可用,适合刚上传就要能命中的场景。全自动(机器增强)是第二条路线异步补充。两条结合:人工保底+机器增强。
- **Q:打码版 URL 是为什么?**
  A:隐私合规红线。素材可能含敏感信息,命中返回打码版,避免原始隐私内容直接外泄。

#### 条目 A3:离线机器增强(enriched)路线——OCR(RapidOCR)+VLM(qwen-vl)补图片文字与看图描述、ASR+LLM 摘要补视频口播,人工内容在前机器内容追加,重写 retrieval_text 重新 embedding,召回相似度从 ~0.38 提升到 ~0.71;按(source_table,source_id)UPSERT 幂等可反复回扫

**这条在说什么**:第二条路线"机器增强",异步离线跑。对图片用 **OCR(RapidOCR)提取图里文字 + VLM(qwen-vl)生成看图描述**;对视频用 **ASR 转写口播 + LLM 摘要**。把这些机器生成的内容**追加在人工内容后面**(人工优先),重写 retrieval_text 再重新 embedding,检索相似度从约 0.38 大幅提到约 0.71。整个过程按(来源表,来源 id)做 **UPSERT 幂等**,所以可以反复回扫不会重复写脏。

**可能的提问与答:**
- **Q:机器增强为什么能把相似度从 0.38 提到 0.71?**
  A:人工描述往往简短,覆盖不全。OCR/VLM/ASR 把图里的字、画面内容、视频口播都补进检索文本,信息更丰富完整,和用户各种提法的语义重合度更高,所以召回相似度大幅提升。
- **Q:UPSERT 幂等是什么?为什么需要?**
  A:UPSERT = 有则更新、无则插入。按(source_table,source_id)做唯一键,保证同一素材反复跑增强时是覆盖更新而不是新增重复行。这样离线任务可以安全地反复回扫、重跑,不怕脏数据。

#### 条目 A4:长视频按 ASR 时间戳分段为多条 rag_assets,media_url 携带 #t=秒数,命中后前端播放器跳转精彩片段;统一应答按「文字必发、图≤3、视频≤1」混排,阈值递增(text 0.35 < image 0.40 < video 0.42)

**这条在说什么**:① **长视频分段**:一个长视频按 ASR 的时间戳切成多条 rag_assets,每条的 media_url 带 `#t=秒数`,命中后前端播放器**直接跳到那个精彩片段**,不用从头看;② **混排策略**:回复时"文字必发、图最多 3 张、视频最多 1 个",且命中阈值**递增**(文字 0.35、图片 0.40、视频 0.42)——越重的媒体要求越高的相似度才放出来。

**可能的提问与答:**
- **Q:为什么命中阈值要按 文字<图片<视频 递增?**
  A:媒体越重,误推的代价越大、越打扰用户。文字便宜,阈值低多发点没关系;视频最重,必须足够相关(高阈值)才推,避免推一堆不相关的图/视频刷屏。
- **Q:长视频分段+时间戳跳转解决什么?**
  A:用户问的往往只对应视频里某一小段,整段返回让用户自己找太差。按时间戳分段、命中直接跳到精彩片段,体验好很多。

### B. AI 销售考核系统

#### 条目 B1:LLM 自动出题——消费数据中台的 knowledge_articles/knowledge_chunks,按场景分类+置信度取素材,以「资深考官」角色(temperature 0.7)生成简单/中等/困难分级的开放式场景题与参考答案

**这条在说什么**:用 LLM 自动出考题替代人工。从数据中台拿结构化知识(按场景+置信度筛),让 LLM **扮演"资深考官"角色**,用较高的 temperature(0.7,要多样性)生成分难度(简单/中等/困难)的开放式场景题,同时给出参考答案。

**可能的提问与答:**
- **Q:出题为什么用较高 temperature 0.7?**
  A:出题要的是**多样性**,题目不能千篇一律。高 temperature 让生成更发散、题目更丰富。而评判(B2)要稳定,所以用低 temperature。
- **Q:题目素材怎么保证质量?**
  A:从数据中台 approved 的结构化知识里取,按场景分类 + 按置信度过滤,优先用高置信度知识出题,保证题目有真实业务依据。

#### 条目 B2:LLM 自动评判——以「资深评审专家」角色(temperature 0.3 保稳定)从实战性/需求把握/销售策略/话术自然度/红线检查五维度逐题打分,按「平均分×10」映射百分制,支持人工二次复核

**这条在说什么**:学员答完,LLM **扮演"资深评审专家"**,用低 temperature(0.3,要稳定一致)从五个维度(实战性、需求把握、销售策略、话术自然度、红线检查)逐题打分,五维平均分×10 换成百分制,还支持人工二次复核校准。这是典型的 **LLM-as-judge** 应用。

**可能的提问与答:**
- **Q:评判为什么 temperature 用 0.3?**
  A:评分要的是**稳定可复现**——同一份答案多次评分应尽量一致,不能忽高忽低。低 temperature 让输出更确定。出题要多样(0.7)、评判要稳定(0.3),这是按任务目标分别设的。
- **Q:LLM 评分可靠吗?怎么保证?**
  A:用明确的五维度评分标准约束、给参考答案对照、低 temperature 保稳定,再加人工二次复核校准。也可以用多次评分一致性(类似 Cohen's Kappa)来验证评判本身可不可信。

#### 条目 B3:针对知识库膨胀到十万级的演进,设计 RAG 出题方案——模糊请求先由 LLM 规划主题→多轮向量检索(pgvector/Milvus)→MMR 去重保多样性,解决固定 top-30 导致题目重复、长尾素材浪费的问题

**这条在说什么**:这是面向未来规模的演进设计(体现你有前瞻性)。当知识库涨到十万级,原来"固定取 top-30 出题"会导致题目老重复、长尾(冷门)知识用不到。改进方案:① 模糊请求先让 **LLM 规划要考哪些主题**;② 按主题做**多轮向量检索**(pgvector 或升级到 Milvus);③ 用 **MMR(最大边际相关)去重**,既相关又多样。这样题目不重复、长尾素材也能被覆盖。

**可能的提问与答:**

- **Q:MMR 是什么?为什么用它?**
  A:MMR(Maximal Marginal Relevance)在选结果时同时考虑"和查询相关"和"和已选结果不重复",平衡相关性与多样性。出题时用 MMR 避免选出一堆很像的素材,保证题目覆盖面广。
- **Q:为什么固定 top-30 会浪费长尾素材?**
  A:固定取相似度最高的 30 条,永远是那批热门知识被选中,冷门(长尾)知识永远排不进 top-30,出的题就老重复且覆盖窄。改成 LLM 规划主题+多轮检索+MMR,能主动覆盖不同主题和长尾。

## 项目六、基于 MCP 协议的多 Agent 共享服务集群(2025.10~至今)

> **项目一句话**:随着 AI 项目越来越多,做架构层面的统一收口——把 LLM 网关、RAG、记忆、Prompt 这些**每个项目都要重复造的能力,抽成共享基础设施**,用 MCP 协议标准化暴露,让所有业务(含数据中台)复用同一套底座。这是从"做单个 AI 应用"上升到"搭 AI 中间件平台"的项目,是简历里架构含金量最高的一个。
>
> **背景知识(先讲清 MCP 是什么)**:MCP(Model Context Protocol,模型上下文协议)是 Anthropic 提出的开放标准,用来**标准化 LLM 应用和外部工具/数据/能力的连接方式**。它定义了 Server 暴露 **Tool(工具)、Resource(资源)、Prompt(提示词模板)** 等标准原语,客户端(Agent)按统一协议调用,传输层支持 stdio、Streamable HTTP 等。一句话:**MCP 就是 AI 世界的"USB 接口标准",让工具和 Agent 即插即用、解耦。**
>
> 获取写作对话历史，获取用户画像，检索写作素材库，组装写作prompt，llm生成文章，保存用户指令+生成结果，提取用户画像
>
> 

### 条目 1:将 LLM 网关、RAG 检索、会话记忆、Prompt 管理抽离为 4 个独立 MCP Server,经 Streamable HTTP 暴露标准化 Tool/Prompt 原语,新业务接入仅需写 Agent 编排层

**这条在说什么**:把之前每个项目都各写一遍的四类公共能力(调模型的网关、RAG 检索、对话记忆、提示词管理)**抽出来,各做成一个独立的 MCP Server**,通过 Streamable HTTP 对外暴露成标准化的 Tool/Prompt。这样**新业务接入时,不用再重写这些底层能力,只需要写自己的 Agent 编排逻辑**(怎么把这些工具串起来),开发成本大幅下降。

**可能的提问与答:**
- **Q:为什么要拆成 4 个独立的 MCP Server,而不是一个大服务?**
  A:① 职责单一、各自独立部署和扩缩容(RAG 压力大就单独扩 RAG);② 标准化解耦,任何业务都能按 MCP 协议复用,不绑定具体实现;③ 一处升级、处处生效,避免每个项目各维护一套重复代码。
- **Q:抽成共享服务最大的价值是什么?**
  A:从"每个项目重复造轮子"变成"复用同一套底座"。新业务接入只写 Agent 编排层,开发提效、能力一致性也有保障,这是典型的中台化/平台化思路。

### 条目 2:设计 REST→MCP 协议桥接网关,基于 FastAPI 构建四层中间件链(认证→日志→配额→路由),用 Trace ID 全链路追踪;内置 MCP Client Manager 管理短连接,按项目维度做 API Key 认证与 Token 配额计量,保障多租户安全隔离与成本管控

**这条在说什么**:① **协议桥接**:外部业务习惯用 REST,内部是 MCP,所以做一个网关把 REST 请求翻译成 MCP 调用;② **四层中间件链**:每个请求依次过认证→日志→配额→路由四道处理(FastAPI 中间件),并用 Trace ID 串起全链路追踪;③ **多租户**:按项目(租户)发 API Key 做认证、按项目计量 Token 配额,实现**安全隔离 + 成本管控**——哪个项目用了多少、超没超额,都能算清。

**可能的提问与答:**
- **Q:为什么要做 REST→MCP 桥接网关?直接用 MCP 不行吗?**
  A:MCP 是给 Agent/LLM 客户端用的协议,但很多现有业务和前端只会发 REST 请求。加一层桥接网关,让外部用熟悉的 REST 接入,内部统一走 MCP,兼顾兼容性和标准化。
- **Q:中间件为什么是"认证→日志→配额→路由"这个顺序?**
  A:有先后依赖——先认证(确认你是谁、是哪个租户),才能记带身份的日志、查这个租户的配额够不够,最后才路由到具体服务。认证不过直接拦掉,后面都不用走,省资源也更安全。
- **Q:多租户隔离怎么做的?**
  A:按 project_id/API Key 区分租户,认证时识别身份,配额按租户计量 Token,数据层(RAG/记忆)也按 project_id 物理隔离。这样多个项目共用一套底座但互不干扰、成本可单独核算。

### 条目 3:构建统一 LLM 网关,实现逻辑模型名→Endpoint 路由(auto/pro/lite),YAML 配置热切换业务无感;Embedding 兼容火山引擎 text 与 vision 两种向量化路径,Token 消耗统一回传配额模块扣减

**这条在说什么**:① **逻辑模型名路由**:业务方不直接写死某个具体模型,而是写 `auto/pro/lite` 这种逻辑名,网关再映射到真实的模型 Endpoint——换模型只改 YAML 配置、**热切换、业务无感**;② **Embedding 兼容**:统一支持纯文本和图文(vision)两种向量化;③ **Token 统一计量**:所有调用的 Token 消耗都回传给配额模块扣减,集中管控成本。

**可能的提问与答:**
- **Q:逻辑模型名(auto/pro/lite)路由有什么好处?**
  A:解耦业务和具体模型。业务只关心"要个强模型还是快模型(pro/lite)",换底层模型、调价、做灰度时只改网关 YAML,所有业务自动生效、无需改代码重发版。这其实就是把"多模型动态路由"做成了平台能力。
- **Q:为什么 Token 要统一回传扣减?**
  A:LLM 调用按 Token 计费,多租户场景必须算清每个项目花了多少。统一在网关层计量扣减,既能成本核算,又能做配额限制(超额拦截),避免某个业务失控烧钱。

### 条目 4:实现多租户 RAG 服务——基于火山引擎 Viking DB 按 project_id 划分独立 Collection 做物理隔离,按 Markdown 标题层级 + 段落边界语义切分,新项目调 ingest_document Tool 即可自助接入

**这条在说什么**:把 RAG 做成多租户的共享服务。① **物理隔离**:用 VikingDB,每个项目(project_id)一个独立的 Collection,数据互不串;② **切分策略**:按 Markdown 标题层级 + 段落边界做语义切分(防语义截断);③ **自助接入**:新项目要建知识库,只需调用 `ingest_document` 这个 MCP Tool 把文档喂进去即可,不用开发介入。

**可能的提问与答:**
- **Q:多租户 RAG 为什么按 project_id 分 Collection 物理隔离?**
  A:不同项目的知识库内容不能互相检索到(数据安全 + 检索精度)。按 project_id 分独立 Collection 做物理隔离,A 项目永远查不到 B 项目的知识,既合规又避免跨项目污染检索结果。
- **Q:"自助接入"体现了什么设计思想?**
  A:平台化、自助化。把"建知识库"封装成一个标准 Tool,业务方自己调用就能接入,不用每次找开发定制,降低协作成本——这正是共享基础设施的价值。

### 条目 5:设计双层记忆架构——对话记忆按 project_id + session_id 隔离支持最近 N 轮召回;用户画像按 user_id 跨项目共享,基于 PostgreSQL UPSERT(ON CONFLICT DO UPDATE)幂等写入,客服 Agent 自动提取用户事实(过敏/预算/偏好),写作 Agent 即时读取,实现跨项目用户认知同步

**这条在说什么**:记忆分两层,隔离粒度不同。① **对话记忆(短期)**:按 project_id+session_id 隔离,只在当前会话内召回最近 N 轮,各会话互不干扰;② **用户画像(长期)**:按 user_id **跨项目共享**——比如客服 Agent 聊天时自动提取出"这个用户过敏/预算多少/偏好什么",用 PostgreSQL 的 UPSERT 幂等写入画像;之后写作 Agent 或别的项目能**即时读到同一个用户的认知**,实现跨项目的用户认知同步。

**可能的提问与答:**

- **Q:为什么对话记忆要隔离,而用户画像要跨项目共享?**
  A:对话记忆是"此时此地这次会话"的上下文,跨会话/跨项目串了会错乱,所以按 session 隔离;用户画像是"这个人是谁"的稳定事实(过敏、预算、偏好),跨项目共享才能让所有 Agent 对同一用户有一致认知,体验连贯。一个要隔离、一个要共享,按数据性质决定。
- **Q:UPSERT(ON CONFLICT DO UPDATE)在这里为什么用?**
  A:用户画像是"有则更新、无则插入"——同一个 user_id 第一次写是插入,后续提取到新事实是更新。UPSERT 一条语句搞定且幂等,反复写同一用户不会产生重复记录。
- **Q:用户事实是怎么提取的?**
  A:客服 Agent 在对话过程中,用 LLM 从用户话里抽取结构化事实(如过敏源、预算、偏好),写入画像表。这样画像是随对话自动积累的,不用人工维护。

### 条目 6:基于 MCP 原生 Prompt 原语构建模板中心,YAML 定义参数 Schema、服务端渲染为 PromptMessage,模板修改即时生效无需重启,实现提示词工程与业务逻辑解耦

**这条在说什么**:把提示词也做成共享服务。用 MCP 自带的 **Prompt 原语**建一个模板中心:用 YAML 定义每个模板的参数 Schema,调用时由服务端把参数渲染成标准的 PromptMessage 返回。好处是**改 Prompt 只改 YAML、即时生效、不用重启服务**,让"写提示词"和"写业务代码"彻底解耦(对应你技能里的 Prompt 版本管理)。

**可能的提问与答:**
- **Q:为什么要把 Prompt 抽成模板中心,而不是写在代码里?**
  A:Prompt 是 LLM 应用的"源代码",改动频繁。写死在代码里,改一句话就要改代码、重新发版。抽成模板中心后,改 YAML 即时生效、无需重启,产品/运营也能调,提示词工程和业务逻辑解耦,迭代快很多。
- **Q:用 MCP 的 Prompt 原语有什么好处?**
  A:标准化。MCP 原生就定义了 Prompt 这种原语,用它做模板中心,所有 Agent 按统一协议拿提示词,和其他 MCP 能力(Tool/RAG)一套体系,不用另造一套提示词管理机制。

### 条目 7:实现 Agent 七步编排流水线(召回历史→读画像→检索知识→渲染 Prompt→LLM 推理→存记忆→提取事实),全部经 GatewayClient 统一调用,Agent 无状态、状态下沉到共享服务;各服务独立构建 Docker 镜像部署至阿里云 Serverless(FC/SAE),按拓扑分层发布、灰度 + 秒级回滚,SIGTERM 优雅终止保障在途请求不丢

**这条在说什么**:① **七步编排**:一个 Agent 处理请求的标准流程——召回历史对话→读用户画像→检索知识(RAG)→渲染 Prompt→调 LLM 推理→存这轮记忆→提取新的用户事实,这七步全部通过 GatewayClient 调共享服务完成;② **Agent 无状态**:Agent 本身不存任何状态,所有状态(记忆、画像)都下沉到共享服务——所以 Agent 可以随意水平扩展;③ **部署**:每个服务独立打 Docker 镜像,部署到阿里云 Serverless(函数计算 FC / Serverless 应用引擎 SAE),按依赖拓扑分层发布、支持灰度和秒级回滚,SIGTERM 优雅终止保证发版时在途请求不丢。

**可能的提问与答:**
- **Q:为什么要让 Agent 无状态、状态下沉到共享服务?**
  A:无状态的 Agent 才能随意水平扩展(任意一个实例都能处理任意请求)、挂了也不丢数据。把记忆、画像这些状态放到共享服务里集中管理,Agent 只做编排逻辑,这是分布式高可用的标准做法。
- **Q:七步流水线把前面几个共享服务怎么串起来的?**
  A:召回历史、存记忆用"记忆服务",读/提取画像用"画像(记忆)服务",检索知识用"RAG 服务",渲染 Prompt 用"模板中心",LLM 推理用"LLM 网关"——七步正好把前面抽离的几个 MCP Server 全用上了,Agent 只负责按顺序编排调用。
- **Q:SIGTERM 优雅终止是干什么的?**
  A:Serverless/容器发版或缩容时会发 SIGTERM 信号通知进程退出。优雅终止是收到信号后先停接新请求、等在途请求处理完再退,避免请求被硬切断报错。LLM 请求耗时长,这点尤其重要。

## 项目六 · 高频追问深挖(真实面试问到的四块)

> 这四块是真实面试按时间轴追问的,层层递进:RAG 召回细节 → 多 Agent 协作 → MCP 原理 → Skill 等其它工具形式。
> ⚠️ 涉及具体数字(知识库多少条、几个 Agent、召回多少)**务必提前定一个真实口径**,别现场卡壳。

### 追问 1:RAG 召回——知识库多少条?每次召回多少?标签/向量/关键词怎么组合?

**① 知识库多少条**(按实填):客服知识库目前在**几千到上万条 chunk** 量级,来自数据中台 approved 数据 + 素材中心 retrieval_text 向量化;考核那块按**十万级**做了演进设计。

**② 每次召回多少**——讲"宽召回、精排、按场景收口":
- 粗召回:向量检索 **top-20**(宽召回不漏);
- 精排:Cross-Encoder rerank 取 **top-5**;
- 实际喂 LLM:语音场景再精简到 **1-3 条**(控 TTFT 和 token),文本场景 3-5 条。

**③ 标签/向量/关键词怎么组合**——三层漏斗,组合而非单选:
```
① Metadata 标签过滤(先缩范围) → ② 混合检索:向量(语义)+关键词(精确) → ③ Rerank 精排
```
- **标签过滤(Metadata)**:先按 category/tags/project_id 把范围缩小,提精度也提速;
- **混合检索(HNSW_HYBRID)**:稠密向量管语义(换说法也命中)+ 稀疏向量/关键词管精确命中(型号、专有名词),用 **dense_weight** 加权(说话模糊调高偏语义、含术语调低偏关键词);
- **Rerank**:Cross-Encoder 对融合结果精排,把最相关的拎到前面。

**一句话**:标签先过滤、向量管语义、关键词管精确、rerank 管精排——漏斗式组合:Metadata 缩范围 → 向量+关键词混合检索(dense_weight 调权) → 精排 top-5。

### 追问 2:多 Agent 共享服务集群——有几个 Agent?它们怎么协作?

**① 有几个 Agent**(按实填):约 4 个业务 Agent——**客服 Agent、内容生成/写作 Agent、销售考核 Agent(出题+评判)、(可选)直播切片 Agent**,共用同一套底层共享服务。

**② 怎么协作(灵魂答案):不靠 Agent 直接对话,而是通过"共享状态层"间接协作——状态下沉、Agent 无状态。**
- 每个 Agent 都**无状态**,所有状态(记忆、用户画像、知识库)下沉到共享 MCP Server。
- 典型协作例子:**客服 Agent 在对话中自动提取用户事实(过敏/预算/偏好)写进共享的用户画像服务;之后写作 Agent / 别的项目 Agent 直接读这份画像**——两者从不直接通信,而是通过共享画像服务实现跨项目用户认知同步。
- 这种"通过共享状态协作"比"Agent 直接对话"更稳定解耦:Agent 可随意扩展、互不依赖,新增 Agent 不用改别人。

**对比加分**:业界多 Agent 有两种思路——**Supervisor 主管调度+子 Agent(直接通信编排)** vs **通过共享状态/黑板模式解耦**。我们选后者,因为场景是多业务长期并存、要跨项目共享认知,解耦比强编排更合适。

**③ 单 Agent 内部**:走七步编排流水线——召回历史 → 读画像 → 检索知识 → 渲染 Prompt → LLM 推理 → 存记忆 → 提取事实,全经 GatewayClient 调共享服务。

### 追问 3:MCP——怎么理解 MCP?工具包怎么解析?工具怎么暴露给模型?

**① 怎么理解**:MCP(Model Context Protocol)是 **Anthropic 提出的开放标准**,标准化 LLM 应用和外部工具/数据/能力的连接方式。比喻:**MCP 是 AI 世界的 USB-C 接口**,工具和 Agent 即插即用、解耦。要点:
- **Client-Server 架构**:Agent 是 Client,能力方是 Server;
- **三大原语**:**Tools**(模型可调的函数/动作)、**Resources**(可读数据/上下文)、**Prompts**(提示词模板);
- **传输层**:本地 stdio,远程 **Streamable HTTP**(你项目用的)。

**② 工具包怎么解析**:Client 连 Server 时先 **initialize 握手**交换 capabilities,再调 **`tools/list`** 拿工具清单;每个工具是标准结构 **name + description + inputSchema(JSON Schema)**,Client 解析后就知道有哪些能力、怎么调。

**③ 工具怎么暴露给模型**:把 MCP 工具转成模型的 **tool-use/function-calling 格式**(name+description+JSON Schema)注入请求;模型自己决定调哪个、传什么参数,输出 tool_call;Client 再用 **`tools/call`** 去 Server 执行,把结果回传模型继续生成。

**一句话**:`tools/list` 解析工具清单 → 转成 function-calling 格式暴露给 LLM → 模型决定调用 → Client 用 `tools/call` 执行并回传。

### 追问 4:Skill——除了 MCP 还有什么工具形式?Skill 怎么存、怎么被 AI 用?

**① 除了 MCP 还有哪些"给模型加能力"的形式**:
1. **原生 Function Calling / Tool Use**(最基础,API 里直接定义 tools);
2. **MCP**(把工具标准化成协议、跨应用复用);
3. **Agent Skills(Skill)**(打包"某类任务的知识+流程+脚本");
4. **内置工具**(web search、code interpreter、computer use);
5. **框架封装**(LangChain Tools 等)。

**② MCP 和 Skill 的本质区别(必答清)**:
- **MCP 偏"连接外部能力"**——运行时调外部工具和数据(偏动作/集成);
- **Skill 偏"封装专业知识和流程"**——更像给模型一本"某类任务的操作手册",可附脚本资源(偏指令/方法论);
- 比喻:**MCP 是给模型插上新工具,Skill 是给模型一本操作手册**;且 Skill 里能写"这一步去调某个 MCP 工具",两者互补。

**③ Skill 怎么存**:本质是**文件系统里的一个文件夹**,核心是 **`SKILL.md`**(YAML frontmatter 关键就 **name + description** 两字段,正文是操作指令),文件夹里还能打包脚本、模板、参考资料。

**④ 怎么被 AI 用(核心:渐进式披露 Progressive Disclosure)**:
- 启动时**只把每个 Skill 的 name+description 加载进上下文**(很轻量),模型只"知道有这么个 Skill";
- **当模型判断任务和某 Skill 相关时,才读取完整 SKILL.md 正文和关联文件**,按指令执行;
- 好处:**可以挂很多 Skill 而不撑爆上下文**——没用到只占一行,用到才完整加载,context 效率远高于一次性塞所有工具说明。

**一句话**:Skill 是打包成文件夹的"任务操作手册"(SKILL.md+脚本),靠渐进式披露使用——平时只占一行 name/description,相关时才完整读取;它和 MCP 互补:MCP 连外部工具(动作),Skill 封装知识流程(方法)。

## 模块一、Python 后端开发(FastAPI、asyncio、Pydantic)

### 核心概念速记

- **FastAPI**:基于 ASGI 的异步 Web 框架,自带 Pydantic 数据校验和自动 OpenAPI 文档,性能接近 Node/Go。
- **asyncio**:Python 的协程并发库,用单线程事件循环处理大量 I/O 等待,适合高并发 I/O 密集场景(如调大模型 API)。
- **Pydantic**:基于类型注解的数据校验/序列化库,FastAPI 的请求体、LLM 结构化输出都靠它。

### Q1. 为什么 LLM 应用后端首选异步(asyncio/FastAPI)?
**答:** LLM 应用是典型的 **I/O 密集型**——大部分时间在等待模型 API、向量库、数据库返回,CPU 几乎空闲。同步模型下一个请求会阻塞整个线程;而 asyncio 用事件循环,在等待 I/O 时切换去处理别的请求,单进程就能扛住成千上万并发连接。FastAPI 原生 ASGI + async def,天然契合这种"高并发、每个请求都在等外部响应"的场景。

### Q2. 同步代码和异步代码混在一起会怎样?如何做"同异步桥接"?
**答:** 在 async 函数里直接调用一个**阻塞的同步函数**(比如某个只有同步版的 SDK、CPU 密集计算),会卡住整个事件循环,导致所有并发请求一起变慢。解决办法是把同步调用丢到线程池/进程池:
- `await asyncio.to_thread(sync_func, args)`(Python 3.9+,最简洁);
- `loop.run_in_executor(executor, sync_func, args)`;
- FastAPI 里把路由写成普通 `def`(它会自动放线程池)。

反过来,在同步上下文里要跑异步代码,用 `asyncio.run()` 或 `asyncio.run_coroutine_threadsafe()`。核心原则:**别让阻塞调用占着事件循环**。

### Q3. 讲讲指数退避重试(exponential backoff),为什么不直接固定间隔重试?
**答:** 调用 LLM/外部 API 会遇到限流(429)、超时、瞬时 5xx。重试是必要的,但固定间隔会在服务过载时形成"重试风暴"加剧雪崩。指数退避让每次重试间隔成倍增长(如 1s、2s、4s、8s),给下游恢复时间;再加 **抖动(jitter)** 随机扰动,避免大量客户端同步重试。还要设**最大重试次数和最大间隔上限**,并区分**可重试错误**(429/503/超时)和**不可重试错误**(400/401,重试无意义)。生产里常用 `tenacity` 库实现。

### Q4. Pydantic 在你的项目里解决了什么?v1 和 v2 有区别吗?
**答:** 两大用途:① **入参校验**——FastAPI 用 Pydantic 模型声明请求体,自动校验类型、范围、必填项,非法请求直接 422,业务代码拿到的一定是干净数据;② **LLM 结构化输出**——用 Pydantic Schema 约束模型返回 JSON,再解析校验。v2 用 Rust 重写核心(pydantic-core),校验速度快好几倍,API 上 `@validator` 改成 `@field_validator`、`.dict()` 改 `.model_dump()` 等。

## 模块二、RAG 检索增强生成全链路

### 核心概念速记

RAG = 检索(Retrieval)+ 生成(Generation):先从知识库检索相关片段,再把片段塞进 prompt 让 LLM 基于事实作答,解决大模型"知识过时、幻觉、无法访问私有数据"的问题。

全链路:**文档切片 → Embedding 向量化 → 建索引 → 检索(混合检索)→ Rerank 重排 → 拼 prompt → 生成**。

### Q5. 文档切片(chunking)有哪些策略?chunk 太大或太小分别有什么问题?
**答:** 常见策略:固定长度切分、按句子/段落切分、递归字符切分(RecursiveCharacterTextSplitter)、按语义切分、按文档结构(Markdown 标题/表格)切分。关键参数是 **chunk_size** 和 **overlap(重叠)**。

- chunk **太大**:检索精度下降(一个 chunk 混入太多无关内容,且稀释了 embedding 的语义),还浪费 context、增加成本。
- chunk **太小**:语义不完整、上下文割裂,可能丢失回答所需的关联信息。

通常加 overlap(如 10~20%)避免把一句话/一个概念从中间切断。实际要按文档类型和检索效果调。

### Q6. 什么是 HNSW?为什么向量检索常用它?
**答:** HNSW(Hierarchical Navigable Small World)是一种**近似最近邻(ANN)**图索引算法。它构建多层的小世界图,上层稀疏用于快速跳转、下层稠密用于精确定位,查询时从顶层贪心导航到底层,在**亚线性时间**内找到近似最近邻。相比暴力 KNN(O(N) 全量比对),HNSW 在百万/千万级向量下能做到毫秒级查询,召回率也很高。代价是**构建慢、内存占用大**,关键参数 `M`(每个节点连接数)、`ef_construction`(构建时搜索宽度)、`ef_search`(查询时搜索宽度,越大越准但越慢)。

### Q7. 什么是混合检索(hybrid search)?为什么纯向量检索不够?
**答:** 纯向量(稠密)检索擅长**语义相似**,但对**精确关键词、专有名词、缩写、编号**不敏感(比如型号"A1234"语义上和别的型号很近)。稀疏检索(BM25/关键词)正好相反,擅长精确匹配但不懂语义。**混合检索**把两者结果融合(常用 RRF 倒数排名融合,或加权融合),兼顾语义和关键词。简历里的 **dense_weight** 就是调节稠密检索在融合中的权重——专业术语多的场景调低 dense_weight(更靠关键词),开放问答场景调高(更靠语义)。

### Q8. Rerank(重排)是干什么的?Cross-Encoder 和 Bi-Encoder(Embedding)区别?
**答:** 第一轮检索为了快,用 **Bi-Encoder**:query 和 doc **分别独立**编码成向量再算相似度,可以预先把所有 doc 向量算好建索引,检索快但精度有限。**Rerank** 是第二阶段,对第一轮召回的 Top-K(比如 50 条)用 **Cross-Encoder**:把 query 和 doc **拼在一起**送进模型做交叉注意力,直接输出相关性分数,精度高很多但慢(不能预计算)。所以工程上是 **"Bi-Encoder 粗召回大量 + Cross-Encoder 精排少量"** 的两阶段架构,兼顾速度和精度。

### Q9. Int8 量化在 RAG 里用在哪?有什么收益和代价?
**答:** 向量量化把 float32 的 embedding 压成 int8(甚至二值),**内存占用降到 1/4、检索更快**,适合海量向量场景。代价是**精度有损**,所以常配合"量化粗筛 + 原始向量精算"的 rescore 策略来补偿。除了存储量化,推理侧也有 Int8 量化(模型权重),原理类似——用精度换速度和显存。

### Q10. Metadata 过滤(元数据过滤)怎么用?为什么重要?
**答:** 给每个 chunk 打上结构化标签(来源、时间、部门、文档类型、权限等),检索时先按 metadata 过滤再做向量检索(或反之)。作用:① **精度**——把检索范围限定在相关子集,排除无关文档;② **权限/合规**——只检索用户有权访问的文档;③ **时效**——只查最近文档。注意过滤与向量检索的执行顺序(pre-filter / post-filter)会影响性能和召回。

### Q11. 如何评估一个 RAG 系统好不好?
**答:** 分两层:**检索质量**(召回的片段对不对)——用 Recall@K、MRR、命中率;**生成质量**(回答对不对)——用忠实度(faithfulness,答案是否忠于检索内容、有无幻觉)、答案相关性、上下文相关性。可借助 RAGAS 等框架,也做人工/A-B 盲评。出问题先定位是**检索没召回**还是**召回了但模型没用好**,两者优化方向完全不同。

## 模块三、Prompt Engineering 与结构化输出

### Q12. Prompt 版本管理和自动化评测为什么重要?怎么做?
**答:** Prompt 是 LLM 应用的"源代码",改一个字就可能影响所有输出,所以要像代码一样**版本化**(存仓库/专门的 prompt 管理平台,带版本号、变更记录)。**自动化评测**:维护一个测试集(输入 + 期望表现),每次改 prompt 跑一遍,用指标(准确率、格式合规率、相关性打分,或用 LLM-as-judge)对比新旧版本,**用数据决定是否上线**,而不是凭感觉。这样能防止"改好了一个 case 却悄悄改坏十个"。

### Q13. 什么是负样本优化?
**答:** 收集模型**答错/答偏/越界**的真实案例(负样本),针对性地在 prompt 里加约束、加反例、加"不要这样做"的指令,或把它们纳入评测集做回归。本质是用失败案例驱动 prompt 迭代,堵住已知的坑。

### Q14. 怎么让 LLM 稳定输出结构化数据(JSON)?JSON Mode 和 Pydantic Schema 怎么配合?
**答:** 几种手段叠加用:① **JSON Mode / Structured Output**——很多模型 API 支持强制输出合法 JSON,从解码层面保证格式;② 把 **Pydantic 模型转成 JSON Schema** 传给模型(function calling / response_format),告诉它字段、类型、约束;③ 拿到结果用 **Pydantic 解析校验**,不合规就重试或修复。这样从"约束生成"到"校验落地"形成闭环。纯靠 prompt 说"请返回 JSON"不可靠,容易夹带解释文字或格式错误。

## 模块四、LLM 推理性能调优

### 核心概念速记

- **TTFT(Time To First Token)**:从请求发出到吐出第一个 token 的延迟,决定用户"感觉快不快"。
- **KV Cache**:自回归生成时,把已生成 token 的 Key/Value 缓存下来,避免每生成一个新 token 都重算前面所有 token 的注意力。
- **Prefix Caching**:把多个请求**共享的前缀**(如相同 system prompt、相同文档)的 KV Cache 缓存复用,避免重复计算。

### Q15. 解释 KV Cache,它为什么能加速?占用什么资源?
**答:** Transformer 自回归生成是逐 token 进行的,每步都要算当前 token 对**之前所有 token** 的注意力,需要历史 token 的 Key/Value。没有缓存的话每步都重算前缀,复杂度爆炸。KV Cache 把历史 token 的 K/V 存在显存里,每生成一个新 token 只算它自己的 K/V 并追加,把每步从"重算全部"降为"增量计算"。代价是**占用大量显存**,且显存占用随序列长度和并发数线性增长——这也是长上下文、高并发推理的主要显存瓶颈(衍生出 PagedAttention 等优化)。

### Q16. Prefix Caching 适合什么场景?能带来什么?
**答:** 当大量请求**共享相同前缀**时收益最大,比如:统一的长 system prompt、同一篇文档的多轮问答、few-shot 模板。把这段前缀的 KV Cache 算一次缓存住,后续请求直接复用,**大幅降低 TTFT 和重复计算**。RAG 里如果把固定指令放前面、变化的 query 放后面,就能最大化前缀命中。vLLM、SGLang 等推理引擎都内置了 Automatic Prefix Caching。

### Q17. 怎么优化 TTFT?
**答:** TTFT 主要由 **prompt 长度(prefill 计算量)** 和排队等待决定。手段:① **Prefix Caching** 复用共享前缀,跳过重复 prefill;② **精简 prompt**、压缩上下文,减少 prefill token;③ **流式输出(SSE)**——虽然不减少首 token 计算,但让用户尽早看到内容,体感更快;④ 推理引擎层面用 **continuous batching、chunked prefill**;⑤ 合理的并发与排队策略,避免请求堆积。

### Q18. 什么是 Token 分级控制和多模型动态路由?
**答:**
- **Token 分级控制**:对不同任务/字段设置不同的 `max_tokens` 上限,该短的短(如分类只要一个词)、该长的长(如总结),避免无意义的长输出浪费时间和成本。
- **多模型动态路由**:不是所有请求都用最贵最强的模型。按任务难度/类型路由——简单任务走小模型(快、便宜),复杂任务走大模型;也可做降级(大模型超时/限流时降级到备用模型)。在**成本、延迟、质量**间动态权衡,是生产级 LLM 服务的关键优化。

## 模块五、大模型微调全流程(SFT、DPO、LoRA)

### 核心概念速记

- **SFT(监督微调)**:用"指令-期望回答"对训练模型,让它学会按你的格式/风格/任务作答。
- **RLHF**:用人类偏好训练奖励模型,再用强化学习(PPO)优化策略模型,流程复杂、训练不稳定。
- **DPO(直接偏好优化)**:跳过奖励模型和 RL,直接用"偏好对(chosen/rejected)"做类似分类的优化,更简单稳定。
- **LoRA/QLoRA**:参数高效微调,只训练少量低秩适配矩阵,冻结原模型权重。

### Q19. 完整微调流程是怎样的?数据清洗为什么关键?
**答:** 典型流程:**数据清洗 → SFT → DPO**。先收集清洗高质量指令数据(去重、去噪、去敏感/有害、格式统一、平衡分布),SFT 让模型掌握任务和格式,再用偏好数据做 DPO 对齐人类偏好。**数据质量决定上限**——垃圾进垃圾出,脏数据会让模型学到错误模式,清洗往往是最耗时也最影响效果的一步。

### Q20. LoRA 和 QLoRA 的原理与区别?为什么用参数高效微调?
**答:** **LoRA**:冻结原模型权重,在权重旁注入两个小的低秩矩阵 A、B(ΔW = B·A),只训练这部分(可能不到原参数的 1%),训练快、显存省、产出的 adapter 很小可热插拔。**QLoRA**:在 LoRA 基础上把**基座模型量化到 4-bit(NF4)** 再训练,显存进一步大降,让单张消费级显卡也能微调几十亿参数模型。用它们的原因:全参数微调显存和算力成本极高,而 LoRA/QLoRA 用极小代价就能达到接近全参微调的效果。工具上我用过 **LLaMA-Factory**(开源、配置化微调)和**火山引擎方舟平台**(托管训练)。

### Q21. RLHF 和 DPO 有什么区别?各自适用什么场景?
**答:** **RLHF** 三阶段:训奖励模型 → 用 PPO 强化学习优化。表达能力强、上限高,但**流程复杂、训练不稳定、超参敏感、算力贵**,要同时维护多个模型。**DPO** 用一个推导出的损失函数,直接在偏好对上优化策略模型,**无需单独奖励模型、无需在线采样,稳定且易实现**。适用:数据是成对偏好、追求快速稳定对齐时优先 DPO;需要复杂奖励塑形、在线探索、或已有成熟 RLHF 流水线时用 RLHF。现在多数团队默认从 DPO 起步。

## 模块六、LangGraph / LangChain 工作流编排

### Q22. LangChain 和 LangGraph 的定位区别?
**答:** LangChain 提供 LLM 应用的组件和封装(模型、工具、`create_agent` 等),快速搭建。LangGraph 面向**有状态、可控的复杂工作流**,把流程显式建模成"节点+边+状态"的有向图,能精确控制循环、分支、人机协同、断点续跑。简单应用用 LangChain 的高层 API;复杂多步、需要精细控制时用 LangGraph(`create_agent` 底层就是 LangGraph)。

### Q23. 什么是 SubGraph(子图)?为什么要用?
**答:** 子图是把一段独立的工作流封装成一个图,再作为节点嵌入到父图里。好处:**模块化、复用、隔离状态**。比如一个大 Agent 里"检索子流程""审核子流程"各自是独立子图,可单独开发测试、在不同主流程间复用,也便于多智能体协作(每个 agent 一个子图)。

### Q24. Checkpointer(状态持久化)解决什么问题?
**答:** Checkpointer 在图执行的每一步保存状态快照,配合 `thread_id` 实现:① **多轮记忆**(同一会话续上历史);② **断点续跑/容错**(崩溃后从检查点恢复,不用从头跑);③ **人机协同 human-in-the-loop**(在某节点暂停等人审批,再继续);④ **时间回溯**(回到历史某步重新执行)。开发用 InMemorySaver,生产用数据库支持的持久化 checkpointer(如 Postgres)。

### Q25. SSE 流式输出是什么?为什么 LLM 应用爱用它?
**答:** SSE(Server-Sent Events)是基于 HTTP 的服务端单向推送。LLM 逐 token 生成,用 SSE 把 token 实时推给前端,用户能像打字机一样即时看到内容,**大幅改善体感延迟**(尤其长回答),还能中途取消。相比 WebSocket,SSE 更轻量、单向、自动重连、走标准 HTTP,正好契合"服务端持续吐 token"的需求。

### Q26. LangSmith 用来干什么?
**答:** LangSmith 是 LangChain 的**可观测性/追踪平台**。它记录每次运行的完整链路:每一步的模型调用、prompt、工具调用、输入输出、耗时、token 消耗、报错。用于**调试**(为什么没调工具、哪步出错)、**评测**(跑数据集打分)、**监控**(线上质量与成本)。设置环境变量 `LANGSMITH_TRACING=true` + API Key 即可开启。

## 模块七、RTC 实时语音链路

### 核心概念速记

- **ASR**:自动语音识别,语音转文字。**TTS**:文字转语音合成。**声音复刻**:用少量样本克隆特定音色。
- **RTC**:实时音视频通信,对延迟极敏感。

### Q27. 一条实时语音对话链路是怎样的?延迟瓶颈在哪?
**答:** 链路:**用户说话 → ASR 转文字 → LLM 生成回复 → TTS 合成语音 → 播放**。每一环都加延迟,端到端体验要求通常 <1~2 秒。优化点:ASR/TTS 都做**流式**(边说边识别、边生成边播放)、LLM 用流式输出并尽早把首句送去 TTS、各环节并行流水线化,而不是串行等全部完成。TTFT 和首句 TTS 延迟是体感关键。

### Q28. 弱网下语音怎么保证质量?讲讲 FEC 和 Jitter Buffer。
**答:**
- **FEC(前向纠错)**:发送端额外发冗余数据,接收端在**丢包时用冗余直接恢复**,无需重传(重传在实时场景延迟太高)。代价是增加带宽,需在冗余度和丢包率间权衡。
- **Jitter Buffer(抖动缓冲)**:网络抖动导致语音包到达时间不均。接收端用一个缓冲区先暂存、按时序平滑播放,消除抖动带来的卡顿。**自适应** Jitter Buffer 会根据实时网络状况动态调整缓冲大小——网络差就加大缓冲(更稳但延迟高),网络好就减小(低延迟)。两者配合是弱网语音流畅的关键。

## 模块八、后端服务治理

### Q29. 限流怎么做?为什么需要?(SlowAPI)
**答:** 限流防止突发流量/恶意请求打垮服务,也控制成本(LLM 调用很贵)。常见算法:固定窗口、滑动窗口、**令牌桶**(允许一定突发)、漏桶(平滑)。**SlowAPI** 是 FastAPI/Starlette 的限流中间件,可按 IP、用户、API key 设置速率(如每分钟 N 次)。LLM 场景还常按 **token 数**而非请求数限流。

### Q30. 熔断和降级有什么区别?
**答:** **熔断(circuit breaker)**:当下游连续失败超过阈值,熔断器"跳闸",一段时间内**直接拒绝/快速失败**不再请求下游,给它恢复时间,避免请求堆积拖垮自己(雪崩);过段时间半开试探,恢复了再闭合。**降级(fallback)**:当主功能不可用时,提供一个**次优但可用**的结果,比如主模型挂了切备用模型、检索失败时返回缓存答案或友好提示。熔断是"保护自己别被拖死",降级是"保证还能给用户一个交代"。

### Q31. 优雅关闭(graceful shutdown)和健康检查为什么重要?
**答:** **优雅关闭**:收到停止信号后,**停止接收新请求,但等正在处理的请求完成**(尤其 LLM 请求可能几十秒)再退出,避免请求被硬切断、数据不一致。**健康检查**:暴露 `/health` 端点供负载均衡/K8s 探活(liveness/readiness)。readiness 不就绪时不给它转发流量,liveness 失败则重启容器。两者保证滚动发布、扩缩容时**用户无感、不丢请求**。

### Q32. 结构化日志和 request_id 链路追踪是干什么的?(structlog)
**答:** **结构化日志**(structlog 输出 JSON 而非纯文本)让日志可被机器解析、检索、聚合,便于在 ELK/Loki 等系统里按字段查询。**request_id**:每个请求入口生成唯一 id,贯穿这次请求经过的所有函数/服务/日志,出问题时按 request_id 就能串起整条调用链,快速定位是哪一步、哪个请求出错。在异步高并发下,这种链路追踪几乎是排障的必备手段。

## 模块九、数据库与模型评估

### Q33. 为什么用 PostgreSQL?它和向量数据库怎么配合?
**答:** PostgreSQL 是成熟的关系型数据库,事务、可靠性、生态都强,存业务数据/元数据/会话。向量检索可以用专门的向量库(Milvus、Qdrant 等),也可以直接用 **PostgreSQL + pgvector 扩展**——好处是业务数据和向量同库,能在一条 SQL 里同时做 metadata 过滤 + 向量检索,运维简单。海量向量、极致检索性能要求时才上专用向量库。

### Q34. 介绍下你用过的模型评估指标。
**答:** 按场景分:
- **BLEU**:看生成文本和参考文本的 n-gram 重叠,偏精确率,常用于翻译,对措辞敏感。
- **ROUGE**:看召回(参考里的内容被覆盖多少),常用于摘要。
- **语义相似度**:用 embedding 算生成与参考的余弦相似度,不拘泥字面、更贴近语义,适合开放式回答。
- **A/B 盲评**:人工对两个版本输出盲选更优,最贴近真实体验,但成本高。
- **Cohen's Kappa**:衡量**多个标注者之间的一致性**(扣除随机一致后的吻合度),用来验证人工评测靠不靠谱、标注标准是否清晰。

BLEU/ROUGE 是字面匹配、快但片面;语义相似度更聪明;开放任务最终还得靠人工 A/B 盲评,并用 Kappa 校验评测本身的可信度。也越来越多用 **LLM-as-judge** 做自动化打分。

### Q35.(综合)如果让你从零搭一个生产级 RAG 问答系统,你会怎么设计?
**答:** 一个能体现全栈思路的串讲:
1. **数据层**:文档解析 → 合理切片(带 overlap)→ Embedding → 存 pgvector/向量库 + 打 metadata。
2. **检索层**:混合检索(向量 + BM25,调 dense_weight)→ Cross-Encoder Rerank → metadata 过滤/权限控制。
3. **生成层**:固定指令放前面(吃 Prefix Caching),把 Top-K 片段拼 prompt,用 Pydantic Schema 约束结构化输出,流式 SSE 返回。
4. **编排**:LangGraph 编排检索→生成→校验循环,Checkpointer 做多轮记忆,LangSmith 全链路追踪。
5. **服务治理**:FastAPI 异步 + SlowAPI 限流 + 熔断降级 + 优雅关闭 + 健康检查 + structlog/request_id。
6. **性能**:多模型路由(简单走小模型)、TTFT 优化、KV/Prefix Cache、Token 分级控制。
7. **评估迭代**:离线测试集 + RAG 指标(召回/忠实度)+ A/B 盲评 + prompt 版本管理与自动化评测,数据驱动迭代。

这条主线把简历上几乎所有技能点都串起来了,面试时按这个骨架展开,既全面又有条理。

## 附:面试应答小贴士

1. **先给一句话定义,再展开**——别一上来堆细节。
2. **主动讲取舍**:任何技术都讲"为什么用它 / 代价是什么 / 还有什么替代方案",体现工程判断。
3. **结合场景**:把概念落到"我在项目里用它解决了 XX 问题",比纯背概念有说服力。
4. **被问到不熟的**:坦诚边界,讲清你理解的部分和你会怎么查证,别硬编。
5. **数字和指标**:能说出量化效果(延迟降了多少、召回提了多少)最加分。
