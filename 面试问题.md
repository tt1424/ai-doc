## 四、录音四（06-17 15:03 · RAG/向量检索深挖 + 手撕代码）

> 本场面试官对 RAG、向量检索、量化的追问非常细，且要求手写代码、抽查计算机基础。

### 1. RAG / 检索（被反复深挖）
1. RAG 检索具体怎么做的？（固定长度切片会把问答切开 → 改成「一问一答」整条切分）
2. 怎么做索引优化？（向量降维：2000→1000 维，准确率没降多少但检索快很多；给数据打标签）
3. 怎么动态调发散/严谨？（稠密+稀疏的混合检索，用户说得模糊调高参数更发散、说得准确调低更严谨）
4. 重排（rerank）怎么做？（查回来的不一定排第一，用一个新模型逐条重新打分，把分数最高的排到第一）
5. 重排再展开：拿一个问题向量化 → 去库里查最相近的 50 条 → 按相似度比较，怎么算「像」？
6. 用户检索时具体怎么查？假设 1000 多个文档切片成 1 万个片段，是拿一个块跟这 1 万个逐个比吗？
7. 切片向量化的模型要和查询时的模型一致吗？（要一致才能比对）
8. 那是不是就是「一个查询向量 × 库里 1 万个向量」做暴力/全局相似度比较？
9. 你们怎么避免全局比对？（打标签做导航：金字塔式大分类→小分类，如耳机→蓝牙/有线→具体型号，比 MySQL 全局查找快很多）
10. 混合检索里稠密向量和稀疏向量分别负责什么？（稠密=语义/模糊，稀疏=精确）

### 2. 量化 / 存储
11. int8 量化是什么意思？（每个维度的数据类型，从 float32 → int8，存储降约 75%）

## 五、四场面试的共性考点（建议重点准备）

| 维度 | 高频问题 | 你需要讲清楚的点 |
|------|---------|----------------|
| **RAG 细节** | 切分方式、阈值设置、重排、混合检索、Redis vs 向量库的取舍 | 为什么按「一问一答」切、阈值怎么定、重排用什么模型、动态调参逻辑 |
| **向量检索原理**（录音四追问极深） | 查询向量 × 库里 N 个向量怎么比、是不是暴力全局比对、怎么用标签/导航加速、稠密 vs 稀疏 | ANN 近似最近邻、用标签做粗筛再精排、稠密=语义/稀疏=精确、切片与查询用同一 embedding 模型 |
| **量化 / 存储** | int8 量化是什么、降多少存储、向量维度、怎么设置 | float32→int8 省约 75%、降维 2000→1000、建索引时指定数据类型 |
| **意图识别** | 是否先做意图识别、模糊查询怎么处理 | 识别失败时的兜底（追问 / 调高发散度再查） |
| **性能优化** | 语音延迟从 15s→2s 怎么做的、首字延迟、语音端到端 | 选快的大模型(Flash)、KV/前缀缓存、Redis 预缓存、降维、打点计时逐项优化、两步转换→一步 |
| **微调** | 工具、数据量、模型大小、卡、次数 | LLaMA-Factory、约 1000 条、3B、4090、目的是更拟人 |

## 六、第五场面试 · 问题 + 参考答案（微调/RAG/Skill 深挖）

> 本场按时间轴追问，主线是：SFT/DPO → 微调与 RAG 边界 → 多意图处理 → 工具调用与链路 → Skill → AI Coding → HR/复盘/评估。下面每题给可直接讲的参考答案。
> ⚠️ 期望薪资、最有成就感的项目点**按你真实情况定口径**，下面给的是模板。

### Q1（00:00-03:15）SFT / DPO 怎么训练，训练出来什么？

**答：** 分两阶段，目标不同：
- **SFT（监督微调）**：用「问题 → 标准回答」成对的数据训练,让模型学会**按我们销售的话术风格回答**。工程上「本地小模型验证 + 云端大模型训练」两步——先本地 LLaMA-Factory + Qwen2.5-3B 做 QLoRA 快速验证数据质量(单卡 4090,40 分钟一轮),通过后上火山方舟对豆包 1.5 做生产级 LoRA。**训练出来的是一个"从不会到会说"、能模仿销售风格的模型。**
- **DPO（直接偏好优化）**：对同一个问题让模型采样多条回复,由**销售主管标注 chosen/rejected 偏好对**,训练模型偏向"好"的那种。**训练出来的是在"会说"基础上"说得更好"的模型**,解决会说但说不到点上的问题。
- **一句话**：SFT 解决"从不会到会说"(风格模仿),DPO 解决"会说但说不好"(偏好对齐),最终沉淀出一个**回复像真实金牌销售**的模型,风格一致性人工盲评从 2.1 提到 4.3(5 分制)。

### Q2（03:15-08:25）微调和 RAG 的边界？动态价格/优惠怎么处理？

**答：** 核心原则是 **"微调管风格、RAG 管知识、实时数据走数据库"**：
- **微调**学**不变的东西**——话术风格、语气、销售套路;
- **RAG**管**会变但更新不频繁的知识**——课程介绍、参数、优惠规则,更新只改知识库、不用重训;
- **动态价格/库存这种实时数据,既不微调、也不放静态知识库**,而是**实时查 Redis / MySQL**,查到再拼进 Prompt。
- **为什么价格绝不能微调进模型**：① 价格天天变,微调进去立刻过时,改一次还要重训,成本高且滞后;② 模型可能幻觉报错价;③ 有泄露真实定价的风险(所以数据中台还把价格脱敏设为硬红线)。
- **所以动态价格/优惠的处理**：实时查数据库拿到当前值 → 拼进 Prompt 给模型 → 模型只负责"用销售话术把这个实时数据讲出来"。**让会变的归数据库、不变的归模型,各司其职。**

### Q3（08:25-12:10）面试官质疑"这是不是更像 Prompt"——训练到底沉淀出了什么？

**答：**（这是刁难题,要讲清微调和 Prompt 的本质区别）
- **Prompt 是运行时的临时指令**,治标、每次都要带、占 context、效果取决于当次写得好不好;**微调是把能力"刻进模型参数"**,治本、长期生效。
- **训练沉淀出三样 Prompt 给不了的东西**：
  1. **把几千条优秀对话的风格压缩进权重**——你没法用一个 Prompt 把几千条样例都塞进去(context 装不下,塞多了还不稳);微调能从海量样本里"学"出风格内化到参数。
  2. **能写更短的 Prompt、降低 TTFT**——风格进了参数,Prompt 里就不用再长篇大论强调人设,首字延迟更低。
  3. **一致性更稳**——不依赖每次 Prompt 写得好不好,模型天然就是这个风格。
- **一句话回应质疑**："Prompt 是临时叮嘱,微调是长期培训。能用 Prompt 解决的我们绝不微调;但'让模型从几千条真实销售对话里学会风格、还要稳定一致、还要短 Prompt 降延迟'这件事,Prompt 做不到,只能微调。" 这正是我们践行的"能 RAG/Prompt 解决就不微调,微调只用来沉淀风格"。

### Q4（12:10-16:50）多意图问题：用户一句话问多个问题，怎么拆分、检索、合并？

**答：** 比如用户一句"这个耳机多少钱、有货吗、防水吗?"——一句话三个意图。分三步：
- **拆分**：先用 LLM 做**意图识别 + 问题拆分**,把一句话拆成几个独立子问题(价格 / 库存 / 参数)。
- **检索（分路 + 并行）**：每个子问题**按数据源路由**——价格、库存走 **Redis/MySQL 实时查**,产品参数/防水这种走 **RAG 知识库**;多路**并行**检索省时间。
- **合并**：把各路结果**汇总到一个 Prompt 里**,让 LLM **统一生成一条连贯的回复**,而不是三段割裂的话。
- **关键点**：拆分靠 LLM、检索按意图分流到不同数据源、最后由 LLM 把多路结果**融合成自然口语回复**。这样既不漏问题,回复又连贯。

### Q5（16:50-21:30）工具调用、答案拼接、数据校验、重复删除怎么做？

**答：** 这是把上一题落到工程细节：
- **工具调用**：用 function calling / MCP 工具,模型根据意图决定调哪个工具——查价格工具、查库存工具、检索知识工具,各管一类数据。
- **答案拼接**：把多路结果(Redis 实时数据 + RAG 知识片段)**拼装成统一上下文**喂给 LLM 生成。**注意拼接顺序固定**(固定前缀)以命中前缀缓存降 TTFT。
- **数据校验**：用 **结构化输出 + Pydantic Schema** 校验字段类型/必填;价格、库存等关键数据再用规则(正则/业务规则)二次校验,不合规就兜底或重试。
- **重复删除**：① 检索结果**去重**(同一知识被多路召回);② 用 **MMR** 去重保多样性,避免召回一堆很像的;③ 数据中台侧构建对话块时按 (session, timestamp, content) 三元组去重。

### Q6（21:30-24:05）把完整链路重新串起来。

**答：**（一口气串完,体现全局)
> **用户语音 → ASR 转文字 → 意图识别(多意图则拆分) → 分路检索(RAG 知识库 + Redis/MySQL 实时数据,并行) → Rerank 精排 → 组装 Prompt(固定前缀触发前缀缓存) → 微调后的 LLM 生成 → TTS 转语音 → RTC 推送给用户播放。**

配一句价值点："这条链路里我重点做了两块——检索侧用'标签过滤+混合检索+rerank'保命中率,推理侧用'固定排序+合并 System Prompt+前缀/KV 缓存'把 TTFT 从 15s 压到 2s,让语音客服真正可用。"

### Q7（24:05-28:15）Skill 用过吗？机制是什么？有什么优点？

**答：** 用过,在 Claude Code 里用 Superpowers 这套 Skill 体系做规范编程。
- **机制**：Skill 本质是**打包成文件夹的"任务操作手册"**,核心是一个 **SKILL.md**(YAML frontmatter 关键就 name + description 两字段,正文是详细操作指令),还能附带脚本和资源。靠**渐进式披露(Progressive Disclosure)**工作——**启动时只把每个 Skill 的 name+description 加载进上下文**(很轻量),模型**判断当前任务和某 Skill 相关时,才读取完整 SKILL.md 和关联文件**。
- **优点**：① **context 高效**——可以挂很多 Skill 而不撑爆上下文,没用到只占一行;② **封装专业知识/流程、可复用**;③ **和 MCP 互补**——MCP 连外部工具(动作),Skill 封装知识流程(方法);④ **团队规范可沉淀**——把"规范编程"的规则做成 Skill,AI 每次按统一规范干活。
- **一句话**：Skill 是按需加载的"任务手册",让模型在需要时才调用对应的专业流程,既省 context 又保证做法一致。

### Q8（28:15-30:20）AI Coding 代码规范怎么保证？

**答：** 我用 Claude Code 命令行 + 一套"规范编程"流程来保证,核心是**别让 AI 乱发挥**：
1. **先读项目架构**,让 AI 理解现有代码风格再动手;
2. **做方案先问清楚**——先讨论方案、我确认了再写,不一上来就改;
3. **一步步做**,不一次性大改一片;
4. **做完自测**;
5. **不改没让它改的代码,要改先问**。
- 再加工程兜底：**把团队规范固化成 CLAUDE.md / Skill**(而不是每次口头说)、lint + 类型检查、code review、必要时 TDD。
- **核心观点**："AI 写代码快,但你需求和规范没描述清楚,它就乱改、得不偿失。所以关键是**把规范固化成配置让 AI 遵守**,再用自测和 review 兜底,人始终是'指挥官'。"

### Q9（30:20-31:40）能不能把客服项目转成 Skill 模式？为什么没做？

**答：** 技术上能转一部分,但我们没这么做,原因是**两者目标不同**：
- **Skill 更适合"开发态 / 给 AI 编码助手用的流程封装"**——它的渐进式披露、按需加载,是为"让一个通用 AI 助手按手册一步步操作"设计的;
- **而客服是"运行态的线上服务"**,核心诉求是**毫秒级低延迟、高并发、稳定可控**。客服链路已经是固化的 workflow(意图→检索→生成),用**代码编排 + RAG + 微调**更可控、性能更好;Skill 那种"模型按需读手册再决定怎么做"的不确定性,在实时语音场景反而是负担。
- **更合适的复用方向是 MCP**：我离职后团队规划的项目六,就是把 LLM 网关、RAG、记忆、Prompt 这些**可复用能力抽成 MCP 共享服务**,让各业务复用同一套底座——**这才是客服这类线上服务该走的"复用"路线,而不是 Skill。**
- **一句话**：Skill 封装的是"开发流程/知识",客服要的是"线上低延迟服务",所以我们用 MCP 共享服务来复用,而不是把客服做成 Skill。

### Q12（35:20-38:30）准确率指标、评测集、上线后真实效果怎么评估？

**答：** 分**离线评测**和**上线评估**两层：
- **评测集**：建一批标注好的测试集(问题 + 理想答案),覆盖高频业务场景,作为"标准题库"。
- **离线指标（分两层定位问题）**：
  - **检索层**：命中率 / 召回率(我们 Cross-Encoder 精排后从 72% 提到 89%)、Recall@K——看"该召回的知识有没有召回";
  - **生成层**：语义相似度、忠实度(有没有幻觉)、人工盲评打分——看"答得对不对、像不像人";
  - 用 **Prompt Pilot 自动化评分 + 批量推理**做回归:每次改 Prompt / 换模型就批量跑一遍和理想答案对比,像单元测试一样。
- **上线后真实效果**：① 看**业务指标**——转化率、人工接管率、用户满意度(这些比技术指标更说明问题);② **收集 badcase** 持续迭代;③ **A/B 测试**对比新旧版本;④ 监控线上**延迟、命中率**等。
- **一句话**：离线用评测集跑"检索命中率 + 生成质量"做回归,上线后看"转化率/人工接管率"等业务指标,再用 badcase 闭环 + A/B 持续迭代——**先定位是检索没召回还是模型没用好,再分别优化**。

## 七、第六场面试 · 问题 + 纠正后的正确回答（北京 · LangGraph 深挖 + 场景设计）

> 来源：`2026年06月25日 13点28分.m4a`(约 20 分钟)。本场侧重 **LangChain/LangGraph 选型与深挖、现场场景设计题、混合检索动态调参、微调流程、PDF 解析、切分与记忆**,最后到薪资。

### Q1. 介绍一下你上一家做 AI 的经历 / 这套体系。

**答（精简串讲版）：** 我上一家是跨境电商公司,主营 3C 电子产品(耳机、数据线、运动相机),在 TikTok、亚马逊引流到私域销售成交。我是 AI 这块的负责人,这两年从 0 搭了一套 AI 生态体系,按诞生顺序是:
1. **AI 实时语音客服**——把传统打字客服升级成语音问答,数据来自真实销售聊天记录;
2. **AI 数据中台**——让销售自己上传聊天记录、我们统一清洗入库,替代每次人工导出;
3. **AI 素材中心**——补上图片/视频多模态,人工打标签,让客服能回文字+图片+视频;
4. **自媒体运营 AI(LangGraph)**——AI 生成商品标题/文章/图,产出再回流素材中心,形成循环;
5. **AI 销售考核**——用中台数据自动出题、考核、点评,替代人工培训;
6. **大模型微调**——让客服回复更像真实销售;
7. **MCP 共享服务集群**——把这些项目里复用的能力抽出来做成集群。
- **一句话收尾**:核心是"以数据中台为底座、各业务取数又反哺"的闭环,帮公司在电商场景降本增效。

### Q2. 客服用的是 LangChain 还是 LangGraph?为什么?

**答：**
- **客服用 LangChain**:客服链路相对线性、固定(意图识别 → 检索 → 生成),走一遍就完,LangChain 的链式编排够用、简单。
- **自媒体运营用 LangGraph**:它要做"选题 → 写作 → 配图"多阶段、还要循环和回滚,是有状态的复杂工作流,LangGraph 的图编排(节点+边+状态)更合适。
- **一句话**:简单线性流程用 LangChain,复杂有状态、要循环/分支/回滚的用 LangGraph——按流程复杂度选型。

### Q3. 那 LangGraph 在你自媒体项目里**具体**是怎么用的?

**答（这题你实际答得有点绕,这是理顺版）：**
- **多阶段工作流 + 子图**:整个内容生成是"选题 → 写作 → 配图"的多阶段流程,我把**每个阶段做成一个独立子图(SubGraph)**,再用一个**总图来统筹调度**它们。
- **Checkpointer 持久化状态**:用 PostgreSQL Checkpointer 把**每一步的状态存进库**。比如用户选完标题,状态就更新成"已选题、待写作",写作节点拿到这个状态才往下走,配图同理——**状态在节点间一步步传递,同时落库**。
- **支持回滚**:因为每步状态都存了,如果"文章写得不行",可以**回滚到上一步保存的状态**重新生成,不用从头跑——故障/不满意的恢复粒度细化到单个节点。
- **全局统一状态 + 用户 ID**:用一个**全局共享状态**对象贯穿流程,并且**和用户 ID 绑定入库**,相当于给每个用户的任务做持久化记忆。
- **一句话**:子图拆阶段、总图调度、Checkpointer 存每步状态(绑用户 ID)、支持任意节点回滚重试。

### Q4. 现场场景设计题:做一个教学平台的 Agent,用户(老师)问它问题,它能创建课程/实验、导入实验内容等。这个结构你怎么设计?

**答（理顺版）：**
- **整体用 LangGraph 的循环结构**(因为要反复交互直到满意):
  1. 老师提问 → Agent 理解意图、调用对应工具(创建课程、创建实验、导入实验内容等);
  2. 执行后把结果返回前端,**让老师反馈**:满意 / 还要改;
  3. 如果不满意,**把老师的反馈连同上下文一起带回去**,告诉 Agent 哪里不行、让它重新生成——**循环**这一步;
  4. 满意了再进入下一步,直到完成。
- **关键设计点**:① 各功能(创建课程/实验、导入内容)封装成 **Tool**,Agent 按意图调用;② 用 **LangGraph 的条件边做循环**(满意就退出、不满意就回到生成节点);③ 用**共享状态 + Checkpointer** 记住当前进度,支持多轮修改和回滚;④ 这其实和我做的自媒体/客服项目是同一套思路——**工具调用 + 状态编排 + 人在环反馈循环**。
- **加分**:如果功能很多,可以拆成**多个子 Agent**(各管一类功能),再用一个**主管 Agent 协调调度**。

### Q5. 用户口语化或专有名词多时,混合检索的稠密/稀疏权重(dense_weight)靠什么信号自动调?

**答：**
- **先做意图识别,再决定怎么查**——用户说完不直接拿原话去检索,先用一个小模型做**意图识别**。
- **两种信号触发调权**:
  1. **识别不清 / 用户说得模糊口语化** → 把 **dense_weight 调高**,让检索更偏语义、更发散,**多召回几个相近结果,再反问用户"是不是这几个之一"**;
  2. **能准确识别、且含专有名词/关键词** → 把 **dense_weight 调低**,让检索更偏关键词精确命中,**只找这个具体的东西**。
- **专有名词的处理**:意图识别能准确判断他要问的是哪个具体东西(而不是仅仅拆关键词),这时调低权重做精确匹配。
- **一句话**:意图识别先行——模糊就调高 dense_weight 偏语义发散、再反问确认;精确就调低偏关键词命中。

### Q6. 这套检索的效果怎么验证?

**答：** 两个层面:
- **检索内部(重排 + 可信度)**:第一次检索是拿**用户问题的向量**去知识库按**余弦相似度**比对,召回 **20 条**;再用一个**小模型(Cross-Encoder)做重排**——把召回的 20 条分别和用户问题一起打分,得到一个**可信度**。可信度高就认为查对了、返回用户;**如果普遍很低,就宁可不答也不答错**,告诉用户"没查到"或"请说得更清楚一点"。
- **整体效果(评测集)**:建一个**专门的评测集**(标准提问 + 正确回答),拿它去批量测,看**回答准确率**和**检索准确率**到底怎么样,作为版本对比和回归的依据。
- **一句话**:重排+可信度阈值兜底(宁可不答不答错)管单次质量,评测集跑准确率管整体回归。

### Q7. 微调的发布/训练是你做的吗?具体流程?怎么验证效果?

**答：**
- **不是全在线上微调,而是"本地小模型验证 + 云端等比缩放"两阶段**:
  1. **本地**用 **LLaMA-Factory + 一个等比例的小模型**先调——我当时下的是 **Qwen2.5-3B**,参数大概 **rank=64、alpha=128**,单卡 40 分钟一轮,主要目的是**验证数据质量**(微调本质调的是数据);
  2. 本地验证数据没问题后,**按模型规模等比例缩放超参**,再到**云端(火山方舟)对生产大模型**做正式微调。
- **训练路径 SFT → DPO**:先 **SFT** 让模型学会正确的回答格式和销售风格(从不会到会说);再 **DPO 偏好训练**——让**销售主管**对模型的多条回复选出更好的,组成偏好对训练,让它"说得更好"。
- **怎么验证效果**:SFT+DPO 都跑通、风格盲评达标,就说明数据基本没问题;再用评测集对比微调前后的回答质量、风格一致性。
- **一句话**:本地小模型先验证数据、云端等比缩放正式训;SFT 学格式风格、DPO 对齐偏好;用盲评+评测集验证。

### Q8. 知识库里 PDF 格式很多(参数、表格等),怎么处理?

**答：** 我之前专门做过 PDF 解析:
- **早期(大模型还不强时)**:走传统 **OCR**——PDF 转图、OCR 提取文字、再人工/规则提取参数并统一单位,**很难做、项目拖了很久**。
- **后来(模型变强后)直接让大模型解析**:在提示词里跟大模型说清楚——**输出成 Markdown / 结构化文本、我要哪些参数、单位统一成什么**,让它直接把我想要的参数和内容抽出来。
- **结论**:直接用多模态大模型识别,比"OCR 提取 + 再处理"那套**强很多、也省事很多**。
- **一句话**:从"OCR 提取再清洗"升级到"直接让大模型按指定结构(Markdown+指定参数+统一单位)抽取",更准更省事。

### Q9. 切分(chunking)怎么做?有特殊标识区分问答吗?

**答：**
- 一开始按**固定长度切**,但会把"一个完整的问答"从中间切成两半,检索只命中半段、回答不全。
- 因为客服知识库本身就是**一问一答**的结构,所以改成**按问答的标签切**——具体是按 **H 标签(Markdown 标题层级)**,**一个问题(一个 H 标签)切一段**,保证每个切片是语义完整的一问一答。
- **一句话**:从固定长度切(会切断问答)改成按 H 标题标签切,一问一答一段,解决语义截断。

### Q10. 记忆(memory)怎么做?

**答：** 分短期和长期:
- **短期记忆**:按 **用户 ID + session** 维护当前会话的最近上下文;如果**上下文过长就压缩**(摘要),避免超出窗口。
- **长期记忆**:把需要长期保留的状态/用户画像**存到向量库(PostgreSQL/pgvector)**里,需要时检索召回。
- **一句话**:短期记忆按 ID+session 存、过长就压缩;长期记忆落向量库持久化,按需召回。
