# RAG 面试题整理

## 一、知识库文档更新后怎么处理？

### 1. 文档更新了怎么办

先说反模式，再说正解，体现 trade-off 意识：

**反模式**：整库重新分块、重新 embedding、全量替换。简单粗暴，但成本随知识库规模线性增长，而且会有"重建窗口期"——旧索引已删、新索引未完成时检索会出现空洞或不一致。

**正解的几个层次**：
- **变更检测**：给每个文档维护 `doc_id + content_hash + version + last_modified`，定期/事件触发 diff，只对哈希变化的文档触发重处理（而不是定时全量扫描）
- **粒度收敛到文档级**：变更只波及该文档自己的 chunk，不影响其他文档——这是最基本的隔离要求
- **索引一致性**：用"先插入新版本 chunk（带 version 标记）→ 切流量 → 再异步删除旧版本"的蓝绿方式，避免检索时新旧并存导致召回重复或语义矛盾
- **软删除 + 异步 GC**：先标记失效，避免同步删除阻塞写路径，垃圾回收异步兜底

### 2.（追问）chunk 边界还能和之前一模一样吗？

**不能，这正是问题的核心**。原因：

chunk 本质是对源文档做"滑动窗口/语义切分"产生的**派生产物**，它没有自己独立的、稳定的身份（identity）。一旦文档中间插入/删除了内容：
- 固定长度切分：后续所有 chunk 的 token 偏移量整体平移，就像在数组中间插入元素，下标全部错位
- 语义切分（按段落/标题）：如果改动让段落合并或拆分了，切分点本身就变了

所以"旧 chunk #5 对应新 chunk #5"这种基于位置/序号的对应关系**在编辑发生后立刻失效**——这是经典的"无锚点序列对齐"问题，跟字符串 diff、数组 diff 是同一类问题。

### 3. 怎么把旧 chunk 和新 chunk 一一对应起来做局部更新？

核心思路：**把"对应关系"从依赖位置/序号，转移到依赖稳定的结构锚点和内容指纹**。

1. **先在结构层建立稳定 ID，而不是在 chunk 层**
   把文档解析成结构树（章节 → 段落 → 句子），给每个结构单元一个稳定 ID（基于内容哈希，或基于结构路径如 `h2[2]/p[3]`）。chunk 永远不跨越结构单元边界——这样 chunk 的身份就退化为"它包含哪些结构单元 ID 的集合"，而不是"它在第几个字节"。

2. **内容寻址（content-addressed）做 diff**
   对每个结构单元算归一化后的内容哈希（去除空白、大小写等噪声），新旧版本对比：
   - 哈希相同 → 单元未变 → 对应的 chunk 和向量原样保留，跳过重新 embedding（这是省钱的关键）
   - 旧哈希消失 → 单元被删除 → 删除对应向量
   - 新哈希出现 → 新增/修改的单元 → 只对这部分重新分块、重新 embedding

   本质上是 Git blob 寻址、rsync 滚动哈希的同一套思想：**用内容指纹代替位置做对齐**。

3. **用序列 diff 算法处理"移动"和"局部编辑"**
   如果不是整段替换，而是段落顺序调整或句子级小修改，简单的哈希集合对比会把"移动"误判为"删除+新增"。这时在结构单元序列上跑 LCS / Myers diff，能识别出"这段只是挪了位置"，从而避免不必要的重新 embedding。

4. **边界缓冲区（buffer zone）兜底**
   即便做了结构锚定，仍可能有 chunk 因为 overlap 策略跨越了相邻单元。解决办法是：变更单元的直接相邻 chunk 也一并标记为"待复核"，重新生成，防止出现"新内容混入旧 chunk 尾部"或检索时语义断裂的问题。

5. **一个朴素但诚实的兜底**
   如果工程量打不过收益（比如知识库不大、embedding 成本可接受），完全可以承认："我们选择把粒度收敛到文档级，文档改了就整篇重新分块嵌入，但通过版本号 + 蓝绿切换保证一致性，不做 chunk 级别的精确对应"——这其实是很多生产系统的真实选择，**关键是要说出"我们权衡过 chunk 级精确对应的工程复杂度 vs 收益，选择了粒度更粗但更稳的方案"**，而不是"没想过这个问题"。

> **题眼**：chunk 不是一等公民，它是源文档结构的派生物；要做增量更新，必须把"对齐"这件事锚定在比 chunk 更稳定的层（结构单元 + 内容哈希），再用 diff 算法把变化范围收紧到最小，最后才落到 chunk 级的增删。

## 二、什么是 chunk？

在 RAG（检索增强生成）系统的语境里，**chunk（分块/文本块）指的是把一篇长文档切分成的一个个小片段**，是知识库建库时的基本处理单元。

### 为什么需要切分

直接原因是技术限制：
- **Embedding 模型有输入长度上限**（比如几百到几千 token），不能把一篇几万字的文档整体塞进去做向量化
- **LLM 的上下文窗口有限**，检索回来的内容要拼进 prompt，不能太长
- **检索粒度问题**：如果把整篇文档当一个单位存起来，用户问一个很具体的小问题时，召回的是"一整篇文档"，里面 90% 都是无关内容，既浪费 token 又稀释了相关信息的权重，导致 LLM 抓不住重点

所以需要把文档拆成大小适中、语义相对完整的小段——这些小段就是 **chunk**。

### 一个简化的流程示意

```
原始文档（10000字长文）
      │
      ▼  分块 (chunking)
[chunk1] [chunk2] [chunk3] ... [chunkN]   每个几百字
      │
      ▼  向量化 (embedding)
[vector1] [vector2] ... [vectorN]
      │
      ▼  存入向量数据库

用户提问 → 向量化 → 检索最相似的几个 chunk → 拼进 prompt → LLM 生成答案
```

### 常见的切分方式

- **固定长度切分**：按字符数/token 数切，简单但可能把一句话从中间切断
- **按结构切分**：按段落、标题、Markdown 区块等自然边界切，语义更完整
- **滑动窗口（overlap）切分**：相邻 chunk 之间保留一段重叠内容，避免关键信息正好落在切分点两侧被拆散
- **语义切分**：用模型判断语义边界（比如话题转换处）来切，效果通常最好但成本也最高

### 切分参数的核心权衡

- **chunk 太大**：召回的内容里夹杂无关信息多，"信噪比"低，还可能超长截断
- **chunk 太小**：丢失上下文，一句话可能因为缺少前后文而变得语义模糊，召回时也容易"只见树木不见森林"

> 这也是为什么"文档更新后 chunk 怎么对应"会成为一个很有深度的考点——因为 chunk 本身就是一个"为了适配模型限制而人为切出来的派生产物"，它没有像数据库主键那样天然稳定的身份。

## 三、召回（Retrieval）相关面试题

### 1. 常见的召回方式有哪几种，各自优缺点？

| 方式 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| 关键词召回（BM25/倒排索引） | 基于词频/逆文档频率匹配 | 对专有名词、精确术语、数字编号召回精准；可解释、低成本 | 无法理解语义，"轿车"搜不到"汽车" |
| 向量召回（Embedding + ANN） | 把 query 和文档映射到同一语义空间算相似度 | 能处理同义词、近义表达、跨语言 | 对精确匹配（型号、人名缩写）反而不如关键词；存在"语义漂移" |
| 混合召回（Hybrid） | 两路并行，结果融合 | 兼顾语义和精确匹配，工业界主流方案 | 多一套融合排序逻辑，调参成本上升 |

**追问：为什么不能只用向量检索？**
答：因为 embedding 模型对**低频词、专有名词、数字、代码片段**等"分布外"内容的表征能力较弱——它们在训练语料里出现少，向量空间里挤在一起，区分度差。比如用户搜"GPT-4o"，向量检索可能召回一堆"GPT"相关但版本不对的内容，而 BM25 这种精确匹配反而能稳准地命中。所以生产系统几乎都是双路并行 + 融合。

### 2. 向量召回和关键词召回的结果列表，怎么融合排序？

最常见的是 **RRF（Reciprocal Rank Fusion，倒数排名融合）**：

```
RRF_score(d) = Σ  1 / (k + rank_i(d))
              i∈召回路
```
其中 `rank_i(d)` 是文档 d 在第 i 路结果中的排名，`k` 是平滑常数（常取 60）。

**为什么用排名而不是直接拿原始分数加权？** 因为 BM25 分数和余弦相似度**量纲完全不同**（一个是 0~∞ 的 TF-IDF 分，一个是 -1~1 的相似度），直接加权需要做归一化且很难调出稳定的全局权重；而排名是无量纲的相对顺序，天然可比，RRF 因此简单、鲁棒、几乎不需要调参，是工业界默认首选。

### 3. 发现语义检索经常召回"看起来相关但答非所问"的内容，怎么优化？

这是典型的 **query 和 document 语义空间不对齐**问题，常见手段：

1. **Query 改写/扩写（Query Rewriting / Multi-Query）**：用 LLM 把用户的口语化、有歧义的提问改写成多个更明确的子查询，分别检索后合并结果，提高召回的覆盖面
2. **HyDE（Hypothetical Document Embeddings）**：先让 LLM 针对 query 生成一段"假设性回答"，再用这段假设回答去做向量检索——因为"答案"和"文档"在语义空间里的分布远比"问题"和"文档"更接近，能显著提升召回相关性
3. **领域微调 embedding 模型**：通用 embedding 模型在垂直领域（法律、医疗、代码）的语义区分度往往不够，用领域内的 (query, doc) 对做对比学习微调
4. **元数据过滤前置**：先用结构化条件（时间、类目、来源）做硬过滤，缩小检索范围，减少"语义相似但业务不相关"的噪声

### 4. 为什么要分"召回"和"精排（rerank）"两个阶段，不能一步到位吗？

本质是**效率和精度的权衡**：

- **召回（粗排）**：用计算成本低的方式（向量近邻搜索 ANN、BM25）从海量文档里快速筛出 top-N（比如 50~100 条），目标是"宁可错召，不可漏召"，追求**召回率（Recall）**
- **精排（rerank）**：用更重的模型（Cross-Encoder，把 query 和 doc 一起输入做联合打分）对这 N 条做精细排序，取 top-K（比如 3~5 条）给 LLM，目标是**准确率（Precision）**

之所以不能一步到位用 Cross-Encoder 直接排全库，是因为它需要把 query 和**每一篇**候选文档拼接后过一遍模型，复杂度是 O(全库大小)，无法在线上做到毫秒级响应；而向量召回可以靠预先构建的 ANN 索引把复杂度降到近似 O(log N)。这就是经典的 **"双塔召回 + 交互式精排"** 粗精排架构。

### 5. 向量数据库底层是怎么做近似最近邻（ANN）搜索的？

文档量一大，暴力计算 query 和每个向量的相似度（KNN）代价太高，所以用**近似算法**换取速度，常见两类：

- **图索引（HNSW，Hierarchical Navigable Small World）**：构建多层图结构，搜索时从顶层稀疏图快速定位大致区域，逐层下探到底层稠密图精确查找。查询快、召回率高，是目前最主流的方案（Milvus、Weaviate、pgvector 都支持）
- **倒排索引（IVF，Inverted File Index）**：先对向量空间聚类分桶，查询时只在最近的若干个桶里搜索，减少比较次数，常和量化（PQ，乘积量化）结合压缩存储

**追问：HNSW 和 IVF 怎么选？** HNSW 召回率和速度更好但建索引慢、内存占用大；IVF 系列（尤其结合量化）更省内存、适合超大规模数据，但召回率会有损失。中小规模知识库（百万级以内）优先 HNSW，亿级以上要考虑 IVF+PQ 做内存压缩。

### 6. 上线前后怎么评估召回模块的质量？

- **离线评估**：构造一批"query - 标准答案文档"的标注集，计算
  - **Recall@K**：top-K 结果里有没有命中标准答案，衡量"漏不漏召"
  - **MRR（Mean Reciprocal Rank）**：标准答案排在第几位的倒数平均，衡量"排得靠不靠前"
  - **NDCG（Normalized Discounted Cumulative Gain）**：考虑多个相关文档及其相关程度分级的综合排序质量指标
- **在线评估**：A/B 测试看最终业务指标——回答采纳率、用户重新提问率（追问率高往往说明第一次没召回对）、人工评估打分
- **反向校验**：定期抽样人审"被召回但其实无关"和"该召回但漏掉"的 case，反哺 query 改写规则和分块策略的迭代

## 四、工程 / 高并发 / 架构相关面试题（37–52）

### 37. 让 AI 写代码的话，你怎么把控架构和规范，不让 AI 乱写？

核心思路：**不要把 AI 当"全权委托的程序员"，而是当"需要被约束的高产实习生"**，靠流程和工具把它框住。

- **架构由人定，AI 只填实现**：模块边界、分层（API 层 / 服务层 / 数据层）、接口契约由人先设计好，AI 只在既定骨架内写具体实现，不让它自由发挥决定项目结构
- **小步快跑 + 强制 Review**：让 AI 一次只改一个小范围，每次产出都人工 Code Review，绝不直接合并大段未读代码
- **规范用工具固化，而非靠提示词**：Linter（ESLint/Ruff）、formatter（Prettier/Black）、type checker（mypy/TypeScript）、pre-commit hook、CI 流水线——这些是"硬约束"，AI 写歪了直接卡住，比在 prompt 里反复叮嘱靠谱
- **测试先行**：要求 AI 配套写单元测试，用测试覆盖率和测试通过与否来校验它的产出是否符合预期
- **上下文喂规范**：把项目的编码规范、目录约定、技术选型写进 `CLAUDE.md` / 规则文件，让 AI 每次都带着团队规范工作
- **关键路径人写**：核心算法、安全相关、涉及钱/权限的逻辑，人主导，AI 只做辅助

> 一句话：**架构和规范的"决策权"留在人手里，AI 负责"执行"，并用自动化工具链做兜底校验。**

### 38. 你们这个对话智能客服，用户并发量大不大？

这是一道"摸底题"，面试官想确认你有没有真实的量级概念。建议回答**结构化地给出量级 + 拆解口径**，而不是空泛地说"挺大的"：

- **先区分口径**：日活/月活用户数 ≠ 同时在线数 ≠ 瞬时 QPS。客服场景真正压系统的是"同时进行中的会话数"和"每秒新消息请求数"
- **给一个可信的量级范围**（按真实项目，举例）：比如日活几万、峰值同时在线几百到上千会话、消息接口峰值 QPS 几十到几百
- **强调长连接特性**：客服是有状态的多轮对话 + 常用 WebSocket/SSE 长连接，连接数压力 ≠ 请求 QPS 压力，要分开评估
- **强调 AI 推理是瓶颈大头**：每条消息背后是一次 LLM 调用，单次响应秒级、且是 IO 密集型，所以系统设计的核心矛盾是"如何用有限的下游 LLM 算力扛住并发请求"——自然引出后面的队列、限流、降级话题

> 没真实数据时诚实说"我们当时大概是 X 量级"，并把**评估方法**讲清楚，比硬编一个夸张数字更得分。

### 39. 高并发场景下，哪个环节最容易崩？你做了什么优化和降级？

**最容易崩的环节：下游 LLM 推理调用**。原因：它是整条链路里最慢（秒级）、最贵、并发上限最低的一环，请求一旦堆积，连接和内存会被迅速耗尽，引发雪崩。

**优化手段**：
- **异步化 + 队列削峰**：请求进来先入队，由消费协程按下游可承载的速率消费，把"瞬时洪峰"摊平成"平稳水流"
- **限流**：对入口做令牌桶/漏桶限流，超过阈值直接快速拒绝，保护后端
- **缓存**：高频相同问题（FAQ）命中缓存/语义缓存，直接返回，不打 LLM
- **连接池 + 超时控制**：对下游调用设置合理超时和连接池上限，避免慢请求拖垮整个进程
- **批处理**：在延迟可接受时合并请求批量推理，提升吞吐

**降级手段**（保证"可用"优先于"完美"）：
- LLM 超时/失败 → 降级到规则引擎 / 检索式 FAQ / 预设话术兜底回复
- 整体过载 → 排队提示"前方拥挤，请稍候" + 返回兜底答案，而不是直接 5xx
- 非核心功能（如个性化推荐、富文本渲染）在高压时先关掉，保住核心问答链路
- **熔断**：下游连续失败时熔断一段时间，避免无效请求继续打满连接

### 40. 你对外提供接口服务用的什么 Web 框架？

**FastAPI**（Python）。选它的理由：
- **原生异步（ASGI）**：基于 Starlette + asyncio，天然支持 `async/await`，特别适合 AI 应用这种"IO 密集、等下游响应"的场景，高并发下用协程比多线程更省资源
- **自动数据校验**：Pydantic 做请求/响应模型校验和序列化，少写大量样板代码，类型安全
- **自动 API 文档**：Swagger/OpenAPI 自动生成，前后端联调友好
- **性能好**：在 Python Web 框架里属于第一梯队，配合 Uvicorn/Gunicorn 多 worker 部署
- **支持 SSE/WebSocket**：流式输出 LLM token、长连接客服场景都能很好支持

### 41. FastAPI 接收请求后，后续异步处理用的协程还是多线程？

**协程（asyncio）为主**。原因：

- 我们的业务是**IO 密集型**（等 LLM 响应、等数据库、等向量库），瓶颈在"等待"而非"计算"。协程在等待 IO 时能让出事件循环去处理其他请求，**单线程就能撑起成千上万并发连接**，没有线程切换和锁的开销
- 多线程在 Python 里受 **GIL** 限制，对 CPU 密集任务并行无益，反而带来上下文切换和同步成本；只在"必须调用同步阻塞库"时，才用 `run_in_threadpool` / `loop.run_in_executor` 把阻塞调用丢到线程池，避免堵塞事件循环
- **真正的 CPU 密集任务**（如本地大规模向量计算、文档解析）会放到进程池或独立服务，不和主事件循环抢资源

> 原则：**IO 密集走协程，阻塞调用隔离到线程池，CPU 密集隔离到进程/独立服务。**

### 42. 用协程的话，对外请求、生产/消费队列是怎么管理的？

典型的**生产者-消费者模型**，用 `asyncio.Queue` 在协程间传递任务：

- **生产者**：FastAPI 的请求处理协程接到请求后，不直接同步调 LLM，而是把任务（带请求标识、用户输入、回调/Future）`put` 进 `asyncio.Queue`，然后 `await` 等待结果（或立即返回任务 ID 走异步轮询/SSE）
- **消费者**：后台常驻一个或多个消费协程，循环 `await queue.get()` 取任务，按可控速率调用下游 LLM，拿到结果后通过 `Future`/回调把结果还给对应的请求协程
- **队列起到削峰、限速、解耦**的作用：入口流量再猛，消费端始终按下游能承受的节奏处理
- **并发控制**：用 `asyncio.Semaphore` 限制同时打到 LLM 的并发数；用队列 `maxsize` 做背压，队列满了就拒绝/降级，防止内存被无限堆积撑爆

### 43. 生产队列和消费队列是两个协程 Task，你是怎么设计调度的？

- **启动**：应用启动时（FastAPI 的 `lifespan` 钩子里）用 `asyncio.create_task()` 把消费者协程拉起来常驻运行，作为后台 Task 一直消费队列
- **生产者**：每个进来的 HTTP 请求本身就是一个协程，作为生产者把任务投递进队列，不阻塞
- **消费者**：可以起多个消费 Task（worker pool）并行消费，数量根据下游 LLM 的并发承载能力设定，用 Semaphore 兜住总并发
- **解耦协调**：生产和消费通过队列通信，互不直接依赖；消费完通过 `asyncio.Future` / `Event` 通知生产端"你的结果好了"
- **优雅关闭**：应用关闭时，停止接收新任务 → 等队列里存量任务处理完（或设超时）→ 取消消费 Task → 释放资源，避免请求丢失
- **异常隔离**：消费协程内部 try/except 包住每个任务，单个任务失败不能让整个消费 Task 挂掉；必要时配重试和死信队列

### 44. 多个用户同时请求进来，消费队列怎么区分该处理哪个用户的请求？

关键：**任务在入队时就携带"身份"和"回执通道"，消费者无需"区分该处理谁"——它平等处理每个任务，处理完按携带的回执把结果精准还给对应请求方**。

- **入队的任务体里带上**：请求唯一标识（RequestID）、用户 ID / 会话 ID、输入内容，以及一个**用于回传结果的 `asyncio.Future`（或回调/会话 channel）**
- 消费者取出任务、调 LLM、拿到结果后，调用任务自带的 `future.set_result(...)`，正在 `await` 这个 Future 的那个请求协程就被唤醒、拿到属于自己的结果
- 这样**结果路由靠的是 Future 对象的引用关系，而不是消费者去"找"是谁的**——天然不会串号
- 多轮对话的上下文则靠 **会话 ID（session_id）** 去存储（Redis/DB）里捞对应历史，保证同一用户的多轮连续

### 45. 生产队列存入数据时，是不是要带上请求唯一标识？

**是，必须带**。RequestID（或 trace_id）的作用：

- **结果路由 / 幂等**：消费完按 RequestID 把结果还给正确的请求方；重复投递时靠它去重，保证幂等，避免同一请求被处理两次
- **全链路追踪**：日志、监控、排查问题时，用同一个 RequestID 串起"入口 → 队列 → LLM 调用 → 返回"整条链路，定位某条请求卡在哪
- **超时/重试管理**：标识哪条请求超时了、需要重试哪条
- **多轮/多用户隔离**：配合 user_id / session_id，区分不同用户、不同会话的请求
- 通常在请求入口处生成（UUID 或雪花 ID），贯穿整个生命周期，并通过日志 / 响应头透出

### 46. 高并发流量暴涨后，多节点部署运行时，架构设计上要考虑什么？

主要从这几个维度：
- **无状态化（Stateless）**：应用节点本身不存会话状态，状态外置到 Redis/DB，这样任意节点都能处理任意请求，才能水平扩展和随意增删节点
- **负载均衡**：前面挂 LB（Nginx/网关）把流量均匀分发到多节点，并支持健康检查、自动摘除故障节点
- **会话/数据共享**：会话上下文、缓存放到共享存储（Redis），不放进程内存
- **数据一致性**：多节点同时读写共享数据，要处理缓存与 DB 一致性、并发写冲突（见 49-51）
- **弹性伸缩**：能根据负载自动扩缩容（K8s HPA），流量暴涨自动加节点
- **限流/熔断/降级下沉到网关层**：统一在入口做保护
- **可观测性**：分布式日志、链路追踪、监控告警，多节点下排障必备
- **消息队列解耦**：跨节点的异步任务用 Kafka/RabbitMQ 这类外部 MQ，而不是单机 `asyncio.Queue`

### 47. 单节点扛不住、改成多节点集群，设计上有哪些注意点？

- **首要：把状态请出进程**。单机时放在内存里的队列、会话、缓存、定时任务，多节点下都会出问题——必须外置到 Redis / 外部 MQ / 分布式存储
- **队列要换成分布式 MQ**：`asyncio.Queue` 是进程内的，多节点不共享，得换成 Kafka/RabbitMQ/Redis Stream，否则各节点队列各管各的
- **幂等性**：同一请求可能被重试或被不同节点处理，业务操作要幂等（靠 RequestID 去重）
- **分布式锁**：对共享资源的并发修改、避免多节点重复执行的定时任务，需要分布式锁（Redis/etcd）
- **会话亲和 vs 无状态**：长连接（WebSocket）要么做 sticky session 让用户固定连某节点，要么把连接状态也外置
- **配置统一**：配置中心统一管理，避免各节点配置漂移
- **数据一致性与缓存同步**：见下题
- **优雅上下线**：节点扩缩容、发版时要能优雅摘流量、处理完存量请求再退出

### 48. 多节点部署主要要解决高可用什么问题？

核心是消除**单点故障（SPOF）**，保证部分节点挂掉系统整体仍可用：

- **故障容错**：一个节点宕机，LB 自动把流量切到健康节点，用户无感知（避免单点 = 全挂）
- **无缝伸缩**：流量涨了加节点、跌了减节点，平滑应对波动
- **滚动发布 / 不停机更新**：逐个节点更新，发版期间服务不中断
- **负载分摊**：把压力分散到多节点，避免单机被打满
- 配合**多可用区/多机房**部署，还能抵御机房级故障

> 一句话：多节点的高可用目标是**没有任何单一节点的故障能导致整个服务不可用**。

### 49. 缓存和数据库的数据一致性问题，是为了避免什么问题？

为了避免**用户读到过期/错误的数据（脏读）**，进而引发业务错误。典型危害：

- 数据库已更新，缓存还是旧值 → 用户读到的是过时数据
- 缓存更新成功但 DB 失败（或反过来）→ 两者长期不一致，数据"分裂"
- 严重业务场景（库存、余额、订单状态）下，脏数据会导致超卖、扣错款、状态错乱等真实事故
- 还要避免**缓存击穿/穿透/雪崩**等因缓存失效引发的 DB 被打垮

> 本质：缓存是 DB 的"副本/加速层"，一致性保障就是为了让这个副本不要给用户提供错误的信息。

### 50. 什么业务场景下会产生缓存和数据库数据不一致？

主要是**"写"操作**的时候，缓存和 DB 是两次独立操作、非原子，中间任何环节出问题或并发交叉就会不一致：

- **并发读写交叉**：A 在更新，B 同时在读并回写旧值到缓存——经典的 "更新 DB → 删缓存" 之间被读请求插入，把旧数据又写回缓存
- **更新双写失败**：先写 DB 成功、再删/更新缓存时失败（网络抖动、进程崩），导致缓存留着旧值
- **多节点并发更新同一条数据**：两个节点几乎同时改，缓存更新顺序错乱，后写的旧值覆盖了先写的新值
- **主从延迟**：DB 读写分离时，写主库后立刻读从库，从库还没同步到，缓存被回填成旧值
- **缓存过期与更新的时序竞争**：TTL 到期重建缓存的瞬间，恰好有写操作进来

> 常见缓解：Cache Aside（更新 DB 后删缓存而非改缓存）+ 延迟双删 + 给缓存设 TTL 兜底 + 必要时用分布式锁/版本号串行化。

### 51. 普通用户单条请求都带 RequestID，为什么还会有多节点数据不一致问题？

因为 **RequestID 解决的是"单条请求自身的幂等和追踪"，而数据不一致是"多条请求之间在共享数据上的并发竞争"，两者根本不在一个维度**：

- RequestID 保证的是：**同一条请求**不被重复处理（幂等）、可被追踪。它管的是"一条请求的生命周期"
- 但数据不一致的根源是：**不同请求、不同用户、不同节点**在**同一份共享数据**上并发读写产生的竞态。比如用户 A 的请求和用户 B 的请求都去改同一个商品库存，各自 RequestID 都唯一且没被重复处理，但它们交叉执行依然会把库存改错
- 换句话说，RequestID 是"纵向"的（一条请求内的幂等），数据一致性是"横向"的（多条请求间的并发协调）。前者管不到后者
- 要解决横向并发问题，需要的是**分布式锁、乐观锁（版本号/CAS）、串行化、MQ 顺序消费、事务**等并发控制手段，而不是 RequestID

> 一句话：RequestID 防的是"同一请求重复"，防不了"不同请求争抢同一份数据"。

### 52. 除了缓存一致性，多节点部署还有其他要考虑的设计方向吗？

还有很多，缓存一致性只是其中一块：

- **服务发现与负载均衡**：节点动态增删时，流量如何自动感知和分发（注册中心 + LB）
- **分布式会话/状态管理**：会话、登录态、长连接状态外置共享
- **分布式锁与并发控制**：跨节点的资源互斥、防止重复执行
- **分布式事务 / 最终一致性**：跨服务、跨库的操作如何保证一致（TCC、Saga、可靠消息最终一致）
- **消息队列与异步解耦**：用 MQ 削峰、解耦、跨节点通信
- **配置中心**：统一配置，避免节点间配置漂移
- **分布式 ID 生成**：全局唯一 ID（雪花算法等）
- **限流 / 熔断 / 降级**：在网关和服务层做统一的过载保护
- **可观测性**：分布式日志收集、链路追踪（trace_id 串全链路）、监控告警
- **数据分片 / 读写分离**：单库扛不住时的水平拆分
- **优雅上下线与滚动发布**：发版、扩缩容时不中断服务
- **幂等性设计**：应对重试和重复消息
- **容灾与多活**：多机房/多可用区部署，抵御区域性故障

> 可归纳为三大方向：**无状态化（状态外置）、并发协调（锁/事务/幂等）、稳定性兜底（限流熔断降级 + 可观测性）。**
